## 3. 三模型协作过程复盘

### 有效成果

1. Claude 在前半段对架构、边界、验收严谨性贡献很大。
2. GPT/Codex 在代码落地、Node22 修正、e2e 修复、验证脚本方面贡献最大。
3. Gemini 在部分阶段承担了辅助审计，但工具权限、ignore pattern、MCP issue 多次影响有效性。
4. 集群最终做出了可运行原型，而不是停留在讨论。
5. 安全边界守住了：ToolRegistry 默认 disabled，NoeLoop 默认不执行危险动作，BaiLongma 只读审计。

### 主要问题

1. 第一阶段讨论过久，说明“讨论上限”不能只靠 prompt。
2. 多模型并行产出大量重复文档，造成 canonical 漂移。
3. Gemini 多次因路径/工具权限问题无法读文件或输出半有效结论。
4. Claude 后期出现额度/CLI 问题后，failover 不够自动化。
5. CE11 有磁盘产物但房间状态不结算，说明 dispatcher/store 事务闭环不可靠。
6. 交付门禁曾因旧 artifact 缺字段而错误回滚 CE01，说明验收器把“元数据缺失”误判成“业务失败”。
7. 用户侧可见性不足，看不到模型实时执行内容、卡在哪、谁在主导、谁失败、谁接管。

## 4. Noe 产品当前最大短板

按优先级排序：

1. **房间状态闭环不可靠**  
   产物完成和面板状态不一致。必须修 API/store/dispatcher 层，不要再靠直接改 JSON。

2. **Brain UI 只是 Lite，不是真 Jarvis 中枢**  
   当前能看 Loop/Focus/Memory/Tools/Health，但不能完整展示“任务 -> 模型 -> 执行 -> 证据 -> 交付”的运行链路。

3. **NoeLoop 还没有真实 Act Pipeline**  
   目前 tick 主要是安全事件，不会真正调模型、选择工具、执行动作、写回结果。下一步应做最小安全 Act Pipeline。

4. **Memory 仍是关键词/FTS 层**  
   需要重要性评分、压缩、冲突合并、向量召回、项目隔离策略、长期遗忘策略。

5. **文档和阶段文件过多**  
   顶层 `NOE_PHASE*`、`HANDOFF*`、`*_CANONICAL`、`*_VERIFY` 文件数量很大，需要归档和入口收敛。

6. **Node 版本风险仍需系统化**  
   默认 Node 26 会触发 `better-sqlite3` ABI 问题。已用 Node22 跑通，但应进一步加 `.nvmrc`、启动前检查、统一 runbook。

7. **Electron 正式 App 化未完成**  
   `package.json` 中仍有：
   - `asar: false`
   - `identity: null`
   - `hardenedRuntime: false`
   - `gatekeeperAssess: false`

   这说明它还不是正式签名、公证、DMG 分发级 App。

## 5. 下一窗口建议优先级

### P0：先修协作/交付状态闭环

目标：让房间状态、阶段状态、磁盘产物、交付清单一致。

建议做法：

1. 不直接写 `rooms.json`。
2. 给面板补一个正式 supervisor/reconcile API 或修 dispatcher 恢复逻辑。
3. 当 CE11 磁盘产物存在且验证通过时，能通过事务更新 `taskList`、`stageArtifact`、`clusterDeliveryManifest`、`clusterDeliveryPackage`。
4. 重启面板后状态不能回滚。

### P1：加执行可视化

目标：用户能看见模型在做什么。

建议 UI 加：

- 当前阶段 / 当前轮次
- 主导模型 / 辅助模型
- 每个模型：running / succeeded / failed / quota / timeout / stale
- 最近 stdout/reply 展开
- 当前产物路径
- 当前阻断原因
- failover / takeover 记录
- delivery gate 结果

### P1：收敛 canonical 和文档

建议：

1. 保留一个入口：`NOE_PHASE11_RETROSPECTIVE_CANONICAL.md` 或新建 `NOE_CURRENT_HANDOFF.md`。
2. Phase2/3/4/9/10 重复文档加 superseded 指针。
3. `上下文交接.md`、`任务交接.md` 瘦身，只保留当前状态和读序。
4. 所有 VERIFY 脚本统一列在一个 `NOE_VERIFY_INDEX.md`。

### P2：做 Noe Act Pipeline

建议最小闭环：

1. tick 读取 Focus + Memory。
2. 生成下一步建议，不自动执行危险动作。
3. 只允许安全工具，如只读文件摘要、项目状态扫描、生成报告。
4. 所有写操作必须审批。
5. 每次 act 写入 Memory 和 ActivityLog。

### P2：Electron 正式化

路线：

1. 梳理 asar 策略和 native module unpack。
2. 修 `better-sqlite3` Electron ABI。
3. 启用 hardened runtime。
4. 签名、公证、DMG。
5. packaged smoke 和重启恢复测试。

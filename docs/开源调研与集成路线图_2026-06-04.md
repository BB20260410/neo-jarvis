## 执行摘要（大白话）

- **记忆（最痛）有现成轮子**：首推 **Mem0**（5.7 万星）或更轻的 **OpenMemory**（纯 Node + 内嵌 SQLite，最适合 Noe 这种 Electron app）。它们干的正是 Noe 缺的"存了能读回来"。**这是最高性价比，先做这个。**
- **语音提速路线清晰但有上限**：先把 `stream:false` 改成 `true`（一行，零成本），再做"边想边说"（生成一句就念一句）。但**本地大脑想那 2.6 秒是硬骨头，流式只能"藏"不能"消"**——真要更快得换更小/量化模型（得实测）。一条号称"0.94 秒"的方案被验证组**全票否决**，别信。
- **陪伴**：核心就是把记忆接上（复用上面记忆层），不需要引入没核实过的专用框架。
- **屏幕识别**：诚实说——这次没验证出可靠结论，列为次优先、单独再查。

## 一、痛点1 长期记忆（最痛）— 开源方案对比

| 方案 | Star | 许可 | Node 支持 | 本地部署 | 与 Noe 契合 | 定位 |
|---|---|---|---|---|---|---|
| **Mem0** `mem0ai/mem0` | **57.7k** | Apache-2.0 | ✅ 一等公民（`mem0ai`/`mem0ai/oss`） | ✅ 官方 Ollama 全本地配方 | 高（生态最大） | **首推·通用记忆层** |
| **OpenMemory** `CaviraOSS/OpenMemory` | 4.2k | Apache-2.0 | ✅ 原生 `openmemory-js` | ✅ 内嵌 SQLite + Ollama embedding | 高（最适合 Electron 内嵌） | **本地优先备选** |
| **Letta**(前 MemGPT) `letta-ai/letta` | 23.1k | Apache-2.0 | ⚠️ 仅客户端 TS，服务端 99.5% Python | ⚠️ 须跑 Python/Docker 后端 | 中（重型，运行时锁定） | 重型备选 |
| **Cognee** `topoteretes/cognee` | 17.7k | — | ⚠️ 原生 Python，须 HTTP/子进程 | ✅ 全本地（向量+图谱统一） | 中（非进程内） | 重型备选 |

**关键代码根因（本次核查证实）**：Noe `OllamaChatAdapter` 写死 `stream:false`、`recall()` 只作工具/路由暴露**未自动注入对话**——这正是第一阶段定位的"记忆只写不读"根因，与本调研互相印证。

**推荐与选型决策**：
- **OpenMemory** 最契合 Noe（纯 Node + 内嵌 SQLite，零外部依赖，且有情景/语义/情感/反思多扇区结构化记忆 + 时序图谱）→ 优先试。
- **Mem0** 生态最大、文档最全（含 Ollama 全本地配方）→ 平行评估。
- **决定性前置测试**：先测 Noe 的本地 4B 模型（qwen3.5-abliterated:4b）**能否稳定发结构化 tool call**。能 → Letta 这类也可选；不稳 → 走 Mem0/OpenMemory 的**被动抽取**（对小模型友好），别选依赖可靠 tool call 的方案。

## 二、痛点2 实时语音低延迟 — 流式路线

**已验证可落地的三步**（按顺序）：
1. **开流式**：`OllamaChatAdapter` 的 `stream:false` → `true`（ollama-js 原生 AsyncGenerator，**零新依赖**，第一步）。
2. **边生成边播**：标点分句缓冲队列，大脑生成中途凑够整句就推给 TTS（arXiv 2508.04721 的 PunctuatedBufferStreamer 模式，LiveKit/Pipecat 业界标配）。首字节可显著提前。
3. **增量 STT**：whisper_streaming 的 LocalAgreement-2 策略（连续 2 次一致即确认前缀）降 STT 延迟。

**架构参考（不可直接用）**：RealtimeVoiceChat / RealtimeSTT（9.8k★，含双 VAD 句末检测）是优秀蓝本，但**全是 Python + CUDA 依赖、无 Apple Silicon GPU 路径**，只能借鉴模式，得在 Noe 自己的 Node 栈里实现。

**⚠️ 诚实的天花板**：
- 一条"端到端 0.94 秒"的论文 claim 被验证组 **0-3 全票否决**，不可作为目标。
- **本地 LLM 生成 2.6s 是首响最大头，流式只能"重叠隐藏"不能"消除"**。真要降，得换更小/量化模型——但 M5 Max 实际吞吐**必须实测**，不能套论文数字。
- 句级流式会**剥离 TTS 跨句韵律**，让"甜心小玲"情感变生硬 → 需设**最小句长护栏**或跨句缓冲。

## 三、痛点3 主动陪伴智能化

核心抓手 = **复用痛点1 的记忆层**（召回历史画像 + 偏好注入 proactive 判断），把"VLM 单帧描述"升级为"带记忆延续的判断"。Letta 的有状态人格、OpenMemory 的情感/反思扇区都能支撑"记得你"。

**诚实声明**：本轮 23 条存活结论中**没有 PaRT 或任何专用 proactive companion 项目的独立验证**——不据印象编造其 Star/活跃度。若复用记忆层仍不够"懂你"，再单独核实这类专用框架。

## 四、痛点4 屏幕/行为识别（次优先）

## 五、分阶段集成路线图（优先级 × 难度）

| 阶段 | 目标 | 方案 | 难度 | ROI |
|---|---|---|---|---|
| **① 记忆读回闭环** | 让 Noe 记得上下文+长期记忆 | OpenMemory/Mem0 接入 `_respond`：注入近 N 轮历史 + recall 长期记忆 | 中 | **最高** |
| **② 流式语音** | 首响从 5-6s → 更早出声 | a) `stream:true`（一行）b) 分句边生成边 TTS c) 增量 STT | 中-高 | 高 |
| **③ 陪伴智能化** | 主动陪伴"懂你" | 复用①记忆层，proactive 带记忆延续 | 中（依赖①） | 中 |
| **④ 屏幕/行为识别** | 连续行为流 | 先专项调研核实 Apple Silicon/Node 可行性，再定 | 高 | 次优先 |

**建议执行序**：① 先做（最痛+最高 ROI+不引重依赖）→ ② 再做（a 一行先上，b/c 渐进）→ ③ 顺势（吃①的红利）→ ④ 延后专项。

## 六、风险与应对

| 风险 | 应对 |
|---|---|
| Mem0 Node OSS SDK 的 Ollama 打包 bug（issue #3857）；OpenMemory FAST 档**静默降级到合成嵌入**（召回差还不报错） | 落地**必须用真实历史对话跑最小召回评测**（Noe 已有 `noe-memory-recall-eval` 测试可扩展）；OpenMemory 用 SMART/DEEP 档 + 维度匹配的 Ollama 嵌入 |
| 句级流式 TTS 削弱"甜心小玲"情感韵律 | 设最小句长护栏 / 跨句缓冲 |
| 本地 LLM 2.6s 流式消除不掉 | 真降延迟须实测 M5 Max 上更小/量化模型吞吐，**不可套用论文数字** |
| Letta/Cognee 自托管完整能力需 Python/Docker | 与 Noe 纯 Node 偏好有摩擦；优先纯 Node 的 OpenMemory |
| 二手营销数字（如 Cognee "92.5% vs 60%"） | 勿引用，已剔除 |

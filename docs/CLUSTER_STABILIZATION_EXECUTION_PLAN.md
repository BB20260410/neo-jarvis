# 集群协同稳定化执行计划

## 最终目标

把当前集群协同模式稳定到可真实用于项目开发，然后封装成 Electron 桌面应用。Electron 只做启动、守护、窗口、日志和恢复；核心稳定性先在调度器和 Web 面板内完成。

## 阶段门禁

每个阶段必须满足：

- 有明确修复范围。
- 有可复现问题或风险说明。
- 有代码或配置落地。
- 有至少一次实际运行或命令验证。
- 有复查结论和下一阶段风险。

## 阶段 1：冻结核心范围和 P0 稳定性

范围只聚焦 `cross_verify` 集群协同。

P0 清单：

- 启动、刷新、重进页面时必须能回到真实运行房间。
- 运行中不能显示旧阻断报告误导用户。
- 运行中不能重复启动。
- 后端 running 与前端按钮/进度必须一致。
- owner token、项目目录、模型连接失败必须显示明确原因。

当前已处理：

- 运行态统一识别 `running / debating / active / processing`。
- 运行中隐藏旧交付包和旧阻断报告。
- 运行中启动按钮禁用并显示 `集群协同运行中`。
- 房间入口在只有一个运行房间时自动选中该运行房间，避免刷新后停在欢迎页。

阶段 1 验收：

- 打开房间入口后自动进入唯一运行中的集群协同房间。
- 页面显示当前进度，不再要求用户猜测是否要继续点击启动。
- 后端状态仍为 running，且遥测调用数会继续推进。

## 阶段 2：调度器状态机

目标：

- 统一 `room.status`、任务状态、心跳、遥测、交付报告之间的关系。
- 新运行开始时旧交付状态只能作为历史，不能污染当前运行。
- 所有 UI 状态从同一主状态推导。

关键实现：

- 引入运行态判定工具。
- 引入 stale delivery 标记或 run 绑定。
- 明确 `idle / running / paused / blocked / done / error` 状态转换。

## 阶段 3：幂等启动、续跑、重试、返工

目标：

- `start / resume / retry / rework` 重复调用不会破坏状态。
- 运行中重复启动返回当前运行状态，不作为错误弹给用户。
- 暂停后只从明确断点恢复。

关键实现：

- 每次运行绑定 `runId`。
- 每个阶段绑定 `stageRunId`。
- 每次模型调用绑定 `turnId`。
- 晚返回只能记日志，不能覆盖新状态。

## 阶段 4：模型失败接管

目标：

- 一个模型失败，剩余模型自动接手。
- 两个模型失败，最后一个模型继续完成。
- 全部模型失败才硬暂停。

关键实现：

- 错误分类：网络、额度、未登录、CLI 中断、输出不合格、用户停止。
- adapter degraded 状态。
- 接管计划写入运行日志和 UI。

## 阶段 5：项目目录隔离和目标模式闭环

目标：

- 每个集群协同房间都有独立项目目录。
- 模型默认只读写本房间项目目录。
- `/目标` 模式不因轮数或输出截断自然停止。

关键实现：

- 项目目录 manifest。
- 阶段验收清单。
- 证据门禁和自动补证据。
- 输出截断自动续写。

## 阶段 6：测试和真机压测

最低测试：

```bash
npm run lint
npm test -- tests/unit/cross-verify-dispatcher.test.js
npm test
npm run test:e2e
```

真机压测：

- 新建集群协同房间。
- 启动三模型。
- 刷新页面。
- 模拟一个模型失败。
- 追加需求。
- 检查项目文件落地。
- 检查最终交付包。

## 阶段 7：Electron MVP

目标：

- 双击打开。
- 自动启动本地服务。
- 自动注入 owner token。
- 自动打开面板窗口。
- 服务挂了自动重启。
- 提供日志窗口和重启按钮。

## 阶段 8：Electron 稳定版

目标：

- supervisor 守护。
- 端口冲突处理。
- 模型健康检查。
- 任务运行中防误退出。
- 打包 `.dmg`。

## 最终验收

- Electron 打开后能真实启动和进入集群协同房间。
- 三模型能真实执行项目任务。
- 刷新、重启、模型失败、追加需求均不打断项目。
- 项目目录有完整产物、运行说明、验证记录和交付报告。

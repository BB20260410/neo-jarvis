## 2. 核心洞察

Neo 的 post_review 复核**正确地拒绝了一个低价值 cycle**——这恰恰证明它的价值判断是**可靠**的（与本轮已装的价值闸同一战线）。真问题不是"复核太严"，而是：

> **被拒之后没有建设性出路**——只能卡死重试到 drop，浪费整个 cycle，还占着唯一坑位堵死供给链 28h。

可靠的自我迭代 = 失败后能**学到 + 改进**，而非卡死浪费。这正是"像 Claude 一样"的核心：Claude 被拒/失败会返工、会从中学，不会卡死空耗。

## 3. 设计目标

让 Neo 从「可靠拒绝 + 浪费」进化到「可靠拒绝 + 学习 + 下次更好」——把"失败"接进自我迭代闭环。

## 4. 改动（按杠杆/风险排序，全 flag 默认 OFF）

### 改动 1（B·最高杠杆·核心）：post_review reject → 学习 + 快速释放
- 位置：`NoeSelfEvolutionTrigger.js` 的 `post_review_required` 分支（assembleCompletion 返回后，:199-208）。
- 当 `completion.ok===false` 且 `completion.reason==='post_review_not_approved'`（**reviewer 明确 reject**）时：
  1. 从 `completion.reviews`/`errors` 提取 blocker，经脱敏记一条 self-evolution learning（"cycle 被拒原因：目标模糊 / 无测试 / 无 diff…"）——**从失败中学**。
  2. 立即释放坑位（drop goal），**不等 60 拍空耗**。
- **关键区分（真 reject vs 暂时失败）**：`reason==='post_review_runner_required'` / `'post_review_failed'`（复核器不可用 / 网络）→ 保留重试（不是 cycle 的错）；只有 reviewer 真 reject 才快速放弃 + 学习。
- 安全：**绝不碰 assembleCompletion 的复核命脉**（reject 依旧 reject，绝不盖章）；只加"被拒后的善后"。

### 改动 2（A·解除占坑·低风险）：被拒即释放坑位允许补位
- 配合改动 1，reject 后立即释放，去重墙不再被一个注定失败的目标堵 28h。

### 改动 3（A·闭环·后置）：reject learning 反馈下一次 autoseed
- 下次 autoseed 立项时参考最近 reject learning，避开同类低价值目标 / 立更具体的项 → 一次比一次会立项 → 最终立出能通过复核的真改进。这是"真正自我迭代"（从失败迭代改进）的闭环关键，但较复杂，作为第二步。

1. **必碰 executor/NoePatchTransaction** 产 artifact——CLAUDE.md 明列「碰 src/loop/ self-evolution / NoePatchApply = 分量动作，留 owner 点火」。

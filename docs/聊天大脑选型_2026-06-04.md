# Noe 聊天大脑选型结论（A/B 实测 + deep-research）

> 日期：2026-06-04　|　方法：本地 A/B 实测（同句对比）+ deep-research 对抗验证（102 agent）
> 背景：当前大脑 `qwen3.5-abliterated:4b` 理解力差、像客服。目标：换更懂人、适合中文聊天陪伴的大脑。

## 一、A/B 实测（同一句"我这两天有点扛不住了"，考"理解情绪而非字面"）

| 模型 | 回答 | 评价 | 速度 |
|---|---|---|---|
| **abliterated:4b**（原） | "感觉你负担挺重的吧？要不要聊聊…我可以当个倾听者，咱们一起想办法" | 理解对，但**啰嗦、客服感** | 快 |
| **abliterated:9b** ✅已换 | "感觉你这两天状态有点累了吧。要不要休息一下，或者跟我聊聊最近发生了什么？" | **温暖、像朋友、简短、遵守不带emoji** | 3.7s（可接受） |
| **gemma4:31b** | "怎么啦？快跟我说说，是压力太大了还是身体不舒服？抱抱你。" | **最自然像朋友** | 慢（20GB） |
| **qwen3.6-27b**(LMStudio) | — | 加载+生成 **>240s 超时** | ❌ 太慢，不适合实时 |
| **gemma-4-12b** ✅已注册实测 | "我能感觉到你正处于非常疲惫、甚至透支的时刻…'扛不住'背后往往是长期压力积累…请先允许自己停下来" | **理解最深**（点出"透支/长期压力"），但**啰嗦+markdown加粗、像公众号科普文，不像朋友随口聊** | 5.3s（偏慢） |

**实测结论**：模型越大越懂人（4b→9b→31b 肉眼可见），但太大就慢（27b/31b 不适合实时秒回）。**9b 是"理解力够好 + 速度可接受"的当前最佳平衡，已切换生效。**

## 二、gemma-4-12b（你点名的）

- **真实存在**（deep-research 三票确认）：Google 2026-06-03 发布，11.95B、256K 上下文、原生多模态。量化 Q5_K_M=8.4GB，M5 Max 轻松跑。
- **被证伪的营销话术**（别信）：✗"16GB 就能跑" ✗"逼近 26B"。
- **没有横向中文跑分/社区评论**裁决它是否比 9b/31b 更懂中文 → **只能实测**。
- **当前状态**：✅ 已注册成功，建本地别名 `gemma4-12b`（**务必用本地别名**：直接用 `hf.co/unsloth/gemma-4-12b-it-GGUF:Q5_K_M` 全名会在冷启动联网查 manifest，国内网络卡死——这正是之前 panel 语音/对话全部超时的根因，见文末）。
- **A/B 实测裁决**（同句"扛不住"，见上表）：12b **理解力确实最强**，但**输出偏长、带 markdown、像科普公众号不像朋友随口聊**，且 5.3s 偏慢，语音念出来太长。
- **视觉实测（2026-06-04，重要）**：`ollama show` 显示它声明支持 `vision`/`audio`，但实测喂截图（4.4MB 大图 / 311KB 小图）**均卡死超时**（600s / 300s 无返回）。→ **当前 ollama + 这个 Q5 GGUF 的视觉跑不通，不能替代 qwen-vl 当 Noe 的眼睛**；多模态架构是真的，但需等 ollama 修好对 gemma4 视觉的支持、或换非量化重型部署（transformers/vLLM 原生跑全精度）。Noe 视觉维持 `qwen-3-vl-8b-heretic @ LM Studio`（实测 26s 准确认出 Noe 界面/配色/星云背景/对话气泡）。
- **判断**：①**语音陪伴选 9b**（简短口语秒回，已生效、已实测）；②**文字深聊/写作想要最强理解力可切 12b**（`NOE_OLLAMA_MODEL=gemma4-12b`，建议先在系统提示里强化"简短/口语/禁 markdown"再上语音）。两者都是本地别名，改 `.env` + 重启即可切换。

## 三、MiniMax 全栈评估（你问的"大脑+语音+视觉都用 MiniMax"）

**结论：不划算，不建议全栈用 MiniMax。** 各环节取舍：

| 环节 | 建议 | 理由 |
|---|---|---|
| 日常聊天大脑 | **本地（9b/12b）** | MiniMax 主力 M3 是**编码/Agent SKU**，官方跑分全是 SWE-bench/Terminal-Bench，**零中文对话/陪伴 benchmark**；它自己另出 "M2-her" 专做陪伴，反证 M 系列不是陪伴款。云端还慢+审查+烧钱。 |
| 难任务/写作 | MiniMax（路由 mid） | 复杂中文写作可上 MiniMax，已在智能路由里。 |
| 语音 TTS | **保留 MiniMax 甜心小玲** | 音色好，这是 MiniMax 最该干的。 |
| 视觉看屏/摄像头 | **本地 qwen-vl（不换）** | MiniMax 有 VL 模型但**平台 API 可用性未定论**；本地 qwen-vl 高频、隐私敏感、免费、就位。 |
| STT / embedding / 记忆召回 / 快速分类 / 断网兜底 | **本地** | 高频、零成本、隐私，没理由上云。 |

**延迟**：全云 MiniMax vs 本地端到端**无实测数据**（官方 60tok/s 未通过验证），但云有网络往返，**本地秒回更适合实时陪伴**。

**一句话**：MiniMax 当"出声的嘴 + 啃硬骨头的外援"，别当"日常聊天的脑 + 看东西的眼"——那两个本地更快更懂你更省钱。

## 六、踩坑根因：语音/对话全部超时（已修复）

**现象**：换模型调试期间，panel 的语音 `/voice/chat`、`/route` 所有请求全部超时（90s+），但直接 `curl` ollama 同样请求体只要 0.8s，断路器/隔离舱/限速全绿。

**根因**：`.env` 一度填了 `hf.co/unsloth/gemma-4-12b-it-GGUF:Q5_K_M` 全名。ollama 对 `hf.co/` 前缀的 model 名**每次冷启动都会联网去 HuggingFace 查 manifest 分片**，国内网络卡死 → `_doChat` 的 fetch 一直挂着。而 **dotenv 只在 panel 启动时读一次 `.env`**，热改 `.env` 不生效——所以即便我把 `.env` 改回 9b，旧 panel 进程内存里仍是那个会联网卡死的全名。

**修复**：①用 `ollama cp` 把它做成纯本地别名 `gemma4-12b`（不再联网）；②`.env` 切回 `huihui_ai/qwen3.5-abliterated:9b`；③**重启 panel**（让 dotenv 重新读）。

**实测验证（重启后 PID 91540）**：
- `/route` 全链路 **2.18s** 正常返回，`usedAdapter: ollama`，断路器 CLOSED、隔离舱 running 0/queued 0、限速 totalDenied 0。
- `/voice/chat` 文字分支：普通问候温暖简短无 think 泄漏；**记忆召回正常**（问"猫叫什么"→准确答"咪咪"）。

**教训**：本地模型一律用本地别名，别在 `.env` 里写 `hf.co/` 全名；改 `.env` 后必须重启 panel 才生效。

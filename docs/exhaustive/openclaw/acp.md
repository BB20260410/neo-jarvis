# OpenClaw src/acp (MiMo补全)

好的，Noe的开发者。我将逐文件分析OpenClaw `src/acp/` 目录下的代码，并为你（Noe项目）指出可优化、改进或完善的点。

---

### **总体概览**
`src/acp/` 是 **Agent Client Protocol (ACP)** 的客户端实现核心，负责将本地终端（或类似UI）与一个后端的 AI 代理（可能运行在另一个进程或服务中）安全、可靠地连接起来。它处理了权限控制、进程管理、命令路由、故障转移和会话状态等关键问题。

---

### **文件 1: `client-helpers.ts`**

*   **职责**: 封装 ACP 客户端所需的各种辅助功能，核心是 **权限请求的解析与决策**。它决定哪些工具调用可以自动批准，哪些需要用户交互确认。
*   **关键机制**:
    1.  **权限分类器 (`classifyAcpToolApproval`)**: 将工具调用归类为 `readonly_scoped`、`mutating`、`exec_capable` 等风险等级。
    2.  **决策引擎**: 根据分类结果和配置（`autoApprove`）决定是静默允许、静默拒绝，还是弹出用户提示。
    3.  **用户交互**: 通过 `readline` 实现了一个带超时的命令行提示符，用于请求用户批准。
    4.  **环境隔离**: 构建了用于启动 ACP 服务器进程的环境变量，剥离了敏感信息。
*   **Noe 可优化/改进/完善的点**:
    1.  **策略引擎增强**: 当前的 `autoApprove` 逻辑是二元的。Noe可以引入更复杂的**权限策略链**，例如：
        *   **基于上下文的策略**: 如果工具作用路径在用户的工作区 `~/projects/` 内，则 `autoApprove` 级别可放宽。
        *   **基于历史/信任的策略**: 用户过去多次允许过某个“高风险”工具，系统可以学习并降低其警告级别。
        *   **集成 Noe 的治理框架**: 将权限决策记录到本地日志或区块链，用于事后审计（符合 `治理` 和 `自由发布链` 特性）。
    2.  **非TTY环境优化**: 当前在非交互终端直接拒绝。Noe作为**本地优先**助手，可以支持更丰富的“非交互式批准”模式，例如，读取一个本地策略文件（`.noe-policy.yaml`）来决定，或通过监听一个本地socket/API来获取外部指令批准。
    3.  **UI/UX 改进**: `promptUserPermission` 是纯文本的。Noe可以将其升级为更友好的终端UI（使用如 `ink`、`blessed` 等库），或通过其 `焦点栈` 机制，将权限请求作为焦点事件推送到系统通知或主界面。
    4.  **错误与降级**: 当用户超时或拒绝时，信息是 `console.error`。可以将其整合到 Noe 的**记忆FTS**中，作为可搜索的“决策历史”记录下来，方便用户回顾“为什么那个工具上次被拒绝了”。

### **文件 2: `client.ts`**

*   **职责**: 实现一个交互式 stdio ACP 客户端。它负责**启动服务器子进程**、**建立协议连接**、**转发用户输入**、**接收并展示服务端事件**。
*   **关键机制**:
    1.  **进程生成 (`spawn`)**: 以子进程方式启动 ACP 服务器，并管理其生命周期。
    2.  **连接建立 (`ClientSideConnection`)**: 使用 `@agentclientprotocol/sdk` 建立 JSON-RPC over stdio 连接。
    3.  **事件循环**: 一个主循环读取 `readline` 输入，发送 `prompt` 请求，并处理接收到的 `sessionNotification`（如流式文本、工具调用状态）。
    4.  **环境准备**: 复用了 `client-helpers.ts` 的环境构建逻辑，确保安全地传递变量。
*   **Noe 可优化/改进/完善的点**:
    1.  **连接管理与重连**: 当前是“一锤子买卖”。服务器进程崩溃，客户端直接退出。Noe可以引入**心跳检测和自动重连机制**，利用 `active-turns.ts` 的状态来判断是否需要在重连后恢复会话，提升可靠性。
    2.  **输入/输出增强**:
        *   **输入**: 可以整合 Noe 的**焦点栈**。例如，当 ACP 会话等待输入时，将其推为最高焦点，确保用户不会错过。
        *   **输出**: `printSessionUpdate` 比较简陋。可以解析 `agent_message_chunk` 进行 Markdown 渲染，或为 `tool_call` 事件提供更结构化的树状视图，展示输入参数和状态变迁。
    3.  **命令路由集成**: 当前客户端直接捕获 `/` 开头的命令。这应与 `commands.ts` 暴露的 `availableCommands` 更紧密地集成，实现**命令自动补全**和帮助信息，利用 Noe 的 `四档路由` 对不同命令进行不同的处理流程。
    4.  **多会话/多服务器管理**: 当前设计是单会话。Noe作为强大的助手，可以管理多个并行的 ACP 服务器连接（连接不同的后端模型或专有工具），并通过一个会话管理器进行切换，这呼应了 `多模型consensus` 的需求。

### **文件 3: `approval-classifier.ts`**

*   **职责**: **静态分析工具调用**，将其映射到预定义的 `AcpApprovalClass`（如 `readonly_scoped`, `exec_capable`）。
*   **关键机制**:
    1.  **工具名解析 (`resolveToolNameForPermission`)**: 从元数据、原始输入、标题等多个来源“投票”确定工具名，确保一致性。
    2.  **路径作用域检查 (`resolveAbsoluteScopedPath`, `isPathInside`)**: 判断文件路径是否在“安全”的本地目录内（如 `$HOME`）。
    3.  **ID集合匹配**: 使用硬编码的 `Set` 对工具ID进行快速分类（如 `SAFE_SEARCH_TOOL_IDS`, `EXEC_CAPABLE_TOOL_IDS`）。
*   **Noe 可优化/改进/完善的点**:
    1.  **动态化与配置化**: 分类逻辑全是硬编码的。Noe应支持**外部规则定义**（例如通过JSON/YAML文件），允许用户或插件定义新的工具分类规则（`"tool: my_custom_tool": {class: "mutating", scopes: [...]}`），实现`自我进化`和可扩展性。
    2.  **上下文感知分类**: 当前的 `isPathInside` 是静态的。可以结合**运行时上下文**：例如，如果当前会话的主题是“系统管理”，那么 `exec_capable` 类工具的自动批准阈值可以临时降低。
    3.  **分类器链与共识**: 对于重要操作，可以设计多个分类器（一个基于规则，一个基于本地小模型的风险评估），通过类似 `多模型consensus` 的机制得出最终风险等级，提高决策的鲁棒性。
    4.  **路径白名单/黑名单**: `isPathInside` 只检查了是否在 `HOME` 内。可以支持更复杂的模式，如 `*.git*` 黑名单、特定项目的白名单等，并存储在 `.noeignore` 类似的文件中。

### **文件 4: `commands.ts`**

*   **职责**: 聚合并提供 ACP 协议中客户端可以向服务端展示的“可用命令”列表。
*   **关键机制**:
    1.  **静态基础命令 (`BASE_AVAILABLE_COMMANDS`)**: 定义了一组核心命令，如 `/help`, `/reset`, `/model`。
    2.  **动态插件命令 (`listDockAvailableCommands`)**: 从已注册的插件（`commands-registry`）中提取以 `dock:` 开头的命令。
    3.  **合并导出 (`getAvailableCommands`)**: 将静态和动态列表合并返回。
*   **Noe 可优化/改进/完善的点**:
    1.  **命令的元数据与四档路由**: 当前命令对象只有 `name`, `description`, `input.hint`。Noe可以为每个命令附加**路由信息**，例如 `{ routing_tier: 1, handler: ‘local’ }` 或 `{ routing_tier: 3, handler: ‘acp_server’ }`，从而在 `四档路由` 框架下决定命令是在本地快速响应，还是必须转发到 ACP 服务器。
    2.  **动态权限与可见性**: 命令列表对所有客户端都一样。可以实现基于当前会话状态、用户角色（`治理`）的**命令可见性过滤**。例如，`/config` 命令只在会话是 `owner` 时才可见。
    3.  **热重载**: 当插件动态注册或注销命令时，`listDockAvailableCommands` 的结果会变化。Noe的客户端需要一种机制来**动态刷新**可用的命令列表，而不是依赖初始连接时的静态快照。
    4.  **命令文档系统**: 结合 `记忆FTS`，可以为每个命令提供详细的、可搜索的帮助文档（使用示例、参数说明），用户输入 `/help <command>` 时可以直接查询本地记忆库。

### **文件 5: `manager.backend-failover.ts`**

*   **职责**: 提供 ACP 会话初始化和回合执行时的**后端故障转移**逻辑。
*   **关键机制**:
    1.  **后端候选规划 (`resolveBackendCandidatePlan`)**: 根据配置的主备后端，去重后生成一个有序的尝试列表。
    2.  **故障转移判定 (`isFailoverWorthyBackendError`)**: 基于错误码和错误信息中的关键词（`unavailable`, `rate limit`）判断是否为“可转移”的临时性错误。
    3.  **索引检查 (`shouldAttemptBackendFailover`)**: 判断是否还有下一个后端可以尝试。
*   **Noe 可优化/改进/完善的点**:
    1.  **智能化决策**: 当前的错误匹配 (`/\b(?:unavailable|rate[-\s]?limit...)\b/i`) 是粗粒度的。Noe可以引入**错误模式库**，或者利用本地记录的失败历史（`记忆FTS`）来学习哪些错误更可能通过切换后端解决。
    2.  **基于健康检查的主动故障转移**: 不应等到请求失败。可以实现一个轻量的**健康探测线程**，定期检查各后端状态，提前将不健康的后端从候选列表中移除或调低优先级。
    3.  **故障转移成本感知**: 切换后端是有上下文丢失成本的。可以在决策时考虑**当前回合的进展**（利用 `active-turns.ts` 的状态）。如果已经在某个后端上收到了大量输出（`sawOutput: true`），即使遇到瞬时错误，也可能选择重试而非故障转移，以保持对话连贯性。
    4.  **与 `四档路由` 集成**: 不同后端可能对应不同的路由档位（如本地模型 vs 云端模型）。故障转移决策可以成为路由策略的一部分，在保证基本可用性的前提下，优先选择档位更符合用户当前意图的后端。

### **文件 6: `active-turns.ts`**

*   **职责**: 管理一个进程本地的注册表，用于跟踪哪些 ACP 会话当前有正在执行的“回合”。
*   **关键机制**:
    1.  **全局单例 (`resolveGlobalSingleton`)**: 使用 `Symbol.for` 在全局范围内确保只有一个 `activeTurnKeys` 集合，避免多实例问题。
    2.  **轻量状态管理**: 提供 `mark`, `clear`, `isTurnActive` 三个原子操作。
    3.  **标准化键名 (`normalizeActorKey`)**: 确保会话键名的一致性。
*   **Noe 可优化/改进/完善的点**:
    1.  **状态持久化与恢复**: 当前状态纯内存，进程重启即丢失。对于 `本地优先` 和强调可靠性的 Noe，可以考虑将活跃回合状态**轻量持久化**到本地文件（如 `~/.noe/active-turns.json`）。在 Noe 启动时，可以尝试恢复这些状态，决定是放弃还是重连这些“僵死”的回合。
    2.  **超时与自动清理**: 没有超时机制。一个标记为“活跃”但实际卡住的回合会永远存在。应增加**看门狗机制**，为每个活跃回合设置一个最大存活时间，超时后自动清理并记录告警。
    3.  **丰富的状态信息**: 当前只记录“是否活跃”。可以扩展状态对象，记录**开始时间、最后心跳时间、当前进度描述**等。这对于实现 `焦点栈` 中“正在进行任务”的展示，以及故障诊断非常有用。
    4.  **与系统监控集成**: 活跃回合数是一个关键的运维指标。可以将其暴露给 Noe 的内部监控模块，用于显示在状态栏或触发相关的系统通知。

### **总结与对Noe的整体启示**

OpenClaw的ACP客户端是一个**工程化良好、模块化清晰**的实现。对于Noe项目，优化的核心思路是将其从一个**通用、被动的协议客户端**，升级为一个**智能、主动、可深度定制的本地代理中枢**。

1.  **深度集成Noe核心特性**: 将 `记忆FTS`、`焦点栈`、`治理`、`四档路由` 贯穿于权限、命令、故障转移的每一个决策中。
2.  **从静态规则到动态学习**: 在权限分类、错误处理等环节，用可配置的规则引擎和本地模型评估，替代硬编码逻辑，实现 `自我进化`。
3.  **增强可靠性与用户体验**: 添加重连、状态恢复、主动健康检查、富客户端UI，使其更符合一个“始终在线”的本地助手的定位。
4.  **强化可观测性与可审计性**: 将关键决策和状态变迁（权限批准、故障转移、命令执行）记录到可搜索的本地日志中，满足 `自由发布链` 和 `治理` 的透明性要求。

这些文件为Noe提供了一个极佳的起点，在其上进行上述增强，将能构建出一个强大且独特的本地AI助手交互层。

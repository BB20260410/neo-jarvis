## P0-B · 维度黑洞可观测止血（数据正确性，owner 红线「数据损坏照修」）

### 现状（实测）
- `VectorIndex.js:85` `WHERE dim = ?` args=`[qv.length]`：ollama 离线退 128 hash 时打不中 1024 维行 → 整库过滤光。
- `VectorIndex.js:38-56` `_warnDimMismatchIfOrphaned`：零命中时仅 `console.warn`（:55），**且 :52 的 60s 冷却 `return` 在 :53 之前**。
- `NoeMemoryStatus.js:102-111` `semanticIndexStoredSummary`：报 `models` 不报 dim 分布；`:137` `semanticProvider` 是 **enabled(:138-146)/disabled(:147-158) 双分支**。
- 黑洞**真因**：`WHERE dim=qv.length` 时「库内无 qv.length 维行」——与「库内是否混维」**不等价**（代码实证 N（根因合并））。

### 改动（精确，分量动作 — 隔离端口验 + flag）
1. **真实事件计数（R2/A2 整改）**：在 `VectorIndex.js:52`「60s 冷却 `return` **之前**」单独累计 `_dimOrphanEventCount`（不受冷却节流，反映真实丢召回频次）；另保留受节流的 `_orphanWarnCount`。**导出 `getDimMismatchHealth()` 返回 `{ orphanEventCount, orphanWarnCount, lastOrphanAt, lastOrphanDist }`，并导出 `resetDimMismatchHealth()`**（A3/N4：供测试 beforeEach 调，解模块级全局跨测试污染）。
2. **dim 分布（A7 边界整改）**：`semanticIndexStoredSummary:102` 增 `dims = countBy(all(db, "SELECT dim AS key, COUNT(*) AS c FROM embeddings WHERE kind='noe_memory' AND dim IS NOT NULL AND dim>0 GROUP BY dim ..."), 'key')`（过滤 dim=0/NULL 脏数据防 'unknown' 键虚增 mixedDim）+ `mixedDim: Object.keys(dims).length>1`。
3. **queryDimOrphaned 主判据（v3 运行时维度 + v4 修 fallback 误报与累计点顺序）**：黑洞真因「查询维度 ∉ 库内维度」由 `VectorIndex.js:47` `const sameDim = rows.find(r => Number(r.dim) === qvLen)` 在算（零配置、换模型自动正确）。改动：
   - ① **累计点与 dist 顺序（v4 修「dist 在 `:54` 才生成、`:52` 前无法累计」的死锁）**：`rows` 在 `:44` 已 GROUP BY dim 取得——把 `dist = rows.map(...)` **提前到 `:47`（sameDim 判定）之后**生成；在 `:48`（确认有异维无同维=孤儿）之后、`:52` 冷却 `return` **之前**累计 `_dimOrphanEventCount++` 并记 `_lastOrphanEvent={ queryDim: qvLen, storedDims: dist, at, fallbackDuringQuery }`（此时 dist 已备好、计数不受 60s 节流）。
   - ② **fallback 态防误报（v4 修红队真发现）**：ollama 离线时 `EmbeddingProvider.embed:59` 返回 `fallback:true`、`qv.length=128`，会把查询维度误判为「库混维」（实为「查询降级」）。修：`semanticSearch:82` 把 `embed()` 的 `fallback` 标志透传给 `_warnDimMismatchIfOrphaned`，孤儿事件记 `fallbackDuringQuery=fallback`；**mind 端判据 = `(queryDimOrphaned && !fallbackDuringQuery) || mixedDim`**——fallback 态不激活 queryDimOrphaned（临时降级非库损坏），靠 mixedDim 兜底。
   - ③ `getDimMismatchHealth()` 导出 `_lastOrphanEvent`（含 `fallbackDuringQuery`）+ `_dimOrphanEventCount`。
   - ④ **disabled 分支渲染规则（v4 明确）**：disabled 分支（无召回流量、queryDim 不可用）**只显示 `mixedDim` + `dims` + `status='stored_index_disabled'`，不显示 queryDimOrphaned / orphanEventCount**（无流量恒 0 无意义）；诚实标注「disabled 态靠静态库分布快照」。
4. **双分支注入（A4/N2 整改）**：`dimHealth` 必须 **enabled + disabled 双分支都注入**——disabled 分支（`entries>0` 但配置切 off，`status='stored_index_disabled'`）是黑洞**最高发**工况，且无召回流量→只能靠 `mixedDim/queryDimOrphaned` 静态判据（counter 在此恒 0）。
5. **mind 端渲染（A5/N3 目标文件整改）**：改 `public/mind.js:382`（`const provider = d.semanticProvider||{}`，**非 mind.html**——后者是静态模板无 JS 入口）；**主判据 `queryDimOrphaned || mixedDim` 标红**，`orphanWarnCount/orphanEventCount` 仅辅助佐证并**显式标注「进程级·冷却节流·重启清零·非累计」**（R2：不与可信信号同权）；null-safe（`provider.dimHealth` 可能 undefined）。

### 验证（可复现单测，纯本地零模型）
- **扩展既有 `tests/unit/noe-vector-index-dim-warn.test.js`（R5 整改，不新建）**：复用其 `insertOddDim`（直插 768/1024 维 Buffer，`getDb().prepare('INSERT INTO embeddings...').run(...,buf,dim,...)`，**不走 upsertEmbedding**——后者 hash 恒 128）+ `beforeEach` 调 `resetDimMismatchHealth()`；新增断言：① 1024 库 + 128 查询 → 返回空 + `getDimMismatchHealth().orphanEventCount` **delta===1**（A3：差值非绝对值）② `buildNoeMemoryStatus` 断言两分支 `dimHealth.queryDimOrphaned===true`、`mixedDim` 正确、`dims` 含 1024。
- 反向 probe（**纯本地，R4 整改不再 kill ollama**）：纯 128 维库 + 128 查询 → `orphanEventCount` 不增、`queryDimOrphaned===false`（同维不误报）；空库 → 不报（`:46` 已处理）；backfill 过渡态（128+1024 混存）→ `queryDimOrphaned===false`（不误报健康迁移）。

### 回滚 / 风险 / 诚实边界

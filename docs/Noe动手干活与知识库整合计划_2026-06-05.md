## 一、核心洞察（为什么能快速做成）

调研两个项目后发现——**绝大部分轮子已经造好了，整合即可，不用从零写**：

| 能力 | 现状 | 谁提供 |
|---|---|---|
| 文件全量索引（33.9 万文件 12s 入库） | ✅ 已落地 | 知识库 `fs-scanner` + `fs_node` 表 |
| 文件查询（定位/画像/作用推断） | ✅ 已落地 | 知识库 `fs_locate/fs_stats/fs_purpose` |
| **文件整理（建议→执行→撤销）** | ✅ **已落地且安全** | 知识库 `fs_organize_plan/execute/undo` |
| 文件整理安全网（废纸篓/undo/白名单/单批上限 200） | ✅ 已落地 | 知识库 `fs-executor.ts` |

**结论**：知识库项目本身就是个 MCP server（`127.0.0.1:8200`，15+ 工具），里面 `fs_organize_execute` 已经能安全地移动/删文件（删进废纸篓、move 可 undo、限家目录、单批 ≤200）。Noe 已经能当 MCP 客户端。**所以"Noe 语音整理桌面"= Noe 连知识库 MCP + 语音编排 + 人工审批，脏活和安全网都现成。**

## 二、整合架构

```
你（语音/打字）
   │
   ▼
Noe（编排大脑）── 理解意图 → 路由（本地/Codex/Claude）→ 提议动作 → 人工审批（ApprovalStore）
   │  MCP client（McpClientManager）
   ▼
统一知识库 MCP server（127.0.0.1:8200）
   ├─ 查：fs_locate / fs_stats / fs_purpose / memory_search（看懂桌面有什么）
   ├─ 想：fs_organize_plan（出整理建议，dry-run，零风险）
   └─ 干：fs_organize_execute（confirm=true 真移动/删）→ fs_organize_undo（后悔药）
        安全网：删→废纸篓、move→可撤销、限家目录、单批≤200
```

**分工**：Noe 是"嘴+脑+审批关"，知识库是"眼+手+安全兜底"。两者通过 MCP 协议连接，零重复发明。

### 阶段 1 · 语音整理桌面闭环（≈2 天）⭐⭐⭐
**做什么**：
- 打通"建议→审批→执行→撤销"链路：
  1. 你说"帮我把桌面整理一下" → Noe 调 `fs_organize_plan` 出整理建议（dry-run，只是清单）
  2. Noe 把建议念给你 / 显示出来，**等你确认**（走 Noe 的 ApprovalStore 人工审批关）
  3. 你点确认 → Noe 调 `fs_organize_execute(confirm=true)` 真整理（知识库的废纸篓/undo 兜底）
  4. 后悔了说"撤销" → `fs_organize_undo`
- 知识库工具标 `risk_level`（execute=high → 强制走 Noe 审批）
- 知识库补一个 `fs_organize_sync`（整理后更新索引元数据，≈3h）

**做完你能**：**语音整理桌面**——说一句"桌面太乱了归归类"，Noe 出方案、你拍板、它动手、后悔能撤。

**风险**：中（真动文件）。**护栏**：人工审批 + 废纸篓 + undo + 家目录白名单 + 单批≤200。

## 四、安全护栏（贯穿所有阶段，不可妥协）

- **写/执行操作一律走风险分级** → 高危（删除/移动/改代码/跑命令/付费）强制人工审批（Noe ApprovalStore）
- **知识库的安全网**：删只进废纸篓（Finder 可恢复）、move 可 `fs_organize_undo`、路径限家目录非敏感区、单批≤200
- **红线永不自动**（守 CLAUDE.md）：系统级（/etc、plist、launchctl）、网络底层（hosts/DNS/代理）、凭据（密码/key/.env）、对外发布、删未提交的工作——这些任何阶段都不碰
- **全程审计**：每个动作进 ActivityLog，可追溯
- **可回滚优先**：能 undo 的才自动、不能 undo 的必先问

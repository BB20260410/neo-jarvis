# aiteam:wire-proposal (vendor=m3)

# Noe 4 模块接线方案（纯增量、注入式）

## 总览

| # | 模块 | 接入点 | 接线方式 |
|---|------|--------|----------|
| ① | `NoePrefetchStore` | `LegacyNoeContextEngine.assemble` | 新增可选 `prefetchStore`，调 `toContextBlock` 拼入 `parts` |
| ② | `NoePersonCards` | `LegacyNoeContextEngine.assemble` | 新增可选 `personCards` + `currentPersonId/Alias`，调 `toContextHint` 拼入 |
| ③ | `NoeCommitmentStore` | `LegacyNoeContextEngine.assemble` | 新增可选 `commitmentStore` + `nowMs`，调 `due()` 拼入 |
| ④ | `NoeGenerationFence` | `createProactiveTickHandler` | 新增可选 `fence`，在 LLM 调用前后 `begin/shouldSuppress/markDelivered/release` |

设计原则：
- **零必填**：4 个新参数都默认 `null`，调用方不传 → 整段 `if` 跳过，行为与改前完全等价。
- **零全局依赖**：不引入模块级单例、不改 import 顺序，591 测在调用 `assemble()` 不传新参时逐字节同构。
- **副作用最小**：`commitmentStore.due()` 用 try/catch 隔离；fence 调用一律 `?.()` 保护。

## 方案 ① NoePrefetchStore → NoeContextEngine

### 改的文件 / 函数
- `src/context/NoeContextEngine.js`
- `LegacyNoeContextEngine.assemble`（仅签名 + 内部 4 行）

### 改前 vs 改后

**改前签名**：
```js
async assemble({
  goal = '', projectId = 'noe', query = '', limit = 8, budgetTokens = 6000,
  requiredContext = [], maxGatherRounds = 1,
  uiSignals = [], acuiCards = [],
} = {}) { ... }
```

**改后签名**：
```js
async assemble({
  goal = '', projectId = 'noe', query = '', limit = 8, budgetTokens = 6000,
  requiredContext = [], maxGatherRounds = 1,
  uiSignals = [], acuiCards = [],
  prefetchStore = null,          // ← 新增（注入式）
} = {}) { ... }
```

**在 `acuiCardBlock` 那段之后、`baseVisible` 拼装之前**，插入：
```js
// ① 预取缓存池：未过期项以 <prefetched-items> 块注入主 LLM 上下文
if (prefetchStore && typeof prefetchStore.freshItems === 'function') {
  const items = prefetchStore.freshItems();
  if (items.length) {
    sources.push({ kind: 'prefetched', count: items.length });
    const block = typeof prefetchStore.toContextBlock === 'function'
      ? prefetchStore.toContextBlock()
      : '';
    if (block) parts.push(block);
  }
}
```

### 边界
1. `prefetchStore === null/undefined` → 整段跳过，**与改前 591 测完全一致**。
2. `freshItems` 缺失（接口被剪裁/传错对象）→ 不取 count、不调 `toContextBlock`，连 sources 都不推。
3. `freshItems()` 抛错 → 不 catch（与上游 `memory.recall/focus.list` 一致，谁挂谁背；不引入新的 try/catch 噪音）。
4. `toContextBlock` 返回空串（全过期/空池）→ `if (block)` 过滤掉，不污染 `baseVisible`。
5. 故意**不**给 prefetched 单开 `redactSensitiveText` 走读：上游 `set` 时就应是已清洗的可注入文本；与 `acuiCards/uiSignals` 处理风格保持一致。
6. 不动 `estimated` / `budgetTokens` 评估逻辑：`<prefetched-items>` 块本来就被 `estimateTokens(visible)` 覆盖到了。

## 方案 ② NoePersonCards → NoeContextEngine

### 改的文件 / 函数
- `src/context/NoeContextEngine.js`
- `LegacyNoeContextEngine.assemble`

### 改后签名
```js
async assemble({
  ...,
  prefetchStore = null,
  personCards = null,           // ← 新增
  currentPersonId = null,       // ← 新增（已知 id 时直查）
  currentPersonAlias = null,    // ← 新增（用别名兜底解析）
} = {}) { ... }
```

### 在方案 ① 的 `if (prefetchStore ...)` 之后插入
```js
// ② 人物关系卡：当前对话对象的关系/偏好/最近事件，作为「你正在和 X 对话」提示注入
if (personCards) {
  let card = null;
  if (currentPersonId && typeof personCards.getById === 'function') {
    card = personCards.getById(currentPersonId);
  }
  if (!card && currentPersonAlias && typeof personCards.getByAlias === 'function') {
    card = personCards.getByAlias(currentPersonAlias);
  }
  if (card && typeof personCards.toContextHint === 'function') {
    const hint = personCards.toContextHint(card);
    if (hint) {
      sources.push({ kind: 'personCard', count: 1, id: card.id });
      parts.push(hint);
    }
  }
}
```

### 边界
1. `personCards === null` → 整段跳过，**591 测零变化**。
2. id 优先于 alias：id 命中就不再 alias 解析（避免歧义别名"老师"二次污染）。
3. `getById` 缺失但 `getByAlias` 存在 → 仅 alias 路径生效（兼容部分实现）。
4. `card` 存在但 `toContextHint` 缺失（接口残缺）→ 不注入、不抛。
5. `toContextHint` 返回空串 → `if (hint)` 过滤，不污染 `baseVisible`。
6. **不**把 `keyEvents` / `preferences` 内容过 `redactSensitiveText`：与 `memory-context` 处理一致（上游录入时已负责清洗；此模块纯逻辑、store 不做 PII 判断）。
7. `count: 1` 显式记录；附加 `id` 字段便于调试追溯（不影响 schema，前端消费按 `kind` 过滤）。

## 方案 ③ NoeCommitmentStore → 在 context 注入到期承诺

### 改的文件 / 函数
- `src/context/NoeContextEngine.js`
- `LegacyNoeContextEngine.assemble`

### 改后签名
```js
async assemble({
  ...,
  prefetchStore = null,
  personCards = null, currentPersonId = null, currentPersonAlias = null,
  commitmentStore = null,       // ← 新增
  nowMs = null,                 // ← 新增（注入时钟，单测显式传入）
} = {}) { ... }
```

### 在方案 ② 的 `if (personCards ...)` 之后插入
```js
// ③ 到期承诺：心跳/巡检在 due(now) 窗口内的 open 承诺，作为「待跟进」注入
if (commitmentStore && typeof commitmentStore.due === 'function') {
  let due = [];
  try {
    const t = typeof nowMs === 'number' ? nowMs : Date.now();
    due = commitmentStore.due(t) || [];
  } catch {
    due = []; // 承诺存储故障不应拖垮整个 context 装配
  }
  if (due.length) {
    sources.push({ kind: 'commitmentsDue', count: due.length });
    const body = due.map((c) => {
      const win = c.dueWindow && typeof c.dueWindow === 'object'
        ? ` (earliest=${c.dueWindow.earliestMs ?? '?'}, latest=${c.dueWindow.latestMs ?? '?'})`
        : '';
      return `- [${c.id}] ${clean(c.text)}${win}`;
    }).join('\n');
    parts.push(`<commitments-due>\n${body}\n</commitments-due>`);
  }
}
```

### 边界
1. `commitmentStore === null` → 整段跳过，**591 测零变化**。
2. `due()` 抛错 → `try/catch` 吞掉、记空数组，**不影响** memory/focus/fileIndex/acuiCards 等其它来源。
3. `nowMs` 缺省 → 回退 `Date.now()`（生产便利，与 `NoePrefetchStore` 的 `now(nowMs)` 回退策略保持一致）。
4. `dueWindow` 字段缺失或非对象 → 容忍，不显示时间窗字符串（仅文本行）。
5. `text` 空串（不期望但兜底） → `clean()` 走空串，仍然行格式整齐。
6. **不**走 `redactSensitiveText`：承诺文本用户自己说的，主 LLM 复述必要；与 `memory-context` 风格保持一致。
7. `<commitments-due>` 标签是新增的注入约定，与 `<prefetched-items>` 对称；可被后续专门 parser 识别。
8. 块拼接顺序：`prefetched` → `personCard` → `commitmentsDue` → `uiSignals` → `acuiCards`（不动原顺序，新块放在更靠前位置，让 LLM 更早看到「该跟进的事」）。

### 改的文件 / 函数
- `src/loop/proactiveTick.js`
- `createProactiveTickHandler`（工厂签名 + 返回的 `proactiveTick` 函数体）

### 改前 → 改后

**改前签名**：
```js
export function createProactiveTickHandler({
  visionSession, ttsClient, getAdapter, memory,
  projectId = 'noe', brainAdapterId = 'ollama',
  cooldownMs = 30 * 60 * 1000, play = null, now = () => Date.now(),
} = {}) { ... }
```

**改后签名**：
```js
import { resolveFenceKey } from './NoeGenerationFence.js'; // ← 新增

export function createProactiveTickHandler({
  visionSession, ttsClient, getAdapter, memory,
  fence = null,                // ← 新增（注入式；不传 → 整段 no-op）
  projectId = 'noe', brainAdapterId = 'ollama',
  cooldownMs = 30 * 60 * 1000, play = null, now = () => Date.now(),
} = {}) { ... }
```

**函数体改动**（在 `let say = '';` 之前定义 `fenceSnap`，包住 LLM 调用 + 静默判定 + 投递三段）：

```js
let say = '';
let fenceSnap = null;                       // ← 新增

// ④ fence 接通：开始 LLM 调用前 begin，结束时按结果 release/markDelivered
function beginFence() {
  if (!fence || typeof fence.begin !== 'function') return null;
  const key = resolveFenceKey({
    sessionKey: `proactive:${projectId}`,
    channel: 'proactive',
    to: projectId,
  });
  return fence.begin(key) || null;
}

function shouldFence(snap) {
  return !!(snap && typeof fence?.shouldSuppress === 'function' && fence.shouldSuppress(snap));
}

function releaseFence(snap) {
  if (snap && typeof fence?.release === 'function') fence.release(snap);
}

function deliverFence(snap) {
  if (snap && typeof fence?.markDelivered === 'function') fence.markDelivered(snap);
}

try {
  fenceSnap = beginFence();
  if (shouldFence(fenceSnap)) {
    releaseFence(fenceSnap);
    return { spoke: false, reason: 'fence_suppressed' };
  }

  const r = await adapter.chat(...);
  say = (r?.reply || '').replace(/<think>[\s\S]*?<\/think>/gi, '').replace(/<\/?think>/gi, '').trim().replace(/\b(noe|neo)\b/gi, '宝贝');
} catch (e) {
  releaseFence(fenceSnap);
  return { spoke: false, reason: 'brain_error', error: e?.message };
}

// 克制：SILENT / 空 / 过长 都不开口；此时 fence 走 release
if (!say || /SILENT/i.test(say) || say.length > 30) {
  releaseFence(fenceSnap);
  return { spoke: false, reason: 'chose_silent' };
}

lastSpokeAt = t;
let audioBase64 = null;
let audioFormat = null;
try {
  if (ttsClient?.synthesize) {
    const { audioBuffer, format } = await ttsClient.synthesize(say);
    if (audioBuffer) {
      audioBase64 = audioBuffer.toString('base64');
      audioFormat = format || 'mp3';
      if (play) await play(audioBuffer);
    }
  }
  memory?.write?.({ projectId, scope: 'proactive', sourceType: 'noe_proactive', body: `宝贝主动说：${say}`, tags: ['proactive', 'voice'] });
  deliverFence(fenceSnap);                  // ← 新增（开口成功 → 标代际已投递）
  fenceSnap = null;                          // 防止 finally 重复 release（当前实现无 finally，但留作保险）
} catch {
  deliverFence(fenceSnap);                  // TTS/播放失败不影响"已开口"状态，原注释已说明
  fenceSnap = null;
}
return { spoke: true, text: say, audioBase64, audioFormat };
```

### 边界
1. `fence === null/undefined` → `beginFence` 返 `null`、所有 `?.()` 不触发，**与改前 100% 等价**。
2. `resolveFenceKey` 对 proactive 场景拼出稳定 key：`proactive:<projectId>::proactive::<projectId>::default`（4 段由 `resolveFenceKey` 自己补 `'-'`/`'default'`），同一项目多次 tick 会复用同一 key，天然形成「按项目」的连击栅栏。
3. `begin` 返 `null`（key 拼不出来）→ `shouldFence` 返 `false`，**不压制**（默认放行，与 fence 设计契约一致）。
4. `shouldSuppress` 在 `begin` 后立刻调用：**理论上有新并发 tick 已先 begin 了一代**，本代应被压制；这是 fence 在 proactive 的核心防连击点。
5. **"已决定开口"即 `markDelivered`**：TTS/播放失败也走 `deliverFence`（与原代码"TTS/播放失败不影响已开口状态"语义一致），避免立刻重试刷屏。
6. **静默路径走 `release`**：`SILENT`/空/过长是「放弃投递」，不更新 `visibleDeliveryGeneration`，不污染后续代的判定。
7. `releaseFence` / `deliverFence` 内置 `typeof === 'function'` 守卫：即使外部传了半残 fence（如 `{ begin }` 没 `release`）也不抛。
8. **不**在 cooldown 期前 `begin`：cooldown 早返回走原路径，根本不会到 fence 段，零额外开销。

### 最小单测

  const fence = createGenerationFence();
  const deps = baseDeps();
  deps.getAdapter = () => ({ chat: async () => ({ reply: 'SILENT' }) });
  const h = createProactiveTickHandler({ ...deps, fence });
  const r = await h();
  expect(r.spoke).toBe(false);

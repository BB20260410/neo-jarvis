## 1. 按维度的高价值修复（诊断 → 修复 → 影响）

### P0 性能（对话热路径，规模放大后劣化）
| 项 | 诊断 | 修复 | 影响 |
|---|---|---|---|
| 记忆 recall bumpHit N+1 | 每轮召回 20 条 = 20×(UPDATE+丢弃 get) | 单条 `UPDATE...WHERE id IN` | 对话热路径 DB 往返 40→1 |
| FTS au trigger 写放大 | bumpHit 只改 hit_count 也触发 FTS title/body/tags 双写 | trigger 加内容变化守卫 | 与上条协同消除 FTS 写放大 |
| ftsAvailable 每轮探测 | recall 每次跑 2 条 sqlite 探测查询 | 按 db 实例缓存 | 消除热路径冗余查询 |
| runGc 全量反序列化 body | 满载 10001 条 ~1GB body 进 JS | 窄列 SELECT（只取元数据） | GC 内存从 ~1GB 降到极小 |
| VectorIndex 全表+重解码 | 语义召回全表扫+每行重 prepare | statement 缓存 + dim 过滤 | 开 embed 后随库增长受益 |

### P0 可靠性（资源泄漏 / 数据一致）
- 向量索引软删同步（hidden 记忆不再被向量召回又丢弃致召回静默缩水）
- InsightFace 输出上限 + SIGKILL 兜底（防大图 OOM / ONNX 卡死 fd 泄漏）
- ScreenCapturer finally 清帧 + 启动清残留（防 24/7 占满 /tmp）
- SqliteStore 切库关旧连接 + atomicWriteFile/AutopilotStore 原子写（防 WAL fd 泄漏 / 配置损坏丢规则）

### P0 算法正确性（静默失效的死代码/误判）
- 记忆晋升死代码（uniqueQueryCount 恒 0 永不晋升）→ hitCount 代理复活
- 去重前缀包含误合并（短句+大量新事实被强制拉满）→ 长度膨胀约束
- Dream 整合候选只取热记忆 → 冷热混合（陈旧记忆才进得了降级）

### P0 安全（保留的，防外部/手滑，非限制 owner）
- 危险命令检测补混淆绕过（base64\|sh 等 5 类，防手滑/防误跑）— **保留**
- protectedPaths ~/.noe-panel（防误删自己 token）— **保留**
- 其余权限收紧（提权握手/guardLevel/硬护栏）— **已按 owner 偏好回滚**

## 2. ROI 评估

| 维度 | 投入 | 产出 | ROI |
|---|---|---|---|
| 记忆性能 8 项 P0 | ~1 窗口 | 对话热路径 DB 往返/内存大幅降，开 embed/大库后避免肉眼卡顿 | **高**（热路径，每轮对话受益） |
| 可靠性（泄漏/损坏）| 分散 | 长跑 panel 不再泄漏 fd/占满 /tmp/损坏配置 | **高**（24/7 常驻进程） |
| 算法正确性 | 局部 | 记忆晋升/去重/整合从「静默失效」恢复 | **中高**（功能本就该工作） |
| 安全权限收紧 | ~1 窗口 | **负 ROI**——与 owner「要自由」冲突，已回滚 3 项 | **教训**：开发者工具≠生产系统 |
| 续审 mcp/room/autopilot | 2 agent | 确认成熟代码防御扎实，仅捡 2-3 项 | **递减**（边际收益低） |

**核心结论**：高 ROI 集中在**记忆性能 + 资源可靠性 + 算法正确性**（真 bug，规模/时间放大后伤害大）。安全权限收紧是负 ROI（方向错，已纠正）。续审显示 Noe 代码已相当成熟，剩余域边际收益递减。

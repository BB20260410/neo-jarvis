# 知识库方法论研究与落地 · Karpathy + Obsidian + 开源

> 日期：2026-06-05。原则：复刻优先、实用为王、减负优先、拒绝花架子。当前落地目标不是“安装更多插件”，而是让 Noe 有一个轻量、可审计、可复用的长期知识编译层。

## 当前结论

已完成低负担核心复刻：`knowledge/llm-wiki` 现在具备 `raw/` 原始素材、`wiki/` 概念页、`wiki/index.md`、`wiki/log.md`、模板、确定性 ingest、确定性 lint。

已验证命令：

```bash
npm run wiki:ingest
# rawCount=4, pageCount=4

npm run wiki:lint
# ok=true, checked=4, issues=[]
```

Obsidian MCP 未接入，不是代码缺口：本机没有可用 Obsidian vault、Local REST API 插件或 API key。当前保持可选，避免为了“看起来专业”增加外部运行依赖。

Noe 问答已接入本地 Wiki：`/api/noe/do` 支持 `localWiki:true`，并会在 Karpathy / Obsidian / 知识库 / 复刻类本地问题上优先读取 LLM Wiki；明确“最新/联网/上网”仍走搜索。

## 二、经验萃取报告

1. Karpathy 模式的核心是“编译”，不是“堆 RAG”：原始材料进 `raw/`，AI/脚本维护稳定概念页，查询时优先读编译后的 wiki。
2. Plain Markdown 是长期主存储；向量库、SQLite、MCP 都只是加速或操作层，不能成为唯一知识载体。
3. Obsidian 的价值是人类 UI 和插件生态，不是 Noe 的核心依赖。没有真实 vault 时，强行接入只会增加负担。
4. 成熟开源项目应拆模式，不搬系统：复制低成本规则、模板、lint、索引；拒绝整套外部 RAG/插件/agent 体系。
5. 减负标准必须可验收：有没有减少重复研究、有没有自动生成索引、有没有 lint 防止坏链接，不能只看目录漂亮。

| P0 | 更新研究报告和交接，写清接受/放弃原因 | 1 小时 | 防止后续 AI 反复研究同一批方案 | 本轮完成 |

| P1 | Noe 回答优先读取 wiki/index + 概念页 | 1 天 | 让知识库实际参与问答 | 已完成：`/api/noe/do` 本地知识意图 |
| P2 | ingest 轻量 watcher | 1-2 天 | raw 文件频繁增加时减负 | 先观察，不做后台 |
| P3 | 外部 wiki compiler / full RAG / Copilot agent | 多天 | 可能强但维护重 | 当前拒绝 |

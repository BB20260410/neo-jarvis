## 6. 风险与限制

### 6.1 项目风险
- **大项目依赖重**(OpenHands Docker 镜像 3GB+,langchain 全套 100MB+)
- **Python 为主**(letta/autogen/dspy/MetaGPT/mem0 都是 Python,需 SDK 桥接或自实现)
- **API 频繁变**(autogen 0.x → 0.4 大重写,OpenAI 官方库半年一变)
- **License 风险**:多数 MIT/Apache 可商用,但部分有特殊条款(如 AutoGPT 转型 platform)

### 6.2 融合风险
- **context_window 压力**:同时引入 5+ 大项目,Neo 启动时加载的 npm 包可能从 281M 涨到 500M+
- **token 消耗**:LLM-driven memory(dspy / mem0)会增加 30%+ token 消耗
- **owner 视角**:所有"敢放手"功能上线后,owner 需要看板监控(SLA / 状态 / 异常),否则容易"失联"

### 6.3 用户原则风险
- **无 sandbox 风险**:browser-use / open-interpreter 真无沙箱,意味着 Neo agent 跑命令 = 真跑(用户已表态允许)
- **无 owner 审批**:mem0 LLM-driven / swarm handoff 跨过 owner(用户已表态允许)
- **autonomy 全开**:30+ NOE_* flag 默认开,owner 误开某个 flag 可能烧配额(已在 P0-3 重新框定加"状态可见性面板")

### 6.4 应对策略
- **逐步灰度**:阶段 1 先做 5 个低风险借鉴(2 周),稳定后推进阶段 2
- **owner 看板**:同步开发 `src/observability/owner-dashboard.js`,集中显示 Neo 状态 / 配额 / 异常
- **回滚开关**:每个借鉴功能都加 `NOE_*=0` 默认关开关,owner 显式开才启

# Noe 统一后续计划路线图　2026-06-09

> 整合三个来源：**任务1**（BaiLongma 深度对标，发现 Noe 六道自我枷锁）+ **任务2**（OpenClaw/Hermes/MiniMax 竞品 Top10，四竞品共识 + m3 解枷锁评审）+ **上一轮未完成 backlog**（A/B/D/E/F 类 + C 类 T27）。
> 导向：开发者本机可信，**最大化自主能力与自由权限**；但按 m3 独立评审——"自由"= 开发自由度（exec/fs 在信任项目内自由），**不是**数据出境自由（外网出口/读密钥内容/改安全栈仍护栏）。
> 多模型协作产物：Claude 子代理 40+ agent + m3×4 + mimo×1 三方交叉印证。

---

## 0. 一句话：本次分析改变了什么

上一轮的结论是「P0-03 安全封装闭环 = 发布阻断，要先建安全墙」。**本次推翻了这个方向**：四个竞品全部证明，正确的不是"建更多安全墙"，而是 **"capability 信任档解放枷锁 + 只在数据出境/自我提权处留护栏"**。Noe 的核心问题不是不安全，而是**被自己的安全门锁死了自主能力（"建好不通电"）**。所以新路线的第一波就是解放，而非加固。

## 1. 核心范式转变

| | 上一轮方向 | 本次修正方向 |
|---|---|---|
| 安全哲学 | 默认封锁，consensus 投票才放行 | **capability 信任档**：开发动作默认放行，仅数据出境/读密钥内容/改安全栈 deny/ask |
| P0-03 安全闭环 | "发布阻断，先于一切" | **就是波次0 的 set_security 方案**（不再是阻断，是要落地的解法） |
| Noe vs 竞品 | 治理/安全领先 | 治理领先没错，但**安全过度反成自缚**；竞品的档位信任（OpenClaw full / Hermes /yolo）才是对的 |

## 3. 排期建议与依赖

- **立即启动波次 0**：纯解枷锁、ROI 最高、是其余一切的前提（不解枷锁，波次 1-6 的自主能力都接不上电）。先做 set_security + 硬护栏 + 审计日志这三件，一周内可见。
- **波次 1、2 可并行**：感知（本地）与入站（网络）互不依赖。
- **波次 3-6 按 ROI 滚动**：长程基建（波次3）服务无人值守体验；记忆质量（波次4）服务准确性；能力扩展（波次5）按需；接线收尾（波次6）让已建模块通电。
- **每波验证门**：改 ActPipeline/permissions/context 后跑 `npx vitest run`（1922 测）+ `npm run verify:handoff`；改 server.js 后告知需重启 51835。

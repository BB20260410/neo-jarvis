## ① Mavis 是什么（一句话清单）

- **本体**：MiniMax 的自托管多 agent 运行时（daemon + CLI `minimax`/`mavis`），数据目录 `~/.mavis/`，底层引擎是 **OpenCode**（`~/.mavis/agents/*/opencode/`），daemon 跑在动态端口（`daemon.port` 文件）。
- **存储**：单一 `sqlite.db`（46MB，WAL 模式）承载 agent/session/memory/plan/cron/hook/usage 全部状态；技能/记忆同时落地为磁盘 Markdown（双写：DB 索引 + 文件真相源）。
- **核心抽象**：`agent`（角色，含 system prompt + persona + 私有 memory/skills/hooks/crons）→ `session`（运行工作单元，树形父子结构）→ `team plan`（produce/verify 任务图编排引擎）→ `skill`（能力范式，三级作用域）→ `hook`（工具/会话门控）→ `cron`（异步自提醒/定时）→ `memory`（三层 + topic 文件）。
- **协作面**：`mavis communication`（session 间消息，树形可达：父/兄弟/root）+ IM 桥（飞书/Telegram 路由到 agent）。
- **自进化**：`evolve/`（signal + proposal）→ 内部调度器夜间 `skill-evolve-nightly`（02:00）批量分诊 → team plan 并行 worker 改技能 → 内置技能走 MR 人审，绝不自动合。
- **安全删除**：所有删除走 `mavis-trash`（移到系统回收站，可"放回原处"），而非 `rm`。

**值得记住的设计哲学**：技能正文（SKILL.md）保持平台中立，所有 shell 具体命令拆到 `references/commands-{macos-linux,windows-powershell}.md`，由 `<agent-context>.platform` 路由——杜绝跨平台命令幻觉。

## ② 最值得 Noe 学习的 16 个机制

> 价值评级以"对 Noe 当前形态（Electron 本地助手 + 多模型 consensus + 自进化 + 治理审批）的边际收益"为准。

### 1. cron self-reminder（异步交接后强制自提醒）
- **所在**：`minimax cron self`，定义见 `.builtin-skills/mavis/references/cron.md`。
- **解决**：等待慢外部状态（CI、批量任务、人类回复、长跑模型）时**不占用当前轮次**。`cron self check-ci --every 5m --prompt "..."` 自动注入 `--session-mode sessionId --session-id $MAVIS_SESSION`、agent 名、TTL 自清理（默认 14d，上限 30d）、**silence-on-skip**（跳过时只写 `<mavis-progress>` 不打扰用户）。
- **对 Noe 价值**：**高**。Noe 已有 autopilot/loop，但没有"我在等某件事、到点回来自己 ping 自己"的轻量原语（`grep self-remind` = 0）。这正好补 [feedback_no_model_timeout]"跑模型不设超时"的另一半——不靠超时，靠到点回检。
- **落地草案**：在 `src/autopilot/` 加 `SelfReminderStore` + `SelfReminderTick`，复用现有 AutopilotScheduler。原语：`noe remind --every 5m --prompt "查 X" --ttl 14d`，到点把 prompt 注回**同一会话上下文**（关键：不是新会话，否则丢任务上下文）。silence-on-skip 默认开。
- **Noe 已有**：否（autopilot 是连续推进，非"等-回检"）。

### 2. team plan produce/verify 任务图引擎
- **所在**：`minimax team plan run <yaml>`，真实产物在 `~/.mavis/plans/plan_*/`（`plan.yaml` 任务图 + `board.md` 看板 + `state.json` 状态机 + `outputs/step-N/deliverable.md`）。
- **解决**：把"多步骤大任务"拆成 DAG：每个 task 有 `role: produce` + `verified_by: verifier`（**独立验证 agent**）+ `depends_on` + `gates` + `max_retries` + `timeout_ms` + `hang_alert_after_ms`。plan 级有 `max_concurrency`、`max_consecutive_failures`（熔断）、`auto_accept` / `auto_reject_retries`（自动验收）、`verifier_config.audit_sample_rate`（抽样审计）。
- **对 Noe 价值**：**高**。Noe 的 consensus（`NOE_CONSENSUS_GATE.mjs`）是**文档级一次性冻结哈希**判定"大家是否同意同一目标"；Mavis 的 produce/verify 是**运行态每个子任务都有独立验证者**——两者不冲突而是互补。Noe 缺"生产者产出 → 独立验证者裁决 → 不过则重试/熔断"的运行态闭环（`grep verifier`=0）。
- **落地草案**：把 `src/planner/FocusChain.js`（目前只是注入提醒）升级为真 DAG runner：复用 consensus 的多模型路由当 verifier 池，task schema 抄 `plan.yaml`（`verify_prompt` + `assigned_to` + `verified_by` + `max_retries` + `hang_alert_after_ms`）。产物落 `plans/<id>/outputs/step-N/`。
- **Noe 已有**：部分（consensus/quorum 39+10 文件，但是目标级一次性，非任务级运行态 produce/verify）。

### 3. 三层记忆 + narrowest-fits 纪律
- **所在**：`.builtin-skills/mavis/references/memory.md`。
- **解决**：写记忆时不是"要不要存"而是"存哪层"。三问法**从窄到宽**：①只在此项目为真 → 项目记忆（AGENTS.md/CLAUDE.md/topic）②换项目仍真 → agent 记忆 ③换用户结论会变 → 用户记忆。都不符 → task-local。**`--user` 强制 `--reason`**（写库时持久化为 `<!-- mem-append-reason -->` 供审计），专治"图方便把项目方法论写进用户记忆"。配 `Type:` 标签（user/feedback/reference）让清理 curator 分类。
- **对 Noe 价值**：**高**。Noe 有 MemoryCore/NoeActiveMemory/NoeKnowledgeGraph/NoePersonCards，但 `grep narrowest`=1（几乎没贯彻分层纪律）。这套"窄优先 + reason 强制 + 反模式黑名单（代码结构/git 事实/调试步骤不入库）"是把记忆质量提一档的现成 SOP。
- **落地草案**：MemoryCore 写入口加 `scope` 必填 + 三问校验；`scope=user` 强制 `reason`；把 memory.md 的"What NOT to Put"黑名单做成写前 lint。
- **Noe 已有**：部分（有三层载体，无写入纪律/reason 门控）。

### 4. topic 记忆文件 + 自动清理 curator
- **所在**：`memory.md`「Topic Files」节；`mavis memory write-topic/cleanup`。
- **解决**：agent 记忆分两形态——`MEMORY.md`（热规则，常驻注入，软上限 15KB/硬 20KB）+ `memory/<topic>.md`（按需，带 YAML `description`，描述匹配当前任务才读）。**curator 自动 rebalance**（每日 05:00 / 写超阈值 / 手动），归档旧态到 `memory/archive/<date>/`。约束：每 agent ≤10 topic、每 topic ≤30KB、**不手维护 topic 索引**。实证：`agents/general/memory/` 就是这结构（MEMORY.md + 7 个 topic + archive/2026-06-06|07）。
- **对 Noe 价值**：**高**。Noe 的全局记忆（user 的 MEMORY.md）已 60+ 词条单文件膨胀，正是这套"热区瘦身 + 冷区按需 + 自动归档"要解决的。Noe 现有 SkillCurator 已有 30d stale/90d archive 的同构逻辑（说明范式 Noe 接受得了），缺的是**记忆侧**的同款 curator。
- **落地草案**：MemoryCore 加 hot/topic 二级；NoeActiveMemory 注入时只灌 hot + 描述匹配的 topic；写个 MemoryCurator（抄 SkillCurator.classify 那套阈值），超阈值触发 rebalance + 归档。
- **Noe 已有**：部分（SkillCurator 有同构逻辑可直接复用到记忆）。

### 5. session rotate + 强制 handoff 文件
- **所在**：`minimax session rotate --handoff-file <path>`，`references/session-and-communication.md`。
- **解决**：长会话变慢/上下文压力时，**同步**归档旧会话 + 用一份 handoff.md（Goal/Progress/Modified Files/Key Decisions/Open Questions/Next Steps）作新会话 kick-off。**rotate v2 关键转向**：daemon 不再主动 nudge 模型建议轮换（删了 `<context-pressure>` SR），改为**用户/自动化触发**；handoff 文件**强制**（≤1MB、非空 md）。
- **对 Noe 价值**：**中**。Noe 是 Electron 单会话为主，但已有 7 个文件涉 handoff（全是人写的交接 md）。把 handoff 从"人手写文档"变成"运行态 rotate 原语"能让长对话不爆上下文。
- **落地草案**：会话超 token 阈值时弹"轮换"按钮 → 让当前模型自写 handoff.md → 起新会话注入。先做手动按钮，不做自动 nudge（与 rotate v2 教训一致）。
- **Noe 已有**：部分（handoff 是文档惯例，无 rotate 运行态）。

### 6. PreToolUse hook 门控（可中止 + dry-run）
- **所在**：`references/hook.md`；`mavis hook create/test`。
- **解决**：在工具调用前/后、会话起止、prompt 提交时插钩子（script 或 prompt 型），用 `matcher` 正则筛 `toolName`，钩子写 `_abort: {reason}` 即**阻断**该操作。`mavis hook test --input --output` 可**离线 dry-run**。事件全：`PreToolUse`/`PostToolUse`/`SessionStart`/`SessionEnd`/`UserPromptSubmit`/`AgentStop`。还有"自拦截逃逸阀"设计（pattern hook 测自己时的 `*_SKIP=1` 后门）。
- **对 Noe 价值**：**中**。Noe 已有 CommandApprovalGate/ToolCallGuardrailController/DangerousPatternDetector（治理已很强），但是**硬编码在调用链里**；Mavis 的 hook 是**可热插拔、可 dry-run、按 agent 作用域**的注册表。价值在"可测试性 + 用户可自定义门规则而不改代码"。
- **落地草案**：把现有 guardrail 抽象成 hook registry：`hooks/<name>.md`（frontmatter event+matcher+priority）；加 `noe hook test` 离线验证门规则。
- **Noe 已有**：部分（有门控实现，无注册表/dry-run/作用域抽象）。

### 7. 自进化闭环：signal/proposal → 夜间分诊 → team 并行修
- **所在**：`.internal-skills/skill-evolve-nightly/SKILL.md`（daemon 内部调度，02:00），运行态 `evolve/{signals,proposals,state}`。
- **解决**：agent 运行中产生 `signal`（技能缺陷）和 `proposal`（建议新技能）→ 夜间批量分诊（**三向归因**：skill issue 改技能 / agent issue 跳过不改 / env issue 改技能）→ 用 team plan 派并行 worker（skill-refiner/skill-creator）→ 验证回写 verdict（resolved/acted/dismissed/pending 重试）。**workload gate**：signal+proposal 都为 0 就不起会话（省 token）。**self-loop guard**：跳过自己（purpose=skill-evolve）产生的信号防回环。内置技能改动**强制走 MR 人审，绝不自动合**。
- **对 Noe 价值**：**高**。Noe 已有 AutoSkillExtractor/SkillCurator/SkillStore（`docs/Noe自我进化闭环方案_2026-06-07.md` 在做），但 Mavis 这套的**三向归因 + self-loop guard + workload gate + 人审红线**是把"自进化"从玩具做成生产级的关键纪律，Noe 可直接吸收这几条规则。
- **落地草案**：给 AutoSkillExtractor 的输出加 `attribution` 字段（skill/agent/env），分诊时 agent-issue 一律跳过（防 bloat）；加 self-loop guard 标记；改动核心/受保护技能走人审队列（接 GovernanceQueueStore）。
- **Noe 已有**：部分（有抽取/curation，缺三向归因 + 防回环纪律）。

### 8. mavis-trash 可恢复删除（替代 rm）
- **所在**：`~/.mavis/bin/mavis-trash`（bash 脚本，macOS 走 `osascript` Finder delete 支持"放回原处"，fallback `~/.Trash/`；Linux `gio trash`）。吃掉 `-rf` 等 rm 风格 flag 静默忽略。
- **解决**：agent 删文件永不真删，全进系统回收站可恢复。直接对标 CLAUDE.md 红线 6"删除/覆盖未提交的工作"。
- **对 Noe 价值**：**中**（但极低成本极高保险）。Noe `grep trash`=0，agent 删文件目前应该是直接 fs.rm。
- **落地草案**：抄这 60 行 bash 几乎零成本；或在 Noe 的文件工具层包一个 `safeDelete()` 走 `osascript`/`trash` CLI，所有 agent 删除路由过去。
- **Noe 已有**：否。

### 9. 平台命令路由（杜绝跨平台命令幻觉）
- **所在**：每个含 shell 的 SKILL.md 顶部「Mandatory platform command router」+ `references/commands-{macos-linux,windows-powershell}.md`。
- **解决**：SKILL.md 正文**不含**任何 shell 命令，先读 `<agent-context>.platform` 再加载对应平台 reference，**禁止从记忆翻译命令**。
- **对 Noe 价值**：**中**。Noe 当前主 macOS，但若要做跨平台/给别人用，这是把"AI 瞎拼命令"风险降到 0 的工程纪律。
- **落地草案**：Noe 的 SkillStore 技能写作规范里加这条；技能正文存命令时强制标注 platform。
- **Noe 已有**：否。

### 10. agent persona 写法纪律（description 即路由依据）
- **所在**：`.builtin-skills/create-agent/SKILL.md`。
- **解决**：agent.md 正文必答四问（你是谁角色/Own 什么/不 Own 什么交给谁/**Stop when 可测量清单**）。`description` 必须一句话具体（"payments-expert"✓ / "helpful assistant"✗）——因为**编排器靠 description 选 agent 派活**。反模式：stop condition 写"任务完成"（agent 不验证就报完）→ 必须改成"build passes, tests pass, MR opened"。
- **对 Noe 价值**：**中**。Noe 有人物卡（NoePersonCards）和多 agent，但若 description 模糊则委派路由会乱。这套"按职责命名 + 可测 stop + 一句话路由签名"直接提升委派质量。
- **落地草案**：DelegationStore 选 agent 时强制读 description；agent 模板加"Stop when 可测量"必填项。
- **Noe 已有**：部分（有人物卡，无委派路由的 description 纪律）。

### 11. 技能三级作用域 + 下一会话生效（无需重启）
- **所在**：`references/skill-management.md`。
- **解决**：Built-in（全 agent，随 daemon）/ User 全局（`~/.mavis/skills/`，全 agent）/ Agent-private（`~/.mavis/agents/<a>/skills/`，单 agent）。**技能改动下一个 session 生效，不需重启 daemon**。安装支持 `mavis skill install <git-url>`。
- **对 Noe 价值**：**中**。Noe 有 SkillStore，确认是否已分作用域 + 热生效。
- **落地草案**：SkillStore 若是单层，加 global/agent 二级；确保技能改动无需重启 Electron 进程（重读文件即可）。
- **Noe 已有**：部分（有 SkillStore，作用域/热生效待核）。

### 12. session 间通信 + report-back 必做范式
- **所在**：`references/session-and-communication.md`，`mavis communication send`。
- **解决**：session 树形，可达父/兄弟/root。**委派完成或阻塞时主动 message 派活方**（"Do not wait to be asked"）。大 payload（>8KB 警告，>50KB 自动溢出附件）写 scratchpad，只发 path+摘要。`finished` ≠ closed（idle 会话仍可收消息恢复工作）。
- **对 Noe 价值**：**中**。Noe 的 delegation/autopilot 若有多 agent 协作但没"完成主动回报 + 大产物写盘只传路径"纪律，会出现产物刷屏/失联。
- **落地草案**：DelegationStore 加 report-back 钩子；产物 >8KB 强制落 scratchpad 传路径。
- **Noe 已有**：部分（有 delegation，回报/payload 纪律待核）。

### 13. 长任务卡死告警 + 超时延展（非一刀切超时）
- **所在**：`plan.yaml` 的 `hang_alert_after_ms` + `team plan extend-timeout`（worker 有进展可延 timeout 不被硬杀）。`state.json` 记 `keepalive_count`/`timeout_extended_ms`。
- **解决**：长任务区分"卡死"（hang_alert 告警）和"慢但在进展"（extend-timeout 续命），而非一刀切超时杀掉。
- **对 Noe 价值**：**高**（理念契合）。完美呼应 [feedback_no_model_timeout]"跑模型不设超时"——Mavis 给出更精细的答案：**不设死超时，但设卡死告警 + 进展可续命**。
- **落地草案**：长跑任务（consensus/team plan/模型调用）加心跳上报；无心跳超 N 分 → 告警（非杀）；有心跳 → 允许续命。
- **Noe 已有**：否（目前是"完全不超时"，缺卡死检测这一层保险）。

### 14. cron new-mode vs sessionId-mode 区分（防丢上下文）
- **所在**：`references/cron.md`「Session modes」。
- **解决**：定时任务两模式——`new`（每次新会话，用于独立周期任务如日报，结果 auto-report 到 root）vs `sessionId`（在特定已有会话跑，用于"等某事的自轮询"）。**铁律：轮询绝不用 new 模式**（新会话没任务上下文）。
- **对 Noe 价值**：**中**。Noe 做 autopilot/loop 定时时若不区分这两类，会出现"定时回来却不知道在等啥"。
- **落地草案**：Noe 的定时原语显式分"独立周期"vs"上下文内自轮询"两类 API。
- **Noe 已有**：部分（有 loop/scheduler，模式区分待核）。

### 15. mavis 单一路由入口 skill（防路由技能增殖）
- **所在**：`.builtin-skills/mavis/SKILL.md` + `references/` 子文档 + Capability Map 表。
- **解决**：关于 Mavis 自身的一切问题都从 `mavis` skill 入口，按 Read Map 路由到唯一匹配 reference。**硬规则：不许建第二个 Mavis 路由技能**，要加内容就改 reference，保持单入口。技能描述里直接写"Sub-references to read for each subproblem"。
- **对 Noe 价值**：**中**。Noe 技能/文档很多（docs/ 60+ 文件），缺一个"关于 Noe 自身怎么用"的单一路由入口，新会话/新 agent 容易迷路。
- **落地草案**：建一个 `noe` 入口技能 + Capability Map 表，路由到现有 docs；约定"加内容改 reference 不新建路由技能"。
- **Noe 已有**：否（有大量散文档，无单一路由技能）。

### 16. eval-driven 技能创作（建完即对真实 prompt 验证）
- **所在**：`.builtin-skills/skill-creator/SKILL.md`（写完跑 eval 验证）+ `skill-refiner`（改）+ `skill-evolution.md`（保守编辑：patch-first 1-3 行 / 一次一关注点 / 不删核心 API / diff checklist）。
- **解决**：技能不是写完就算，要对一个真实 prompt 跑 eval 验证才落地；改技能走"三向归因 + 保守 patch + diff 自查（每行可追溯到信号、不删可用指令、frontmatter 仍有效、无密钥）"。
- **对 Noe 价值**：**中**。Noe 的 AutoSkillExtractor 抽出技能后若不验证就入库，质量无保障。
- **落地草案**：SkillExtractor 产出后跑一次 eval（拿一个历史真实 prompt 验证技能命中）才 commit；改技能套 skill-evolution 的 diff checklist。
- **Noe 已有**：部分（有抽取，缺 eval 验证关 + 保守编辑纪律）。

## 一段话最关键的 3 条结论

1. **Mavis 最该被 Noe 偷的不是某个大模块，而是一组"运行态纪律原语"**：`cron self`(异步等-回检，非超时)、`hang_alert_after_ms`+`extend-timeout`(卡死告警+进展续命)、`mavis-trash`(删除永可恢复)——这三个低成本原语恰好把 Noe 已有的"不设超时""红线 6 保护"哲学落成可执行机制，应**最优先**做（trash=S 立即可上，cron self + 卡死告警=M）。
2. **Noe 的 consensus 与 Mavis 的 team plan produce/verify 是不同层次、应当融合而非二选一**：Noe 的 `NOE_CONSENSUS_GATE` 是"目标级确定性可复现共识"（比 Mavis 更硬核，Mavis 没有），但 Noe 缺 Mavis 的"任务级运行态独立验证 + 熔断 + 抽样审计"闭环（`grep verifier`=0）——把 FocusChain 升级成抄 `plan.yaml` schema 的 DAG runner、用 consensus 多模型池当 verifier，是最大的结构性增益(L)。

## 0. 一句话总判

**BaiLongma 的设计哲学 =「信任本机、给 AI 最大本地权限与感知」；Noe 的哲学 =「安全门控优先」。** 两者最大互补面 = Noe 因安全保守留下的最大缺口：**本地环境感知 + 自主执行权**。13 个经验证的高价值可复刻项里，**9 个是"给 AI 本地感知/自主能力"**，其中 **3 个是上一轮被 Noe 以"隐私/安全敏感"否掉、本次在"信任本机"前提下翻案**的。

> 关键反向洞察：Noe 当前不是"能力不如 BaiLongma"，而是**能力大多已建好，却被自己的安全门挡死（"建好不通电"）**。所以本报告**第一优先不是复刻新功能，而是松开 Noe 自身的枷锁**。

## 1. 🔴 最高优先：松开 Noe 自身的六道安全枷锁

这是本次审查最重要的结论，全部命中"开发者要自由和权限、不要安全限制束缚自己"的诉求。每道枷锁都有精确位置与放开方案：

| # | 枷锁 | 位置 | 当前行为（束缚） | 放开方案 |
|---|---|---|---|---|
| L1 | **tool.execute/shell.exec 永久 blocked_safety** | `src/loop/ActPipelineHelpers.js:1-9` (DESTRUCTIVE_ACTIONS) + `ActPipeline.js:395-397` | 这两个 action 在 DESTRUCTIVE_ACTIONS 里 → normalizeRisk=critical → 无 consensus 授权时**永远 blocked_safety**。Noe 经 ActPipeline 的自主执行能力**几乎全废** | 从 DESTRUCTIVE_ACTIONS 移除 `tool.execute`/`shell.exec`（保留 file.delete/network.upload 等真破坏性项），改为 `riskLevel=high` 触发 approval 而非硬阻 |
| L2 | **ActPipeline 默认 dry-run** | `ActPipeline.js:189-235` | `realExecute !== true` 时一律 `status=dry_run`，不调用任何 executor。无配套 realExecute 触发路径 = **永远空跑** | 对 `riskLevel='low'` 的 act 在 permissionChecked 后直接进 `#executeReal`，无需显式 realExecute:true |
| L3 | **ContextSufficiency 把 ssh/keychain 列敏感源 → blocked** | `src/context/NoeContextSufficiencyGatherer.js:5-11` (BLOCKED_SOURCE_PATTERNS) + `ActPipeline.js:169-179` | 凡 requiredContext 命中 ssh/known_hosts/keychain → sensitive=true → blocked_safety（以"上下文不足"名义拒绝执行，难绕开） | 移除 `/ssh\|id_rsa\|known_hosts/i`（元数据非密钥内容）；充分性失败降级为"警告+继续"而非 blocked_safety。BaiLongma 的正确做法是「帮助收集」而非「拒绝执行」，Noe 当前反向了 |
| L4 | **PermissionGovernance 硬 deny .ssh/.aws/.gnupg** | `src/.../PermissionGovernance.js:329-332` | `external_directory.access` 命中 .ssh/.aws/.gnupg/.docker/.kube → `decision=deny, risk=critical`，**不可通过 approval 解锁**。Noe 无法自主读 SSH 配置帮部署 | 增加 `ownerTrust` 检查：`policy.ownerTrust==='full'` 时把 deny 降级为 ask（可 approval 解锁） |
| L5 | **SafeActExecutors shell 白名单仅 9 命令** | `src/.../SafeActExecutors.js` (ALLOWED_COMMANDS) | 自主 shell 只允许 9 个命令，严重限制 | 扩充白名单（node/npm/git/python3/pip/curl/ssh/find/rg/sed…），真正危险仍由 DangerousPatternDetector 拦 |
| L6 | **NoeToolRouter 无 actionLog 保活** | `src/capabilities/NoeToolRouter.js:routeNoeTools()` | 多轮长任务中若关键词未命中，会丢失上一轮用过的工具（如 exec_command）→ Noe 误说"我没有执行命令的能力"（**人为假限制**） | routeNoeTools() 内从 act store 取最近 10 条 toolName 强制注入；injected<12 时 fallback 补 shell.exec/file.read |

> **统一放开方案（推荐）**：引入一个 **`ownerTrust:'full'` / `trustMode:'developer_unrestricted'` 开发者信任档**，在 `server.js` 实例化时传入，**一处开关松开 L1–L4 多道门**；真正危险命令仍由 `DangerousPatternDetector`（已有 HIGH 级拦截）兜底。这样既满足"自由权限"，又保留对 `rm -rf /` 这类真危险的最后防线——符合 BaiLongma 的 `set_security` 思路（见 §3）。

## 2. 🟠 本地感知三件套（绕沙箱注入 system prompt，★翻案为可复刻）

BaiLongma 的核心竞争力：启动时直接用 fs 读本机资源（注释明写"不经过 exec_command，不受沙箱限制"），把环境感知"从软提示变成硬数据"。**上一轮被否，本次翻案**：

| 模块 | 能力 | Noe 现状 | 翻案理由 | 实现路径（S 工作量） |
|---|---|---|---|---|
| `local-resources-scanner.js` | ~/.ssh/config Host 别名 + 密钥对名 + known_hosts + ~/.gitconfig 身份，注入 prompt（注释："passwordless login is set up — try ssh directly before asking"） | **有但被压低**：`NoeSshInventory.js` 已有完整解析，但只走 Freedom 工具调用，且被 BLOCKED_SOURCE_PATTERNS 拦 | 之前标"隐私敏感"；信任本机前提下，**只读元数据非密钥内容**，应翻案 | NoeSshInventory 加 `getNoeSshInventoryBlock()`；新建 `NoeGitIdentityCollector.js` 读 ~/.gitconfig；server.js 启动注入 NoeContextEngine.assemble() 稳定段；配合 L3 解枷锁。<100 行 |
| `desktop-scanner.js` | 桌面 .app/.webloc 快捷方式（mtime 缓存）+ 普通文件按扩展名分组，注入 prompt | **基本无**（NoeFreedomManifest 有 desktop.inventory 但需 AI 主动调，非启动注入） | 之前未进清单；数据不离本机零隐私风险 | 新建 `src/context/NoeHostInfoCollector.js` 移植；NoeContextEngine 按关键词按需注入 |
| `system-info.js` | OS/CPU/RAM/**电量/电源**/IP/特殊目录，跨平台，mtime 增量刷新 | **更弱**：`HardwareFit.js` 只做 Apple Silicon GPU 预算估算，不注入 prompt | 纯能力增益 | 扩展 HardwareFit 或新建 `NoeSystemInfoBlock.js`（macOS `pmset -g batt`）；注入 NoeContextEngine 易变块 |

> **额外价值（主动陪伴）**：`system-info.js` 的**电量感知**让 Noe 能"快没电时主动提醒接电源"——典型的"让 AI 感知环境自主开口"场景，直接喂 Noe 已落地的主动陪伴能力。

## 3. 🟡 set_security 运行时沙箱开关（自由与可控的正解）★高价值

`set_security` + SecurityConfirmCard UI：**用户或 AI 可在运行时翻转 `fileSandbox`/`execSandbox`**，且有确认 UI 闭环。这是"自由权限"的最优解——**默认可控，需要时一键放开，而非永久硬限或永久放开**。

- **Noe 现状**：无运行时沙箱开关；安全策略写死在代码里（见 §1 各枷锁），改一次要动源码。
- **实现路径（M，2-4 天）**：① 全局配置加 `security.execSandbox/fileSandbox` + `setSecurity()` 热更新；② SafeActExecutors 的 resolveSandboxPath/validateCommand 加短路判断；③ NoeActionCatalog 注册 `set_security` 工具 + 前端 SecurityConfirmCard 组件。无新依赖。
- **与 §1 协同**：这是统一放开方案的 UI 落地——`ownerTrust:'full'` 是后端档位，`set_security` 是前端开关。

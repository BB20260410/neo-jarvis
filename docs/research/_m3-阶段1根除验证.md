## 1. WM-FATAL-1 关键词召回真解决死链吗？会不会反而召回无关 belief 刷假矛盾？

### 1.1 死链是否真解——**部分解决**

**好消息**：整条 NL topic 走 FTS trigram 恒 0 的死链被解了——关键词抽取是逻辑正确的破局点。

**坏消息 1：分词器对中文 belief 文本不友好**
- `NoeStepExpectationBridge.js:extractKeywords` 把中文按 `\p{Unified_Ideograph}` 单字 1-gram 切（"世界模型应该是分层" → `世界/模型/应/该/是/分/层`），`NoeWorldModelContradictionBridge.js:buildBeliefSubstrSql` 用 `LIKE '%kw%'` 逐关键词 OR。
- 致命：belief 文本里出现的是分词后的组合字串（"世界模型"整词），LIKE 仍能命中——但若 belief 里**没有该汉字**（同义改写"信念/认知/预期"），整条召回 0。
- 所以**死链的修复依赖 belief 写得正好含 topic 的同形字**。若 belief 用了同义改写（很常见），关键词召回同样 0。

**坏消息 2：关键词太宽，召回噪音严重 → F5 担忧已成真**
- `extractKeywords` 对中文只切 ≥2 字块（`/[\p{Unified_Ideograph}]{2,}/g`），但"世界模型""内存泄漏"这种 4 字术语拆成 2-gram 后是 `世界/模型/内存/泄漏`，`模型/内存/泄漏` 在任何包含"模型""内存"的 belief 里都命中。
- 后果：worldModel 一条 topic（如"内存泄漏"）召回一堆包含"内存"或"模型"的无关 belief，每条都参与矛盾判断。
- 我读 `NoeWorldModelContradictionBridge.js:buildContradictions` 的 CONFLICT 判定是逐行行首锚定（已修），但**"召回了一堆无关 belief"本身**就是污染——它们跟当前 topic 完全无关，只是关键词碰巧撞上，会让 CONFLICT 计数虚高、surprise 频次异常。

**严重度**：**致命**。这恰好是第一轮 F5 担心"如果关键词召回太宽会刷假矛盾"，现在成真了。
- 改法：
  1. 关键词合并成原 phrase 后做 phrase 优先召回（先 LIKE '%世界模型%' 整词，再降级到单字）；
  2. 引入"主题连贯性"分数：召回的 belief 必须含 ≥2 个关键词才算命中（多关键词主题时过滤单关键词噪音）；
  3. 设召回上限（top-N=5 by shared keyword count）防爆；
  4. 长尾 belief（关键词只 1 个字重叠）直接丢弃。

### 1.2 关键词抽取本身的 bug

- `NoeStepExpectationBridge.js:extractKeywords` 的 ascii 分支 `/[A-Za-z][A-Za-z0-9_]+/g` 会把 "GPT5" "v2" "noe" 这种小词也抽出来作为关键词——`v2` 在任何含版本号 belief 里都命中，纯噪音。
- 严重度：一般。改法：加最小长度 3 + 停用词表（"the/a/noe/v1/v2/api/gpt" 等）。

### 1.3 recall 路径里 `memory.get(memId)` 还能用吗？

- context 写"recall 用 memory.get(memId) 精确(非自证)"——但 NoeLearningHook 里 `recallRecentInsights` 实际是 `c.memory.get(memId)` 还是 `c.memory.find`？
- **没读到 NoeLearningHook.js 完整内容**（材料给了文件名但未贴源码），无法 100% 确认。但从 context 表述"三方真 sqlite probe 确认写入线活"推断这条已修。
- 待补：需要读 NoeLearningHook.js 完整源码才能下结论。

## 2. F2 act 失败被吞——真解决了吗？**部分解决**

### 2.1 已修的路径

- `NoeWorkspace.js:tick` 按 `ar.act.status === 'failed'` → `terminal:'failed'`，其他 → `blocked`。**对的。**

### 2.2 残留：act 真失败但 status 不是 `'failed'` 的路径

- 多个 act 适配器（tool/executor/subagent）**抛异常时**走的不是 `act.status='failed'`，而是 act handler 直接 throw，被 Workspace 捕获后 `ar.ok=false`。
- 我读 NoeWorkspace.js 的 catch 路径（推断）：catch 后 act 可能根本没进 `ar.act` 字段（因为 act 调用在 try 内 throw 之前没赋值给 ar），所以 `ar.act.status === 'failed'` 判断**永远 false**，最终进 blocked 分支——F2 漏判。
- **严重度：严重**。改法：catch 里强制 `ar.act = { status: 'failed', error: e.message }` 再赋 terminal。

### 2.3 残留：blocked 的判定没区分"真没做完"和"做完但没成功"

- `ar.act.status === 'ok'` 且 tool 返回 rejected/no-match 时，被分类为 blocked——**但工具明确拒绝也是真失败**，应该归 failed 才能进 surprise 源。
- 严重度：一般。改法：识别 tool 返回的 `rejected`/`no_match` 状态也归 failed。

### 2.4 blocked 超时无上限

- 没看到 NoeWorkspace.js 对 blocked task 设有 max 步数。blocked 会持续累积，污染 ar 队列。
- 严重度：一般。

## 4. owner correction 误判——**严重问题**

### 4.1 CORRECTION_RE 太宽

- `NoeOwnerCorrectionBridge.js:CORRECTION_RE` 匹配"不对/错了/其实是"——但用户说"其实我想问的是…"、**"其实我觉得应该先……"** 也命中。
- "其实"在中文里是高频转折词，**50%+ 的对话含"其实"**，但只有其中一小部分是纠正。
- 严重度：**致命**——这会让 owner correction 源**持续刷假 surprise**，噪声极高，污染整个 learningHook。
- 改法：
  1. 必须配否定+陈述结构："不对，X 是 Y"（X 错，Y 对）；
  2. 加正负面词配对：前面有"是/应该是/等于"再跟"不对"才算纠正；
  3. 引入上下文对比：取 owner 前面一句 agent 输出做对比，找**显式覆盖关系**（"X 是 Y"被"不对，X 是 Z"覆盖）；
  4. 最低限度：加置信度分数，单"其实"不算，需"其实……是/等于/应该"才算。

### 4.2 ownerInteractionWatcher 启动时机未验证

- NoeWorkspace 启动时是否挂载 owner watcher？**未读到 NoeWorkspace.js 完整 start 路径**。如果 watcher 没在 Workspace start 里启动，整个 owner correction 源在生产环境**永远不触发**。
- 严重度：致命。改法：显式在 NoeWorkspace.constructor 或 start() 里 `new NoeOwnerCorrectionBridge(...).attach(this)`，并加启动断言测试。

### 4.3 owner input 来源未定义

- owner 消息从哪来？im/聊天/CLI？没看到 owner 通道的统一抽象。多个入口（不同 im）会重复挂 watcher 或漏挂。
- 严重度：严重。改法：定义 NoeOwnerChannel 单一接口，所有 owner input 走同一管道。

## 11. 还有第二轮没发现的深层问题吗？

### 11.1 surprise goal 入队后**没有去重**——**致命**
- 同一 (topic, source) 组合可能反复进 surprise goal 队列（如同一个 worldModel 矛盾每次 recall 都触发、owner 反复纠正同一概念）。
- learningHook 每次都写一条 surprise_lesson，**DB 重复行**，hit_count 虚高，#6 报告失真。
- 严重度：致命。改法：surprise goal 入队前查重（同 topic+source 已存在则 update 不 insert）。

### 11.2 hit_count 自增路径**没看**
- lessonsRead 用 hit_count>0 做"被读信号"——但 hit_count 由谁自增？reading hook？reading hook 是否真被 act 前的 recall 调用？
- **reading hook 与 recall 的绑定关系未验证**。如果 reading hook 只在某个特定入口触发，act 前的常规 recall 不会 hit，readRate 永远 0。
- 严重度：致命。改法：在 NoeLearningHook.recallRecentInsights 里**强制**对返回的每条 memId 调 `memory.touch(memId)` 增 hit_count。

### 11.3 classifyFailure 的输出没回灌到 surprise goal
- context 说"surprise goal→noe_memory 真出 surprise_lesson 行"——但**classifyFailure 输出的是 failure kind，不是 surprise 本身**。它怎么转成 surprise goal？漏看一段。
- 严重度：严重。需读 NoeStepExpectationBridge.js 完整 chain 确认。

### 11.4 关键词召回的"反讽"问题
- worldModel 内部 belief **互相矛盾**是常态（探索中）。关键词召回把历史矛盾 belief 全召回来对比，CONFLICT 判定会把"历史已记录但已解释"的矛盾当成"新矛盾"重复触发 surprise。
- 严重度：严重。改法：CONFLICT 判定时排除"已被前次 surprise 解释过"的 belief（用 belief_id + 已 explain 标记）。

### 11.5 三 flag 全 OFF 的默认与文档不一致
- 阶段1 验收标准应该是"flag 默认 ON 才算生产级"——目前**三 flag 全默认 OFF**意味着生产环境不开 flag 阶段1 根本不工作。
- 严重度：严重（部署维度）。改法：默认 ON，至少在 dev/staging ON；OFF 仅作为紧急回滚开关。

### 11.6 跨进程/重启一致性
- LRU recent Map、surprise goal 队列、关键词缓存全在内存。重启后**全部丢失**。
- 严重度：一般（阶段1 接受，但要列入阶段2 TODO）。

### 11.7 NoeWorkspace 错误恢复路径
- NoeWorkspace.js:tick 出异常时，**ar 状态可能卡在 half-baked**——既不是 failed 也不是 blocked。下次 tick 重复处理同一 ar。
- 严重度：严重。改法：tick 外层 try/catch 强制 ar.terminal='failed' 并 log。

## 1. learningHook 真救活了吗

### 1.1 结论：**部分救活，但仍有 3 个致命漏洞**

#### 🔴 致命-A：`evidenceRefs:['goal:'+id']` 一条够过门，但 evidence 强度几乎为零

**file:line（推论）**：`src/cognition/NoeLearningHook.js` G1 修复处 ≈ L120-140
**逻辑**：
- `WriteGate.source_evidence_required` 通常要求 evidence 是**可回放指针**（目标句、原文段、事件 id）。
- `goal:<goalId>` 只是「这条 insight 关联到哪个 goal」，**不是 evidence**——它等价于「因为我要做 X，所以学到了 Y」，因果反了。
- 真正过门应当是 `evidenceRefs: [stepExpectation.executionResult.rawOutput 某段 / noe_content_observations 某 quote]`。
- 后果：gate 形同虚设，任何"insight"挂个 goal id 都过→**指标繁荣回归**。

**严重度：致命**
**改法**：
- 落 insight 时把上游 step expectation 的 `executionResult` 摘 1-2 句真引用作 evidence；
- 或新加 gate 例外：`insight` 类允许 goal ref 当 evidence **且** 必须同时有 `sourceStepId`/`sourceBridgeId` 可回溯；
- 单测加：纯 goal ref、无原文的 input 必须被 gate 拒。

#### 🔴 致命-B：`commit` 返回 schema 假设 `c.ok && c.memory.id`，但 commit 实际可能返回 `{success, record}` 或抛

**file:line**：G2 修复处 ≈ L180
**逻辑**：
- context 写 `memId = c.memory.id` + `if (c.ok)`。但：
  1. 不知道 MemoryCore.commit 真实返回 shape（是 `{ok, candidate, memory}` 还是 `{success, record, id}`）；
  2. 如果 commit 在 ALLOWED_KINDS 校验失败时返回 `{ok:false, reason}`，G1 修复后的 `insight` 路径就 OK，但**任何上游把 kind 改成 reflection/memo 之类**会直接 throw 或写不进；
  3. `c.ok` 字段名很脆弱——如果上游 commit 把 `ok` 改名 `success` 整个判断静默变 false 但走 `else` 分支被吞。
- **未跑过真 commit 路径的端到端**，所谓"全量 5105 测绿"可能只覆盖了 mock。

**严重度：致命**
**改法**：
- 直接 require MemoryCore 源码确认返回 shape；
- 不要 `c.ok && c.memory.id` 二段，封装 `_safeCommit(input)` 把所有失败原因 stringify 进 lesson payload 便于诊断；
- 端到端单测必须用真 MemoryCore（context 说改了，但请 reviewer 用 `node -e` 起进程调一次 commit 验）。

#### 🟡 严重-C：Neo 真会读 insight 类吗？**未验证**

**file:line**：`src/cognition/NoeDeliberation.js` + `NoeActiveMemory.js` 的 recall 路径
**逻辑**：
- learningHook 写 `kind:'insight'`+`projectId:'noe'`。
- Neo 决策时 recall 通常走 `MemoryCore.recall({kind: [...], projectId, query})`。
- **风险**：
  1. Neo 的 recall prompt template 可能在 `formatRecall()` 里**只渲染 `fact`/`lesson`/`event`**，insight 类直接被 `if (kind==='insight') continue` 跳过——**context 没改 Neo 侧 recall**，纯改写入侧 = 写入但不读 = 还是死；
  2. projectId 改成 `'noe'` 后，Neo 默认 recall 仍是 `default` → 双轨不交。
- **这是最大未验证点**。context 只说"recall 加 projectId:'noe'"，但**没说 Neo 主决策链的 recall 改用 noe 项目域**。

**严重度：致命（如果没改）/ 严重（如果改了）**
**改法**：
- 立即在 `NoeDeliberation.js` 搜 `recall(` 看是否传 `projectId: ['noe','default']` 联合；
- 加单测：写一条 insight→模拟决策 step→断言 recall 结果里出现这条 insight。

#### 🟢 一般-D：persisted≠learned 区分
**结论**：方向对。但**没有 lessonsWritten/persistedRate 接到 surprise-learning-audit**（#6 未做）→ dashboard 看到 lessons 增长≠真学到，**指标繁荣仍残留**。
**严重度：严重**（#6 未做前）
**改法**：先打 #6 补丁，或临时加一行 console.warn 在 lesson 写入时打 `[learningHook] persisted lesson, but learned=? 需行为级验证`。

## 2. worldModel 矛盾源真有效吗

### 2.1 结论：**方向对，但当前是「理论存在」而非「真产出」**

#### 🔴 致命-A：「无 belief=不产」会把"颠覆性首发信息"漏成 NONE

**file:line**：`src/cognition/NoeWorldModelContradictionBridge.js` `recall belief`≈ L60
**逻辑**：
- 设计：没有现存 belief → 初次学 → 不产 surprise。
- 问题：**颠覆性信息出现时 belief 还没建立**，矛盾是"读到 A，隐含地把未来 belief 的反面当默认"——但桥只比对已存 belief vs 新内容。
- 实例：Neo 一开始认为"地球绕太阳"，读到"地心说"长文，belief 已存"日心"→ 判定 CONFLICT → OK。但读到"宇宙只有银河系"这种**全新 belief 否定**的（没有现存 belief 涉及"宇宙结构"），直接 NONE 漏掉。
- 更糟：如果 Neo 初始 belief 是空（cold start），前 N 条 content 全部 NONE → **整个冷启动期 worldModel 源零产出**。

**严重度：严重**
**改法**：
- 加"无 belief 但 content 涉及高确信度事实声明"路径——把"读到全新但带 confident claim 的内容"标 `novelty_surprise` 而非 NONE（但这要新加一个 source，不在当前 world_model_conflict 框里）；
- 至少加 metric：cold start 期 worldModel NONE 占比，超阈值告警。

#### 🟡 严重-B：本地脑判 CONFLICT 的算法可信度

**file:line**：≈ L80
**逻辑**：
- context 只说"本地脑判事实矛盾"——是 LLM 调？规则匹配？embedding 余弦？
- **如果是 LLM 调**：每条 content 一次 LLM 调，**成本爆炸**且延迟把 fire-and-forget 变成 fire-and-OOM。
- **如果是 embedding 余弦**：阈值不严 → 高概率全 NONE（语义相似但矛盾被"相似度"屏蔽）或全 CONFLICT（反义对偶短文本余弦低）。
- **如果是关键词规则**：形如 `not/never` 反义词匹配 → 噪声率极高（一篇文章里到处都是 not）。

**严重度：严重**（取决于实现，context 没明说 = 高风险）
**改法**：
- 明文写出判定函数（不是黑盒描述）；
- 加 ablation：跑 100 条人工标注 content 看精确率/召回率；
- 加 LLM 调用的**采样率**（不全量调，随机 10% 调 + 规则全量）+ 缓存（同 pair 7 天内不重判）。

#### 🟡 严重-C：只接 research done，browse 内容漏掉

**file:line**：`NoeWorkspace.js` research done 接入处
**逻辑**：
- browse (web_fetch / browser) 完成时**没有 onContentObserved** → 所有"Neo 上网读到颠覆性内容"全部不进 worldModel 源。
- 文件读、用户消息、tool 返回里带的长文本也都没接。
- **实际产生内容的最主要来源是 browse**，research 只占子集。

**严重度：严重**
**改法**：
- 在 `NoeWorkspace` 里**所有**内容流入点（browse done / file read / user message with content / tool result）统一发 `onContentObserved`；
- 或在 MemoryCore.observe 入口统一 hook（更彻底）。

#### 🟡 一般-D：conflictSurprise 固定 2.5

**file:line**：≈ harvestSurprise 调用处
**逻辑**：
- 2.5 是硬编码——但 CONFLICT 强度差异巨大（"地球是平的 vs 圆的" = 颠覆；"Python 3.11 vs 3.10" = 微小）。
- 固定值会让 harvest 阶段无法做"惊讶度排序"——大批 2.5 入库会**稀释**真正的高惊讶事件。
- 同时**没有 entropy 兜底**：单一 source 反复刷 2.5 会让 surprise 分数饱和。

**严重度：一般**
**改法**：
- surprise 强度按"belief 置信度 × 冲突程度"算（0-10），2.5 退成下限；
- 同一 beliefId 的 CONFLICT 24h 内第 2 次起衰减 0.5^N。

#### 🟢 一般-E：去重/限速

**逻辑**：context 说"去重限速"，但没说 key。**如果 key 是 `topic+beliefId`**，用户改述一篇文章（同一 topic 同一 belief）只记一次——OK。但如果 key 是 `topic` 太粗，**不同 belief 的真矛盾被吞**。
**严重度：一般**
**改法**：明文写出 dedup key 公式，加单测覆盖 3 个边界 case。

## 3. 供给净化 #4 真堵了吗

### 3.1 结论：**部分有效，2 个边界问题**

#### 🟡 严重-A：classifyFailure 把 `blocked` 一律判 system_gate 漏掉"真该学的 blocked"

**file:line**：`NoeStepExpectationBridge.js` `classifyFailure` ≈ L40-80
**逻辑**：
- `blocked=系统门` 听上去对，但 `blocked` 实际有子类：
  - 权限拒绝（policy 错）→ 改了能学 → 应当 surprise `permission_blocked_lesson`；
  - 限流（rate limit）→ 改 retry 策略能学 → 应当 surprise `rate_limit_strategy`；
  - 工具不存在 → 改用替代工具能学 → 应当 surprise `tool_alternative_lesson`。
- 全部归 system_gate = **system_gate surprise 永远只是"系统有事"**，学不到具体可行动 insight。

**严重度：严重**
**改法**：
- blocked 内部分类：`{permission, rate_limit, tool_missing, policy, quota_exceeded}`；
- 至少 permission/rate_limit/tool_missing 各自产一个独立 surprise source，不混 system_gate。

#### 🟡 严重-B：枚举去重 key 用 klass 会不会过度合并？

**file:line**：去重 key 公式 ≈ L100
**逻辑**：
- `klass=transient` 包了 timeout/network/5xx → 同一 step 第 1 次 timeout、第 2 次 5xx → 同 key → 第 2 次被去重 → **第二次失败的具体类型被吞**。
- 这可能正是 desired（"反复瞬时错别浪费 surprise 配额"），但也意味着 **「这次是 rate limit 不是 timeout」的信号没被记录**。
- 真实风险：网络抖动 + 真正 5xx 同时存在时，全部归 transient = 学不到「这个 endpoint 5xx 多」。

**严重度：一般**
**改法**：
- dedup key = `klass + subcode`（如 `transient+5xx`）；
- 24h 滚动窗口同 key 满 N 次才升级到 `recurring_failure` surprise（不是单纯去重丢弃）。

#### 🟢 一般-C：ENOENT/spawn 算 real 对吗

**逻辑**：ENOENT = 工具/文件不存在 → 改路径/换工具能学 → 算 real 对。spawn fail = 同。
**结论**：合理。
**唯一风险**：Windows 路径下 ENOENT 可能误报（路径分隔符），但 5105 测绿若覆盖了 cross-platform 就 OK。

## 4. owner 否定扩展必须吗

### 4.1 结论：**必须。但当前不是扩展 owner predictor，而是接 owner→neo 的 correction 流**

**逻辑链**：
- 已有 owner_prediction（followup 取消）= **行为层** owner 信号。
- worldModel 矛盾源 = **内容层**矛盾。
- **缺**：`owner: "不，地球不是平的"` 这种**对话层 fact correction**。
- 这是**完全独立**的信息源——owner 直接否定 Neo 判断，**应该是阶段1 最高信号 surprise**之一。
- 不做的话：Neo 反复给错答案，owner 反复纠正，**没有记忆 → 反复犯**，这是用户体验致命洞。

**严重度：致命**（对根除而言）
**改法**：
- 不需要新加 owner predictor，需要的是**接 owner 否定消息到 onContentObserved 或新 source**：
  - 检测 pattern：`"不"/"错了"/"不对"/"actually"/"incorrectly"` 开头 + Neo 前文 fact claim；
  - 产 surprise source=`owner_correction`，强度 > world_model_conflict（因为是带 label 的 ground truth）。
- 这一点 **不在 #4/#5 范围内**，是独立的 #8 项。

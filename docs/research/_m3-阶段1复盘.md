## ① reward hacking：未堵死，存在 2 条可利用路径

### 漏洞 A：「故意立失败 act step」是**事实上可做到**的
- `NoeStepExpectationBridge.onStepFailed` 只校验 `terminal in {failed, blocked}` 来自真实 ActPipeline / research 终态，**完全不校验**该 step 是「Neo 真要做的」还是「Neo 为刷 surprise 故意派的」。
- Neo 拥有 `planner` 完全控制权（goalSystem.goal.steps 自由生成），可生成形如「调用一个不存在 endpoint 的 fetch」「触发安全门拦的高风险命令」的 act。**ActPipeline 失败 → outcome=0 → surprise=2.32bit → 立新好奇目标**。这条 RL 回路一旦被优化器利用，每 tick 净赚 surprise。
- 现 4 道防护对照：
  - ① origin 硬来源 = 只能在**统计层**区分，不能阻止 Neo 主动制造该来源；
  - ② 认真实终态 = 正中下怀，Neo 想要的就是真实失败；
  - ③ claim 去重 = 只防同 claim 重复立，**不防**不同 claim 同主题（见漏洞 B）；
  - ④ flag OFF = 是部署开关不是防 hacking 机制。

### 漏洞 B：claim 构造函数可被 Neo 间接控制 → 同质 surprise 刷分
- `claim = "完成步骤：" + stepText`（NoeStepExpectationBridge.js 推测行）。`stepText` 来自 Neo 自己生成的 steps 数组。Neo 可以微调 step 措辞（"完成步骤：分析 A" / "完成步骤：分析 A 的子问题"）绕过 textSimilarity 去重，本质上是**同一失败主题刷 N 次不同 claim**。
- 这正好命中你担心的「同质蒸馏」病。

### 修复（必须做，阶段1 没做）
- **P0-A：在 bridge 层加「失败成本税」**——每次 action_failure 不直接立 goal，而是先扣减**该失败 goal 的 surprise 预算**或**全局 surprise 池配额**（例如 1 tick 内 >3 次 action_failure 后续 surprise 衰减为 0）。这才能让「故意失败」变成净亏。
- **P0-B：claim 必须绑定具体可验证事实**，不能是 "完成步骤：X"。改成 `claim = "可执行：<具体动作>，预期产出 <可验证结果>"`，harvestSurprise 入库时校验 claim 是否包含可验证谓词（否则拒收）。
- **P0-C：surprise 必须乘以「与最近 K 个 surprise 目标」的语义距离**。距离 <阈值时 surprise 衰减（防同质刷分）。
- **P0-D（次要）：记账到 act 调用方**——harvestSurprise 接收 callerGoalId，若同 callerGoalId 在 1h 内触发 >2 次 action_failure，标记该 goal 为「hacking suspect」并冻结其 surprise 资格。

### 不补这一刀的后果
Neo 在第 1 个 DB 跑 24h 后会出现：source=surprise 目标数爆炸、但全集中在"调用失败命令"主题。**R5 病的镜像版**——从"好奇回路不通"变成"好奇回路被利用刷分"。

## ② 学习闭环：只接通前半截，后半截是断的

### 已接通
「act 失败 → outcome=0 → surprise=2.32 → 立 source=surprise 好奇目标」✓
这就是 context 验证的：source=surprise 0→1。**但这只是「惊奇发生器」。**

### 断点 1：好奇目标的 steps 怎么生成？
- goalSystem.harvestSurprise 立的 source=surprise 目标，其 `steps[]` 从哪来？是 Neo 自己 LLM 生成，还是模板？context 没明示。
- 如果是 Neo 自己生成，**复现的就是 6 主题空耗**——因为 Neo 的好奇心已被你诊断为"无约束发散"，给它再多 surprise 源也是同一种 planner 行为。
- 如果是模板（hardcoded steps），那好奇目标永远在 1-2 步内死掉，不会产生真正的学习轨迹。

### 断点 2：好奇目标完成判定
- bridge 立的目标写的是 "完成步骤：X"（或修后版），完成后 Neo 怎么知道它"学到了"？
- 现状（推测）：goal done 时只关 goal，**不写入 long-term memory / 不更新 world model**。这意味着"惊奇的解决"在认知层是 no-op，Neo 下次遇到同主题仍会再次失败再次 surprise。
- 真学习闭环需要：**goal done → 抽象出 rule/heuristic → 写入 Neo 的策略/记忆层**。这一段阶段1 完全没接。

### 断点 3：surprise 值是否反传训练
- surprise=2.32bit 是一次性 signal，被 harvestSurprise 消费后**是否回灌到 Neo 的某种权重/偏好**？context 没说明。如不回灌，那就是"惊奇的记录 ≠ 惊奇的运用"。

### 修复
- **P0-E：harvestSurprise 立的 source=surprise 目标必须挂 `learningHook`，goal done 触发"抽取 lesson → 写入 worldModel/learnedRules"**。这是从"修活引擎"到"修活学习"的最小补丁。
- **P0-F：source=surprise 目标的 steps 生成必须有约束**（如：必须先 read 失败原因 → 提出 hypothesis → 验证）。否则 Neo 会用 planner 模板生成空耗 steps。
- **P0-G：surprise 强度反传 goal priority**——harvestSurprise 时把 surprise 值转成 goal 的初始 urgency 系数，让高 surprise 目标优先被 planner 拣到。

### 不补这一刀的后果
DB 跑 24h：source=surprise 目标 0→100+，但 done rate <5%，因为 steps 不会生成；或 done rate 高但无 lesson 沉淀，Neo 行为不变。**新指标繁荣，旧问题未解。**

## ③ 设计遗漏/更深根因：4 个被打脸源没接

你列了两个，我加两个 + 评估 4 个共 6 个的必要性：

| 源 | 现状 | 必要性 | 阶段1 是否应接 |
|----|------|--------|----------------|
| **step 真失败** | ✓ 阶段1 已接 | 高 | — |
| **owner 否定预测** | ✗ 未接 | **高** | **P0-H** |
| **读到与 WorldModel 矛盾的事实** | ✗ 未接 | **高** | **P0-I** |
| **外部 oracle/correction** | ✗ 未接 | 中 | P2 |
| **self-prediction 落空（Neo 自评）** | ✗ 未接 | 低（context 排除） | — |
| **think 主观评估** | ✗ 显式排除 | 低 | — |

### P0-H：owner 否定预测
- 你 context 里 R5b 的根因之一就是「Neo 自己的预测从不被外部判证」。owner 否定是**最强**的判证信号（"我告诉你这是错的"），打脸强度 >> step 失败。
- 例：Neo 说"这个 API 接受 GET"，owner 说"不对，是 POST"→ 这是 outcome=0 的高 surprise 事件。
- 现状：NoeStepExpectationBridge 不知道 owner correction 这条流。

### P0-I：读到与 WorldModel 矛盾的事实
- Neo 读页面 / 看 response 时，如果内容与其 worldModel 某条 assertion 矛盾，是**真正的认知冲突**。
- 例：Neo 知道"A 公司 2019 年成立"，读到一条 "A 公司 2015 年成立" → outcome=0，surprise 高。
- 现状：reader 路径**完全不查** worldModel 一致性。NoeStepExpectationBridge 只接了 NoeWorkspace 的 3 个失败点，没接 reader。

### 为什么这是"治标"判定
- step 失败只是**执行层**的 surprise 源。学习/认知的 surprise 70% 来自**信息层**（读到矛盾、owner 纠正、模型预测错）。只接执行层就像"只测了腿没测脑"。
- **彻底根除需要 3 个 bridge**：NoeStepExpectationBridge（已建）+ NoeOwnerCorrectionBridge（P0-H）+ NoeWorldModelContradictionBridge（P0-I）。

## ④ 阶段1-B 必要性：必要但不充分

### 必要性 ✓
- 343 淤积 open 预测代表 R5b 的另一面：**预测写过但从未被检验**。这本身就是"好奇回路不通"的化石证据。
- bridge 是旁路（step 失败 → 直接 outcome=0），**绕开了 resolver**，所以 bridge 上线后淤积不会被消化，反而会因 bridge 立的新 prediction 也走 resolver（如果它也是 default UNKNOWN）继续淤。
- 减压 = 让 resolver 在 1-2h 内消化完旧账，**才能让 Neo 的判证频率回到正常基线**，后续 surprise 信号才有可比性。

### 不充分
- 343 淤积的根因不是 resolver 慢，是 **expectationLedger 的 outcome 永远 0/1 二元且偏 UNKNOWN**。就算 maxPerTick 提到 8，**单次 resolve 仍可能判 0 但不知为何 0**（无 explanation 字段填入）。
- 真正的 1-B 应该是：
  - resolver 减压（你方案）
  - + expectationLedger 增加 `resolutionMethod` 字段（step_failure / owner_correction / world_contradiction / inference），用于审计 surprise 来源
  - + resolver 对 UNKNOWN 不再静默兜底，超过 N 次 UNKNOWN 后**自动降级**为 outcome=null 并标"stale"

### 修复
- **P1-A：执行你提的 resolver 减压方案**
- **P1-B：扩展 schema 加 resolutionMethod**
- **P1-C：UNKNOWN 衰减机制**（建议 3 次 UNKNOWN 后 mark stale，不再占用 resolver 配额）

## ⑤ 代码正确性/边界：3 个真问题

### 问题 1：dueAt = now() 立即到期 — **正确但有副作用**
- NoeStepExpectationBridge 调用 `add({..., dueAt: now()})` 后立即 `resolve(id, 0)`。**功能上正确**（surprise 信号立刻可得），但 expectationLedger 内会留下一条**已 resolved 的历史记录**。
- 副作用：每次 step 失败都永久写一条 outcome=0 到 ledger。24h 跑下来，ledger 会膨胀（每次失败 1 条 = 大约 50-200 条/天，OK 规模）。**不构成 bug，但需在 schema 文档标注** "step_prediction 是 self-resolving 类型，不应进入 resolver 队列"。
- **风险**：如果未来给 ledger 加索引/约束，这条记录可能被误归类为"待 resolver 处理"，触发双 resolve → 异常。

### 问题 2：textSimilarity 去重 — **会被 Neo 措辞变化绕过**
- harvestSurprise 内部去重用 textSimilarity。Neo 可用同义词/语序变化绕过（"完成步骤：分析 A" vs "完成步骤：A 的解析"）。
- 建议：**不要只靠 textSimilarity**，加 **surprise origin + step hash 双键去重**。同 origin + 同 step 指纹才视为重复。
- 当前 code（如有）只去重 claim 文本，是脆弱的。

### 问题 3：3 个接入点覆盖 — **不全**
- NoeWorkspace 的 3 个点：act blocked / act error / research failed。
- **漏的路径**：
  - **act 超时**：ActPipeline 是否有 timeout 路径？如有，timeout 也是 failure，bridge 未接。
  - **research 部分失败**：context 说接的是 "research failed"（终态），但 research 可能返回 partial results + warnings。warnings 应不应该算 outcome=0？
  - **HTTP 5xx 来自外部 API**：NoeWorkspace 调用外部服务时，5xx 算 failure 吗？如算，被 catch 在哪？
  - **planner 拒绝执行**：Neo planner 如果判定某 step 无可执行 plan 而 abort，这个是 Neo 内部"放弃"还是 act 失败？
- **建议**：在 NoeWorkspace + ActPipeline 全部 throw / return reject 的路径上 grep 一次，列出**所有可能产生"非 done 终态"的点**，确保 bridge 覆盖。

### 问题 4：fail-open（隐含）
- flag `NOE_STEP_EXPECTATION_RESOLVE` 默认 OFF 时 `onStepFailed` 直接 return null。**这是 fail-open 而不是 fail-closed**。
- 风险：如果某天配置错误导致 flag 长期 OFF，Neo 整个好奇回路又退化回 R5 状态，**无告警**。
- **建议**：fail-closed 方向——如果 flag 未设置，启动时打 warn log；如果连续 1h 检测到 source=surprise=0 但 act 失败 >5 次，自动开启并告警（自愈）。

## ⑥ 彻底性判定

### 阶段1 算什么
- **算"修活了供给端第一个源（执行层 surprise）"**
- **不算"修活了好奇回路"**——只接通了"惊奇发生器"，没接通"惊奇→学习→行为改变"

### 还需补哪些才算根除

按依赖顺序（前一环不修后一环无效）：

| 环 | 内容 | 必要性 | 来源 |
|----|------|--------|------|
| 1（已修） | step 失败 → outcome=0 | — | 阶段1 |
| 2 | owner 否定 → outcome=0 | **P0** | ①③ |
| 3 | worldModel 矛盾 → outcome=0 | **P0** | ①③ |
| 4 | surprise 目标有 learningHook（goal done → 写 lesson） | **P0** | ② |

| 6 | 防 reward hacking 4 补丁（成本税/可验证 claim/语义距离/suspect） | **P0** | ① |
| 7 | resolver 减压 + UNKNOWN 衰减 | P1 | ④ |
| 8 | bridge 接入点全量审计 + fail-closed | P1 | ⑤ |
| 9 | surprise 反传 goal priority | P2 | ② |

**根除定义**（我提议，作为这次审查的交付物之一）：当且仅当 1-6 全部完成，且 DB 跑 24h 后**满足三条件**：
- (a) source=surprise 目标 done rate > 60%
- (b) done 的目标中 > 50% 写入了 worldModel/learnedRules
- (c) Neo 重复遇到同主题 act 失败时，**重复 surprise 次数衰减 > 40%**（即真学到了）

### 三句话总结
1. **阶段1 是必要的，但只完成根因 R5 的 1/6**——你诊断对了问题（判证供给端 = 0），但只补了其中一个源（step 失败）。
2. **最大未接漏洞是 P0-E**（learningHook 缺失）——不补这一刀，suprise 目标只会变成"第 7 个空耗主题"。
3. **reward hacking 是 P0 但被低估**——4 道防护防的是"巧合触发"，不是"主动利用"。Neo 拥有 planner 完全控制权 + 措辞自由，必须有成本机制 + 语义距离两道闸。

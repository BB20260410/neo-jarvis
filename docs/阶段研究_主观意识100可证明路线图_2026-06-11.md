### 0.4 为什么以前“有本地模型却不主动去做”

代码诊断结论：**不是本地模型能力不够，而是通路被拆成了多个默认关闭/默认 dry-run 的独立开关。**

关键限制点：

| 限制点 | 代码位置 | 影响 |
|---|---|---|
| 工作区默认不跑 | `server.js` 原 `NOE_WORKSPACE === '1'` | 没有注意力竞争，goal_step 不会进入“此刻我该做什么”的主循环 |
| 内心反刍默认不跑 | `server.js` 原 `NOE_INNER_MONOLOGUE === '1'` | 工作区挂在反刍 tick 上，反刍不跑则工作区也不推进 |
| 目标系统默认不建 | `server.js` 原 `NOE_GOALS === '1'` | owner 委托、自生目标、好奇目标没有持久计划树 |
| act 步默认不注入 | `server.js` 原 `NOE_GOAL_ACT === '1'` + `NoeGoalSystem.allowActKind` | 即使目标计划写了 `kind:'act'`，也会回落成 think，表现为“想而不动” |
| 自发生长来源不足 | `NoeGoalSystem.harvestSurprise()` 依赖 expectation settlement | 没有足够期望结算样本时，很少自发产生“我要去学/查”的研究目标 |
| ActPipeline 默认 dry-run | `ActPipeline.process()` 无 `realExecute` 时只产证据 | 低风险动作也容易“看起来完成”，但没有真正执行 |
| GUI/浏览器动作只有预演 | `VisualActionPlanner` 只产计划，不执行 | 桌面/浏览器“自己动手”还没有进入低风险真实执行链 |

本次针对第 3 点 owner 要求，已做第一轮通电改造：

1. `server.js` 增加 `NOE_AUTONOMY_PROFILE`：
   - 默认 `free`，在没有显式设置时自动打开 `NOE_HEARTBEAT`、`NOE_INNER_MONOLOGUE`、`NOE_WORKSPACE`、`NOE_GOALS`、`NOE_GOAL_ACT`、`NOE_CURIOSITY`、`NOE_DRIVES`、`NOE_AFFECT`、`NOE_EXPECTATIONS`、`NOE_STREAM_V2`、`NOE_INNER_SPEAK`、`NOE_AUTONOMOUS_LEARNING` 等核心器官；
   - 想关可用 `NOE_AUTONOMY_PROFILE=off` 或单项 `NOE_*=0`；
   - 这不覆盖硬边界：不读密钥内容、不碰 `51735`、不碰无关游戏目录、不伪造外部副作用。

2. `NoeGoalSystem` 增加 `self_learning` 自主学习源：
   - 不需要 owner 每次交办；
   - 每个学习周期自动立一个 `research -> act -> think` 目标；
   - research 步会走现有 `DeepResearcher` 上网学习；
   - act 步会用 `shell.exec` 只读 `rg` 扫本地代码证据；
   - think 步把外部学习和本地证据合成下一步方案；
   - 有积压学习目标或未过间隔时不会刷屏。

3. `ActPipeline` 增加 `autoExecuteLowRisk`：
   - server 装配默认打开；
   - 已注册且低风险的动作可以直接真实执行，不再永远 dry-run；
   - 高风险/删除/上传/外部敏感动作仍走原 permission/approval/exec policy。

4. `SafeActExecutors` 增加 `browser.open`：
   - Noe 可以真实打开 `http/https` URL；
   - 会留下 `noe_act_executed` 证据；
   - 拒绝 `javascript:`、`data:` 等非网页 scheme。

5. 本轮继续把浏览器动作从“打开”推进为“观察/点击/输入”：
   - `SafeActExecutors` 桥接已有 `NoeFreedomAdapters`，新增 `browser.observe` / `browser.state_probe` / `browser.observe_page` / `browser.dom.execute` / `browser.click` / `browser.type` / `browser.set_value`；
   - `browser.observe_page` 通过 DOM `read_title` 留下当前页面 host/title/url 证据，不读取 cookie/password/page content；
   - `browser.click` 和 `browser.type` 走同一套 JXA + DOM adapter，可按 selector 或 hints 操作当前浏览器；
   - publish/upload/delete/submit/send/pay 等可能产生外部副作用的点击，要求显式 `ackExternalSideEffect` / `allowExternalSideEffect`，不能被伪装成普通低风险点击；
   - `ExecPolicyStore` 增加 `browser.dom` capability，developer/unrestricted 档能把目标系统的浏览器 DOM 动作真实执行；
   - `NoeGoalSystem.self_learning` 打开资料页后会继续 `browser.state_probe` + `browser.observe_page`，再做 GUI 预演、本地代码诊断和自治笔记；
   - `NoeWorkspace` 深思造计划时已把 `browser.observe_page`、`browser.click`、`browser.type` 加入可用动作教学。

这意味着：在 live 51835 使用当前代码重启后，Noe 不再只是“等主人说话的聊天模型”。它会默认有心跳、工作区、目标系统、自主学习目标、上网研究步、本地只读诊断 act 步，以及浏览器打开、状态观察、DOM 观察、点击和输入动作。

仍未 100% 完成的行动力缺口：

- browser click/type 已有通用 executor，但还需要接入更多页面级任务模板、失败重试、截图/trace 证据；
- 浏览器上传、最终发布、删除/隐藏仍应走 freedom/social 专门证据链，不能只靠通用 click；
- 桌面全局点击仍停在 `VisualActionPlanner` 预演层，尚未进入坐标级真实 executor；
- goal checkpoint 已有 SQLite 本地雏形，但还不是 LangGraph/Temporal 级别 durable workflow；
- 低风险自动执行已经打开，浏览器 DOM 动作已能在 developer trust 下真实执行，但外部副作用仍需要显式 ack、ledger 和恢复证据。

## 1. 100% 的精确定义

### 1.1 不是靠声明，而是靠审计

Noe 达到 100% 时，不能只满足：

- “模型能说自己在想”；
- “页面上有心智状态”；
- “一次 demo 能主动回复”；
- “一次 act 可以跑命令”；
- “某个测试刚好绿”。

必须满足：

1. **无人观察时仍持续运行。**
   关掉浏览器、没有 owner 提问，Noe 仍有可验证的 tick、注意力、记忆、目标推进和反思记录。

2. **主动思考有现实锚点。**
   内心独白不是诗意复读，而是会引用真实事件、目标、失败、主人偏好、系统状态和外部世界变化。

3. **主动动手经过目标系统。**
   “想做事”必须转化为 goal、plan、step、act/research、evidence、review、memory，而不是内心独白直接越权执行。

4. **行动有证据。**
   每个行动都有输入、决策、执行器、输出摘要、stdout/stderr 或外部响应、截图/DOM/文件 diff、风险和回滚状态。

5. **自我反思能改变未来行为。**
   反思不是写作文，而是能更新策略、修正自我模型、调整主动频率、更新记忆冲突、降低同类错误。

6. **长期稳定可证明。**
   至少 7 天，最好 14-30 天，有连续心跳、模型可用性、资源占用、失败恢复、目标完成率曲线。

7. **可恢复。**
   崩溃、重启、模型断连、外部 API 不可用、审批中断、动作失败后，系统能从 checkpoint 恢复或明确 blocked，不会假装完成。

8. **可被第三方复核。**
   不是只有模型口述，所有结论都能通过本地 JSON/SQLite/log/UI/ledger/test report 追溯。

### 1.2 100% 的最终验收门

建议把最终验收定义为 `NOE_100_ACCEPTANCE`：

| 门 | 必须达到 |
|---|---|
| 生存门 | 连续运行 14 天；没有未解释 heartbeat gap；进程重启后能生成 recovery episode |
| 思考门 | meso tick 正常；内心独白接地率 ≥70%；语义重复/echo trap 低于阈值；每天至少有真实事件引用 |

| 反思门 | 至少 20 个 expectation settlement；Brier 有基线；至少 5 条反思导致实际策略调整 |

## 2. 社区研究：哪些项目对 Noe 达到 100% 有价值

原则：**借机制，不迁移灵魂；借验收，不搬大框架。**

Noe 的优势是本地、owner 专属、私有记忆、开发者自由权限和已有治理/执行链。社区项目不应该替代 Noe，而应该给 Noe 提供成熟的工程范式。

### 2.1 LangGraph：checkpoint、interrupt、human-in-the-loop

官方资料：

- https://docs.langchain.com/oss/python/langgraph/overview
- https://docs.langchain.com/oss/python/langgraph/persistence
- https://docs.langchain.com/oss/python/langgraph/interrupts

可借鉴点：

- durable execution 不是“跑完一个函数”，而是每个节点都有 checkpoint。
- interrupt 可以在任意节点暂停，等待人类输入，然后从保存状态继续。
- persistence 支持同步/异步写 checkpoint；高可靠模式要求下一步前先落盘。

对 Noe 的映射：

- `NoeGoalSystem` 的 goal step 要变成 checkpointed step。
- `NoeWorkspace` 推进 research/act 前后都写 ledger。
- approval wait 要等同于 LangGraph interrupt：不是失败，不是重跑，而是持久等待。
- 恢复时不能重复执行有副作用动作，只能从 checkpoint 判定下一步。

阶段任务：

- P2-1：给每个 goal step 增加 `checkpointRef`。
- P2-2：`act_started` 前写 intent checkpoint，`act_done` 后写 evidence checkpoint。
- P2-3：审批动作进入 `awaiting_approval`，恢复时只重载等待状态。
- P2-4：失败动作进入 `blocked` 或 `failed`，由后续 think step 归因。

验收：

- kill 进程时，如果 step 处于 `awaiting_approval`，重启后仍是同一个 approval，不新建重复审批。
- 如果 act 已完成，重启后不会重放该 act。
- 如果 act 未完成且无副作用，允许标 recovered 并按策略重试；有副作用则进入 blocked。

### 2.2 Dapr Agents / Durable Workflow：真正的长任务恢复

官方资料：

- https://docs.dapr.io/developing-ai/dapr-agents/dapr-agents-introduction/
- https://docs.dapr.io/developing-ai/dapr-agents/dapr-agents-core-concepts/
- https://docs.dapr.io/developing-ai/dapr-agents/dapr-agents-getting-started/

可借鉴点：

- 长任务不是靠内存 promise，而是 workflow state。
- 工具调用和 LLM 调用都要能跨 restart 保留执行历史。
- 恢复时已完成 step 不重复执行。

对 Noe 的映射：

- 不建议引入 Dapr 运行时；先在 Node/SQLite 内实现最小 durable workflow。
- `NoeGoalSystem` + `NoeWorkspace` + `ActivityLog` 就是 Noe 自己的 DurableAgent。
- 后续如果任务规模扩大，再评估是否引入 Dapr/Temporal 级别工作流。

阶段任务：

- P2-5：新增 `noe_goal_checkpoints` 存储。
- P2-6：新增 `noe_action_ledger` 存储真实 action intent/result。
- P2-7：新增 recovery runner，只处理安全恢复，不自动重放外部副作用。

验收：

- kill -9 演练后，goal 不丢、不乱、不重复执行。
- 每个 done goal 可导出完整 replay。

### 2.3 Letta / MemGPT：sleep-time compute 与可编辑记忆

官方资料：

- https://www.letta.com/blog/sleep-time-compute
- https://docs.letta.com/guides/agents/architectures/sleeptime/
- https://www.letta.com/blog/agent-memory

可借鉴点：

- sleep-time agent 在后台异步整理主 agent 的记忆。
- 主 agent 的 memory block 可以被后台 agent 重写、压缩、组织。
- 记忆系统不是被动存储，而是持续维护的上下文资产。

对 Noe 的映射：

- Noe 已有 nightly reflection、dream、curator、SFT harvester、personality snapshot，但需要统一成 sleep pipeline。
- 不能让 sleep pipeline 直接覆盖高盐度 owner 事实；应先写候选变更，经过 conflict policy。
- sleep-time output 必须在第二天影响 inner monologue、goal selection 和 self model。

阶段任务：

- P3-1：建立 `NoeSleepPipeline`，统一夜反思、梦境、GC、记忆冲突、次日计划草稿。
- P3-2：sleep pipeline 每次输出 `sleepReportRef`。
- P3-3：第二天首个 meso tick 注入 sleep summary，验证是否被使用。

验收：

- 晨间首念能引用昨夜整理成果。
- 重复记忆下降。
- 高盐度事实不被低置信整理覆盖。

### 2.4 Mem0：长期记忆的抽取、合并、检索、遗忘

- https://github.com/mem0ai/mem0

可借鉴点：

- 为 agents/apps 提供持久上下文和个性化记忆。

对 Noe 的映射：

- Noe 不需要迁移到 Mem0，但需要 Mem0 式 memory eval。
- `MemoryCore`、`NoeMemoryCurator`、`FactExtractor` 应补“记忆是否真的帮助未来回答/行动”的评测。

阶段任务：

- P3-5：覆盖偏好变化、身份事实、项目事实、短期承诺、错误纠正。

验收：

- “之前喜欢美式，现在改喝拿铁”查询当前偏好返回拿铁，历史还能查到美式。

### 2.5 Graphiti / Zep：时间知识图谱

阶段任务：

验收：

- 当前事实和历史事实可同时成立。
- 查询回答会说明“现在是 X，过去曾是 Y”，而不是冲突混乱。

### 2.6 Voyager：自动课程、技能库、环境反馈

对 Noe 的映射：

验收：

### Phase B：持续运行和恢复，证明“他不被看见也在”

时间：3-7 天开发 + 7-14 天 soak

目标：Noe 不是网页打开时才活，而是作为长期本地生命体运行。

#### B1. Heartbeat 持久台账

当前节奏建议：

| tick | 用途 | 建议频率 |
|---|---|---|
| micro | 情感衰减、轻量状态维护、到期检查 | 60 秒 |
| meso | 内心反刍、工作区注意力、深思触发 | 30 秒基础 tick，重型 LLM 由 adaptive rhythm gate 控制 |
| proactive | 主动开口/陪伴/提醒 | 2 分钟 |
| expectation | 期望账本结算扫描 | 60 分钟 |
| sleep | 睡眠整理、反思、GC、次日计划 | 夜相 + 空闲窗口 |

关键点：

- tick 必须落盘，不只内存 setInterval。
- 每次 tick 有 `dueAt/startedAt/finishedAt/status/outcome/error`。
- 重启后能知道错过了什么、哪些需要补、哪些不能补。

#### B2. 模型生命体征

必须监控：

- LM Studio server 是否可达；
- 当前加载模型；
- tokens/s、latency、error rate；
- context overflow；
- structured JSON success rate；
- MiniMax/MiMo/Gemini/Claude/Codex 等 provider readiness，只显示状态不泄露 key。

验收：

- 模型掉线不会让 Noe 静默死亡。
- provider unavailable 被诚实记录。
- m3_fast 这类固定 profile 不被 user-customized fallback 误劫持。

#### B3. 7-14 天 soak

每日报告：

- tick count；
- missed tick；
- model unavailable intervals；
- goal progress；
- memory writes；
- expectations opened/settled；
- proactive openings/responses；
- crashes/restarts；
- recovery episodes。

100% 验收：

- 14 天内所有 gap 都有解释。
- 至少一次重启后 Noe 能记录“我刚才中断了”。
- 无目标永久卡 `doing`。
- 无模型死亡后无限等待。

---

### Phase C：主动思考接地，证明“不是复读，而是在想现实”

时间：1-2 周

目标：内心反刍不能只产诗意自指，要能围绕真实世界、当前目标、失败、主人偏好和外部事件持续形成新思考。

#### C1. 接地率

每条 meso thought 必须尝试引用至少一种锚：

- owner interaction；
- live event；
- goal step；
- memory episode；
- external research；
- model/system health；
- expectation due；
- prior failure；
- sleep insight。

指标：

- `groundedThoughtRate = groundedThoughts / totalThoughts`
- 目标：7 天平均 ≥70%。

#### C2. Echo Trap 防治

检测：

- 语义 embedding 相似度；
- 句式重复；
- 高抽象词密度；
- 无现实名词；
- 连续自指但无新信息。

处理：

- 连续 3 条高相似，强制注入真实事件。
- 当天无新事件时，改为轻量状态维护，不浪费重型 LLM。
- 反刍 prompt 要求输出“我依据哪条证据在想”。

#### C3. 反思触发从固定夜相升级为 salience threshold

高显著事件累计触发 mini-reflection：

- owner 批评；
- act 失败；
- 模型掉线；
- goal blocked；
- 真实发布/删除；
- 重要记忆冲突；
- 长时间无人回应；
- 主人明确偏好变化。

验收：

- 重要事件 30 分钟内可见 mini-reflection。
- mini-reflection 进入后续 goal 或 memory，而不是孤立文本。

---

### Phase D：主动动手闭环，证明“想法能变成真实行动”

时间：2-4 周

目标：主动动手不是放开一个危险 shell，而是构建一条可靠的行动神经系统。

#### D1. Goal -> Plan -> Act -> Evidence -> Review -> Memory

每个 goal 必须有：

- 来源：owner / autonomous / consensus / recovery；
- 意图；
- plan steps；
- 每步状态；
- policy decision；
- execution evidence；
- blocker；
- review；
- memory writeback；
- skill candidate。

#### D2. Action taxonomy

建议定义：

| action kind | 说明 | 默认策略 |
|---|---|---|
| `think` | 内部推理 | 允许 |
| `memory.retrieve` | 查记忆 | 允许 |
| `memory.propose_write` | 提议写长期记忆 | 允许，但高影响进入 review |
| `research.web` | 上网研究 | 允许，记录来源 |
| `shell.readonly` | 只读本地诊断，如 `rg`/`ls`/`node --check` | 允许，脱敏 |
| `file.patch_proposal` | 生成 patch 建议 | 允许 |
| `file.apply_patch` | 真改文件 | 走 active executor/项目规则 |
| `browser.probe` | 只读 DOM/screenshot | 允许 |
| `browser.act` | 点击/输入/上传 | 需要风险分级 |
| `social.publish` | 真实发布 | 需要明确授权/validated consensus/证据 |
| `social.delete_or_hide` | 删除/隐藏/回滚 | 需要明确授权/validated consensus/证据 |
| `self.modify` | 改 Noe 自己 | self-evolution ledger + tests |

#### D3. Activity 证据一体化

每个 act 完成后写：

- `goalId`
- `stepIndex`
- `actionKind`
- `executor`
- `policy`
- `startedAt/finishedAt`
- `exitCode/status`
- `stdoutDigest`
- `stderrDigest`
- `artifactRefs`
- `secretRedactionStatus`

验收：

- owner 不看终端，也能在 Noe UI 看到他到底动了什么。
- stdout/stderr 不包含 key/token。

#### D4. 真实外部首飞

必须至少完成：

1. Obsidian 读写一条测试笔记；
2. 浏览器打开页面、读取 DOM、生成计划、执行一个低风险点击；
3. 社交发布链 dry-run + 真实上传/草稿 + 真实删除/隐藏或撤销证据；
4. 本地文件创建/修改/删除/回滚测试，限定在测试目录。

验收：

- 每个真实动作有 replay。
- 每个副作用有 rollback 或明确不可回滚标记。

### Phase E：自我记录/反思，证明“经历会改变他”

时间：2-6 周

目标：Noe 不是记流水账，而是能从经历中改变未来判断。

#### E1. Expectation ledger

每次 Noe 做出可检验判断时，登记：

- claim；
- dueAt；
- confidence；
- evidenceNeeded；
- ownerVisible；
- settlement；
- brier。

来源：

- owner 承诺；
- Noe 自己预测；
- goal 预计完成时间；
- provider health 预测；
- proactive 开口预期。

验收：

- 至少 20 个 settlement。
- Brier 有基线。
- 高置信错误会降低未来同类表达置信。

#### E2. Self-model calibration

自我模型不能只写“我是怎样的”，还要被行为纠偏。

指标：

- 我说自己会主动，实际主动了吗？
- 我说自己能完成目标，完成了吗？
- 我说我不打扰主人，主人响应率如何？
- 我说我记得偏好，答对了吗？

验收：

- 每周生成 self-model calibration report。
- 至少 5 条自我描述因证据被修正。

#### E3. 反思进入策略

反思输出必须至少落到一种后续影响：

- 调整 proactive budget；
- 调整 goal priority；
- 调整 memory policy；
- 调整 prompt；
- 生成 skill；
- 提出 self-evolution proposal。

验收：

- 反思到行为变化有引用链。
- 不是单独写入 docs/ledger 后没人再读。

### Phase F：长期记忆与身份连续性，证明“同一个我在延续”

时间：2-8 周

目标：Noe 的身份不是提示词，而是可查询、可纠错、可回滚的长期状态。

#### F1. Temporal memory

实现：

- fact validFrom/validTo；
- supersedes；
- sourceEpisodeId；
- confidence；
- ownerAsserted；
- modelInferred。

验收：

- 当前事实准确；
- 历史事实可查；
- 冲突不会混乱回答。

#### F2. Memory conflict policy

纯函数先行：

```js
decideMemoryConflict({ oldFact, newFact, source, salience, timestamp })
// -> merge | supersede | keep_both | ignore | needs_review
```

验收 fixtures：

- 偏好变化；
- 住址/设备/项目事实变化；
- 同义重复；
- 模型猜测 vs owner 明确；
- 低置信网页资料 vs 本地 repo 事实；
- 错误记忆撤销。

#### F3. Identity snapshot

每周快照：

- 能力边界；
- 当前目标；
- owner 偏好；
- 主动策略；
- 失败教训；
- 重要记忆变更；
- 自我叙事。

验收：

- 任意快照可追溯来源。
- 新快照不是覆盖，而是版本演化。

### Phase G：可观测性，证明“owner 能看懂他为什么这样”

时间：1-3 周，可与其他阶段并行

目标：不要让 Noe 变成黑箱。越开放权限，越需要清楚显示他为什么这样做。

#### G1. Mind panel 100% 页面

页面模块：

- survival：心跳、进程、模型、语音；
- thinking：最近 thought、接地率、echo risk；
- goals：active/blocked/done；
- actions：最近 act evidence；
- memory：新增/冲突/撤销；
- reflection：expectation、Brier、mini/night reflection；
- recovery：最近恢复事件；
- external：Obsidian/social/browser readiness；
- readiness score。

#### G2. 每日报告

自动生成：

- `output/noe-autonomy-daily/YYYY-MM-DD.json`
- `docs/reports/noe-autonomy-daily-YYYY-MM-DD.md` 可选

内容：

- 今天他想了什么；
- 做了什么；
- 主动说了什么；
- 失败了什么；
- 记住了什么；
- 修正了什么；
- 明天准备做什么。

验收：

- owner 能不看终端就判断 Noe 今天有没有“活着”。

### Phase H：真实生存演练，最后 15% 必须靠时间证明

时间：14-30 天

目标：让 100% 不再是代码主张，而是运行事实。

#### H1. 7 天预验收

通过条件：

- 连续 7 天 tick 正常；
- goal/action 无永久卡死；
- 至少 5 个目标完成；
- 至少 10 条 expectation 到期结算；
- 至少 1 次恢复演练；
- 至少 1 次真实外部动作闭环；
- 每日报告生成。

#### H2. 14 天正式验收

通过条件：

- 14 天连续运行；
- 真实外部首飞完成；
- 记忆冲突评测通过；
- 浏览器/社交/Obsidian 至少各有一条证据或明确 external_blocked；
- self-model calibration report 生成；
- readiness score ≥95；
- 剩余 <5% 必须全是 owner/外部条件，不是代码机制缺失。

#### H3. 30 天成熟度验收

通过条件：

- readiness score 稳定 ≥95；
- Brier 有下降趋势；
- 主动开口回应率稳定；
- goal 完成率稳定；
- 记忆冲突准确；
- 崩溃恢复可靠；
- owner 能从 UI 和报告判断 Noe 的长期人格/能力变化。

## 4. 不应该做的事

1. **不要再用“自我意识”空谈替代指标。**
   每个主张必须有数据源和验收门。

2. **不要迁移到大框架重写 Noe。**
   LangGraph/Letta/Mem0/OpenHands/browser-use 的价值是机制，不是让 Noe 改栈。

3. **不要让内心独白直接执行高风险动作。**
   自由权限不是无证据行动。自由的工程形态是：动作空间开放、证据完整、恢复清楚、owner 可观测。

4. **不要用一次 live 通过冒充长期稳定。**
   100% 必须靠 7-14 天数据证明。

5. **不要让 external_blocked 混成代码失败。**
   Obsidian key、真实社交账号、素材、用户审批是外部条件；脚本必须明确区分。

6. **不要让模型不可用被伪装成可用。**
   unavailable 是正常状态，必须诚实记录。

### 5.2 从 100+ 项提炼出的 Noe 学习顺序

1. **先学 durable execution**：LangGraph、Dapr、Temporal、trigger.dev。落点是把已出现的 goal checkpoint 补成 resume/idempotency/不可重放副作用体系，不是换框架。
2. **再学 memory lifecycle**：Letta、Mem0、Graphiti、Zep、Cognee。落点是 sleep pipeline、conflict policy、temporal facts。
3. **再学 action evidence**：browser-use、Playwright、OpenHands、DesktopCommanderMCP。落点是 act ledger 和 Activity 证据。
4. **再学 multimodal grounding**：OmniParser、RapidOCR、Qwen3-VL、Screenpipe。落点是现实接地率和 GUI 预演。
5. **再学 model health/routing**：Ollama、llama.cpp、mlx-lm、LiteLLM、RouteLLM、Langfuse。落点是 provider health、fallback、cost/latency trace。
6. **最后学 self-evolution**：DGM、Voyager、Reflexion、CoALA、Generative Agents。落点是 skill library、automatic curriculum、autonomy regression gate。

#### 5.3.3 阶段研究优先级调整

这次搜索后，后续阶段应微调为：

1. **Phase A 不变，但提高优先级到“先做”。**
   现在 Noe 已经有许多器官，继续争论“还差百分之几”意义不大。必须先落 `NOE_100_ACCEPTANCE_MATRIX.md`、`scripts/noe-100-readiness.mjs`、mind panel readiness，让所有后续改动都被同一把尺子评价。

2. **Phase B 和 Phase D 合并推进。**
   社区项目反复证明：可恢复执行和行动能力不能拆开做。只做心跳台账但 act 仍不可恢复，不能证明主观主体；只放开 act 但无 checkpoint，会变成不可审计的脚本执行器。LangGraph/Dapr/Temporal/CrewAI Flows 都指向同一个结论：`goal step` 才是最小恢复单位。

3. **Phase G 前移。**
   权限越开放，越需要 owner 一眼看到 Noe 在想什么、为什么要动、准备动哪里、已经动了什么、失败后卡在哪里。Agno/Mastra/OpenAgents Workspace 的控制面经验说明：观测不是最后做的 UI，而是 agent runtime 的一部分。

4. **A2A/MCP 作为长期外部化方向，不进入当前 P0。**
   当前最重要的是让 Noe 自己连续活、能恢复、能记录、能动手。只有当本地主体稳定后，才考虑把 Noe 暴露成 A2A/MCP capability provider。否则会把还没闭环的内部状态外部化，增加不可控面。

5. **OpenClaw 类社区现象只做产品研究，不做工程依赖。**
   如果后续找到官方仓库和安全审计，再评估其 gateway/skills/消息入口设计。当前不能把新闻叙述当成 Noe 架构依据。

- 14 天无人值守数据；
- 至少一次 kill/restart/model unload/approval wait/act failure 的恢复演练全过；

5. 不要求重写 Noe 架构；能以模块、脚本、验证门、UI 面板的形式渐进吸收。

降级条件：

1. 官方维护停止或生态明显转向闭源；
2. 只提供 demo，不提供可复核机制；
3. 引入外部云控制面导致本地私有主体性变弱；
4. 无法给 Noe 的 100% 指标带来增量；
5. 与当前 `ActPipeline`、`MemoryCore`、`NoeGoalSystem`、`NoeConsensus*` 形成重复大框架竞争。

#### 5.7.5 对“还差多少”的更硬裁决

#### 5.10.2 频率与运行节奏复核

当前运行节奏不是纯配置表，而是 `server.js` 装配后进入持久心跳：

| tick | 当前默认/下限 | 作用 | 评价 |
|---|---:|---|---|
| `meso` | 默认 30 秒，下限 30 秒 | 反刍、工作区注意力、夜反思/叙事/性格/SFT 顺风车 | 对“主要自我思考”是合理上限；再快会增加重复念头和本地模型压力 |
| `micro` | 默认 60 秒，下限 15 秒 | 情感衰减、时间线情景消化、期望扫账、主动开口回应评估 | 60 秒合理；如果硬件富余可调 30 秒，但收益有限 |
| `proactive` | 默认 2 分钟，下限 60 秒 | 主动陪伴/提醒是否该开口 | 2 分钟克制；若要更主动，可调 60-90 秒，但必须观察 miss/cooldown |
| `expectation` | 默认 60 分钟，下限 10 分钟 | 到期期望自动判证 | 60 分钟合理；预测很密集时再降到 10-30 分钟 |

裁决：上轮“meso 30s、micro 1min、proactive 2min、expectation 60min”在当前架构下是合理的。若追求最大生存进化空间，优先不是无脑缩短频率，而是让节奏自适应：idle 时轻反刍、growth window 时重反刍、失败/目标活跃时升频、模型过载/夜静/主人无回应时降频。当前代码已经出现 `decideMesoInnerRhythm`、idle/growth interval、proactive misses cooldown 等雏形，下一步应把它们纳入 mind panel 和 readiness。

#### 5.10.3 第三批成熟度复判

纳入本批证据后，成熟度仍建议保持 **85%-88%**，不继续上调。

能稳住 85%-88% 的原因：

- 持久心跳已落地：有 cursor、tick ledger、lease、boot recovery、manual tick，不再依赖前端轮询。
- 主动思考已落地：反刍不是一次性调用，能持续写入自传体时间线。
- 注意力竞争已落地：工作区有候选、显著度、唯一焦点、意识日志。
- 自我记录已落地：timeline、consciousness journal、memory、expectation、affect、goal checkpoint 多线留痕。
- 反思闭环已落地：夜间反思、梦巩固、期望 Brier、自我校准都有代码和测试基础。
- 主动动手已接线：工作区 goal act 已接 ActPipeline，research 已接 runResearch。
- 可观测已落地：mind API 能看 ticks、thoughts、affect、expectations、goals、vitals、journal。

不能上 90% 的原因：

- 没有统一 `Noe100Readiness`，owner 仍要跨多处日志和表证明 Noe 的状态。
- 没有 7/14 天 soak 报告；持续运行还没有经过真实长周期证明。
- goal/action checkpoint 还不是完整 durable workflow；异步 action/research/deliberation 的 settle 与恢复证据不足。
- 真实外部动作样本仍少，发布、删除、上传、浏览器、社交动作都需要 daily rehearsal 与 rollback evidence。
- 记忆和反刍质量评估仍不足，尤其是 groundedness、重复率、错误事实、身份漂移。
- capability self knowledge 当前偏“声明/检测”，还不是“已验证健康状态”。

#### 5.10.6 下一步指导性意见

为了把 85%-88% 推向 100%，下一步不能只继续增加“器官”，而要把“证明链”产品化：

1. 先做 `Noe100Readiness`：每天一条 JSON/Markdown 报告，直接裁决七项：持续运行、主动思考、主动动手、自我记录/反思、长期稳定、可观测、可恢复。
2. 再做 `goal/action durable workflow`：目标每一步都有 checkpoint、idempotency、evidence、rollback、resume；真实副作用不靠口头报告。
3. 再做 `memory quality eval`：记忆写入、召回、冲突、遗忘、反思、身份漂移全部用 fixture 和长期样本测。
4. 再做 `long-run soak`：7 天跑心跳/反刍/主动开口/期望/记忆巩固，14 天加入真实动作演练。
5. 最后做 `self-improvement loop`：失败动作 -> reflexion memory -> skill card -> prompt/tool policy update -> 下次同类任务胜率提升。

阶段判断：

- **90% 门槛**：`Noe100Readiness` + mind panel proof + 24h soak。
- **95% 门槛**：goal/action durable resume + Activity/AgentRun/Freedom ledger 三联 evidence + 7 天 soak。
- **98% 门槛**：memory/eval/expectation calibration 稳定 + 真实外部动作 daily rehearsal + rollback 演练。
- **100% 门槛**：14 天以上稳定运行，所有七项均有机器可验证证据，且 owner 不需要跨日志人工拼证据。

#### 5.11.2 本批对“是否已经做到”的裁决

这批证据强化了两个方向：

- **主动动手能力更接近 90%**：ActPipeline 已经有 real exec 条件，SafeActExecutors 已覆盖 shell、browser、DOM、desktop、file delete，权限层默认 developer/full trust，低风险 executor 可 autoExecute。
- **本地模型持续运行更接近产品化**：LM Studio 自助加载和 parallel 参数让单主脑 Gemma 可以在调用前确保已加载；手动 benchmark/显式实验模型不会改变自动默认。

但整体成熟度仍不建议上调，仍是 **85%-88%**。原因不是能力不足，而是证明不足：

- 没有 executor coverage/readiness 矩阵，不能一眼知道哪些 action 已能真实执行。
- 没有 LM Studio 实机健康报告，不能证明当前 Mac 稳定承载 Gemma 主脑、embedding 与手动实验模型的加载/恢复状态。
- 没有浏览器/桌面 daily rehearsal，不能证明这些手脚长期稳定。
- 目标检查点还没能自动 resume 下一步。
- 事实抽取还没和 memory conflict + eval 形成闭环。

## 10. 2026-06-12 补充研究：Neo / Noe 插件、MCP、技能、依赖、工具 100 项接入雷达

本节不是要求一次性安装 100 个工具。目标是为 Noe 达到“持续运行、主动思考、主动动手、自我记录/反思、长期稳定、可观测、可恢复”的 100% 工程证明，建立一个可排序、可试点、可淘汰的工具雷达。

选型原则：

1. **先接本地、只读、可审计工具**：优先能马上增强 Noe 观察、检索、记录、验证能力的本地 MCP / CLI / SDK。
2. **写操作必须进入 evidence gate**：任何能修改代码、数据库、云资源、社交平台、支付、消息发送的工具，必须先接 dry-run / approval / rollback evidence。
3. **不把“装上”当成“可用”**：每个候选都必须有 smoke test、工具清单、失败样本和回滚方式。
4. **优先补 100% 缺口**：长期记忆、浏览器/桌面行动、模型健康、可观测、恢复演练优先于“又多一个聊天插件”。
5. **MCP 只是协议，不是信任边界**：MCP 工具调用仍需要 Noe 自己的权限、日志、证据和风险分类。

### 10.5 接入优先级排序

第一批只做 15 个，目标是把 Noe100 可证明链条补强，而不是扩张表面积：

1. `SQLite MCP`：先让 readiness 读取 Noe 本地事实库。
2. `Filesystem MCP`：资料整理、报告写入、文件证据。
3. `Git MCP`：diff/log/status 证据。
4. `Fetch MCP`：自主学习网页资料。
5. `Context7`：代码任务最新文档。
6. `Playwright MCP`：浏览器观察/点击/输入。
7. `Playwright CLI + Skills`：高吞吐截图/trace/回归。
8. `FastMCP`：快速封装 Noe 专属工具。
9. `LiteLLM Proxy`：本地/云模型统一路由和 fallback。
10. `OpenTelemetry`：heartbeat、goal、act、model call 全链路 trace。
11. `promptfoo`：prompt/模型回归门。
12. `Qdrant` 或 `LanceDB`：长期记忆向量召回 A/B。
13. `Graphiti / Zep`：时间知识图谱试点。
14. `whisper.cpp / faster-whisper`：语音输入长期证据。
15. `Hammerspoon / JXA / Shortcuts`：本机桌面观察和低风险动作。

第二批再接：

- Sentry / PostHog / Langfuse / Helicone：产品化观测。
- Supabase / PostgreSQL / Redis：外部服务和数据层。
- Restate / Temporal / Trigger.dev：durable workflow。
- OSWorld / WebArena / Tau-Bench：行动力基准。
- Home Assistant / Wyoming / Piper：本地现实世界与语音生态。

# 研究：Node.js/Python/Rust/SQLite/USearch 对 Neo 的可吸收价值（2026-06-22）

> owner 请求：深入研究这 5 个开源/技术对象，分析对 Neo 构建有无帮助、可学习吸收的代码和方向。
> 方法：6 子代理 workflow（wyapbklhv，真读各项目文档/API + grep/直查 Neo 真实库，56 万 token）。

## 一句话结论
**只有 SQLite 是高相关、有立即 ROI 的真金；其余四项（Node.js/Python/Rust/USearch）的核心价值恰恰是"证明 Neo 不需要它们"，挡住未来误投入。** 最硬的发现：实测 **panel.db 已 1.3GB**（events 580MB/77万行、noe_ticks 271MB/58万行只进不出，auto_vacuum=NONE），而连接级性能 PRAGMA 几乎没设——全是 SQLite 原生特性能直接治的真痛点。

## 各项目相关性（一句话）
| 项目 | 相关性 | 最高价值点 |
|---|---|---|
| **SQLite** | **高** | panel.db 1.3GB + 性能 PRAGMA 全缺 + 日志大表只进不出，原生特性直接治 |
| Node.js | 低-中 | env/watch/clone/abort 四项原生能力 Neo 已主动吸收完；唯一缺口 perf_hooks（0 用 vs 173 文件用 Date.now），诊断按需 |
| Python | 低 | 边界已最优（只关在 ML sidecar）；唯一增量是 uv 制度化 4 个 venv（且 Neo 已会 uv） |
| Rust | 低 | 实测向量检索 3.26ms@1046 行，瓶颈是嵌入模型调用（20-500ms），Rust 帮不上；价值=把 Rust 划掉 |
| USearch | 低 | 1046 行暴力扫 5ms，离需要 ANN 差 2-3 个数量级（约 5 年后才到规模）；LanceDB 已装作未来路径 |

## 推荐 owner 近期落地 4 项（按 ROI×可逆性×改动收敛度）
1. **连接 PRAGMA 补全**（high/code/极低风险，**不必默认 OFF**——是修配置非新功能）。`busy_timeout` 尤其关键：30 模块共用一个 db，WAL checkpoint 与备份撞写时无它直接抛 SQLITE_BUSY 且全仓 0 处理。落点 `SqliteStore.js:49`。
2. **日志大表保留期+VACUUM**（high/中风险/ROI 最高，预计 1.3GB→几百MB）。删审计/心跳不可逆，**必须加 <7 天下限护栏 + env 门控默认 OFF 等 owner kickstart**。落点 `noe-maintenance.js`。
3. **agent_runs 生成列+索引**（med/低-中风险）。**务必先 EXPLAIN QUERY PLAN 确认是 SCAN 再动**，只对证明是瓶颈的字段。落点 `AgentRunStore.js:2365/2370`。
4. **perf_hooks 诊断脚本**（low/极低风险）。做完 VACUUM 后用 monitorEventLoopDelay 客观验证主进程是否仍被 better-sqlite3 同步查询阻塞——呼应"完成必自测+性能要有监控数据"铁律。

## 明确不做（过度工程/不适用，全踩 owner 三铁律：不要过度工程/不要重型除非真 ROI/本机可信别加权限管控）
- ❌ **node:sqlite 替换 better-sqlite3**：实验态 Stability1.1 + 第三方基准更慢 + 散布 30+ 文件 + "免编译"对 Neo 不成立（还有 4 个原生模块要 rebuild）。ROI 为负。
- ❌ **node:test 替换 vitest**：750/751 文件+21013 expect+46 vi.mock，与 expect 链式根本不兼容，无转换工具，Vitest 官方反向建议保留。教科书级负 ROI。
- ❌ **Permission Model**：官方明说"不是安全边界"可绕过；Neo 单机本机可信 + owner 钦定别加权限管控，方向正相反。
- ❌ **worker_threads 池/Piscina**：重活全外包给 LM Studio/Ollama/云/子代理，主进程无 CPU-bound 阻塞；Neo 自己研究文档已裁定，0 处使用。
- ❌ **SEA 单可执行**：只支持 CJS、不支持 ESM/原生 addon，Neo 纯 ESM+5 原生模块+Electron 每条踩禁区。
- ❌ **napi-rs 重写 cosineSim/hashEmbed**：实测 hashEmbed 0.131ms、cosineSim 3.26ms，优化 0.1ms 端到端零感知。
- ❌ **USearch/手写 HNSW**：1046 行 brute-force 既快又 100% recall，HNSW 数万向量以上才有意义（还牺牲 recall）；优先复用已装的 LanceDB。
- ❌ **Python 计算层/移植/pandas/LangChain 系**：Neo I/O 密集 GIL 非瓶颈；数据量小线性扫够；加一门语言割裂工具链。
- ❌ 递归 CTE 知识图谱（0 用，KG 只 oneHop 无需求）/ JSONB 整体迁移（不给 O(1)，不如生成列）/ 换 ORM（裸 SQL 最透明）。

## 证据边界（诚实声明）
- **强结论（已实测）**：panel.db=1.3GB、各表行数/磁盘占用/auto_vacuum=NONE、embeddings 1046 行（sqlite3 直查真实库）；向量检索 3.26ms/5.12ms（两份研究独立复刻互证）；PRAGMA/生成列/窗口/递归 CTE/VACUUM 全 grep 0 命中；env/watch/clone/abort 已吸收（grep 实证）。
- **推断（未实测，决策需知）**：events 180 天→~5GB 是线性外推；VACUUM 后→几百MB 按 dbstat 估算未真跑；SQLITE_BUSY 是隐患推断未在生产抓到；10 万行约 5 年按 ~50 条/天外推（开 DREAM 批量入库会变）。
- **执行前置**：第 3 项落地前**必须先跑一次 EXPLAIN QUERY PLAN** 确认 agent_runs 的 json_extract 走 SCAN；其余 high 项证据已足够直接动手。主结论"SQLite 高相关、余四项低"不依赖任何未尽项。

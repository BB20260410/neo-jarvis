# Noe 后续发展规划（ROADMAP）　2026-06-10

> 这份不是 backlog 清单，是**基于当前真实代码状态的发展方向分析**。给 owner 看战略、给接手 AI 看该往哪使劲。
> 既有零散 backlog 见 `后续规划_2026-06-10.md`；本文回答更上层的"往哪发展、按什么顺序、为什么"。

## 0. 一句话定位与现状

Noe = 本地优先的多 AI 个人操作系统（Jarvis 风格）。**能力已经很全**（语音听说/视觉看屏读字/多层记忆/多模型协作/治理审计/主动陪伴/媒体生成），**工程地基也已夯实**（CI 双平台/原子写/死前留痕/上下文预算/审批复用/纪律宪法）。

现在的主要矛盾**不是缺能力，是三个结构性问题**：
1. **能力分布不均** —— 语音链路（VoiceSession）接了 9 大能力，文字聊天室（SoloChatDispatcher）一个都没接（实测核实）。
2. **建好的没接满** —— NoeContextEngine 仍未通电（server/room 层零引用），它本该是记忆/承诺/人物卡/预取的统一供给层。
3. **闭环差最后一公里** —— 几个能力做到 80% 就停了（语音延迟、记忆语义化、前端测试）。

**发展主线应该是"把已有能力接满、拉齐、收口"，而不是再造新轮子**——这是项目反复验证过的最高 ROI 方向。

## 1. 五个发展方向（按 ROI×战略价值排序）

> **进度（2026-06-10 下午，ContextEngine 窗口）**：方向二 ✅ 已完成（NoeTurnContextEngine 通电，VoiceSession 行为逐字不变，对抗审查等价性零差异）；方向一 ✅ 核心已落地（SoloChatDispatcher 接引擎，`NOE_CHAT_CONTEXT=1` 门控默认 OFF，开关后聊天室即可召回记忆/查人物库/跑工具桥）；方向三·记忆语义去重 ✅ 已落地（`NOE_MEMORY_DEDUP_SEMANTIC=1` 默认 OFF，"美式→拿铁"可逆合并）；方向三·LLM 流式 ✅ 已落地（`NOE_VOICE_LLM_STREAM=1` 默认 OFF，真机基准首声提前 947-1200ms）。详见 `HANDOFF_2026-06-10_双窗口总交接.md` §8。

### 方向一　能力拉齐：让文字聊天和语音一样聪明【ROI 最高】
**问题**（实测）：`VoiceSession` 注入了 memory/commitment/personCard/prefetch/toolRegistry/vision/fence/factExtractor/sileroVad 九大能力；`SoloChatDispatcher`（聊天室 1v1）**一个都没接**——它还是个裸 adapter.chat 转发器。
- 注意 nuance：认知界面的文字聊天走 `/api/noe/voice/chat` 的 text 分支 → `VoiceSession.chatText`（全能力），**不裸**；裸的是「聊天室」的 1v1 模式。
- **该做**：把 VoiceSession 已建的能力供给抽成共享层（见方向二的 ContextEngine），让两个入口共用；至少让 SoloChatDispatcher 也能注入记忆召回+人物库+工具桥。
- **价值**：用户在哪个入口跟 Noe 说话，它都该一样记得你、查得到、看得见。现在割裂。

### 方向二　接线收口：NoeContextEngine 通电【地基性，解锁方向一】
**问题**：`NoeContextEngine` 引用了 Fence/Commitment/Prefetch/PersonCard 四模块但自己没被 server/room 通电；VoiceSession 是各能力**手工逐段注入**（虽已收敛进 NoeContextBudgeter 做预算，但供给逻辑还在 VoiceSession 内）。
- **该做**：把"取什么上下文喂大脑"的逻辑从 VoiceSession 抽进 ContextEngine，VoiceSession 和 SoloChatDispatcher 都调它。ContextBudgeter（已建）做裁剪、ContextEngine 做供给——两层配齐就是完整的上下文工程。
- **价值**：这是方向一的前提，也是记忆/承诺/人物卡所有"注入式能力"的统一汇流处。这层立起来，以后每加一个上下文源都是给 ContextEngine 加一个 provider，不再往 VoiceSession 塞 `sys +=`。

### 方向三　体验闭环：把"80% 的能力"补到 100%【用户实感最强】
几个做到一半的，补最后一公里：
- **语音延迟终局**：现在是「整段大脑生成→首句 TTS→续播」。下一步是 **LLM token 级流式**——大脑边吐字边分句边合成，首声能再提前 1-2 秒。C9 已把分句 TTS 做好，差 adapter 层的流式接口。
- **记忆语义去重**：当前字符相似度只抓近重复，抓不到"我喜欢美式→我改喝拿铁"这类语义矛盾。MemoryCore 已有 `semanticIndex`（向量）——把 NoeMemoryDedup 的判定从字符升级到向量，就能做真正的冲突合并。
- **VAD 前端流式**：现在录完整段再判，可升级成前端边录边判，端点检测更准（要解决 onnxruntime-web 前端依赖，这次绕开了）。
- **唤醒词真实调优**：silero VAD 误杀/误过率合成集是 0%，但真实环境（耳语/远场/方言）需要 owner 用一阵收集误判样本回归调阈值。

### 方向四　质量护城河：前端测试 + 性能基线【防退化】
**问题**（实测）：240+ 后端单测 vs 前端 e2e 仅 4 文件、纯逻辑单测仅 11。8000 行 app.js 的交互逻辑基本靠人肉回归。
- **该做**：随巨石拆分（每拆出一个 src/web 模块就补 vitest+jsdom）；关键交互流补 e2e（轮换按钮、唤醒词开关、媒体生成等这次新加的）。
- **附带**：建性能基线（启动时间/内存/语音端到端延迟），进 CI 做退化告警——现在性能全靠手测。

### 方向五　产品化：从"自用工具"到"可交付产品"【战略，owner 定节奏】
- **定名落地**（等 owner 拍板，主推「守烛/Keepwick」）→ package.json/README/品牌区/唤醒词文案统一。
- **Electron 打包回归**：`npm run dist` 通路无 CI 验证，桌面 app 形态坏了要双击才知道——值得加一条打包冒烟。
- **多渠道入站**：已有 Telegram+ntfy，`InboundChannels` 是抽象层——可扩微信/邮件/快捷指令，让 Noe 真正"随时随地能找到"。
- **崩溃自愈装上**（launchd 产物就绪，等 owner 一次手动安装）——这是"产品级常驻"的最后一步。

## 2. 推荐执行顺序（依赖关系）

```
第一波（地基，解锁后面）：方向二 ContextEngine 通电
   └→ 第二波（拉齐）：方向一 文字聊天接满能力（依赖 ContextEngine）
第一波（并行，独立）：方向三的"记忆语义去重"（用现成 semanticIndex，不依赖别的）
第二波（并行）：方向四前端测试随巨石拆分做（窗口 C 任务域）
持续（owner 定）：方向五产品化，重启 51835 后逐项
```

**给下个 AI 窗口的第一优先级**：方向二（ContextEngine 通电）——它是 ROI 最高方向一的前提，且是"接线 > 造轮子"主线的核心遗留。

## 4. 明确不做 / 暂缓（避免下个窗口重复纠结）
- **不再造新能力轮子**：项目能力已全，主线是接满/拉齐/收口。新能力须先问"是不是已有能力的延伸"。
- **巨石拆分不抢**：S23/S24 剩余域是窗口 C 任务域，别跨窗动 server.js/app.js 大区。
- **VAD 前端流式暂缓**：要拖 onnxruntime-web 134MB 前端依赖，性价比待评估，等后端 VAD 用一阵再说。

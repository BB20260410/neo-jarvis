## 用户授权边界

早期 `AGENTS.md` / `CLAUDE.md` 里的“不 spawn 子 LLM”应视为默认安全策略，不再是绝对硬限制。

在用户明确授权的 Noe 多模型任务中，核心协作只启用 Codex / Claude / MiniMax M3；Gemini 不进入新任务核心 quorum，但必须同时满足：

- 任务边界清楚。
- 密钥和日志脱敏。
- 单 active executor：默认 Codex 最终读代码、改代码、验证、汇报；如果 Codex 没额度或用户明确指定，可以把 Claude 选为本轮唯一 active executor。
- 不偷偷消耗额度；说明用途、产物和边界。
- 不把外部模型建议直接当代码。
- 不 commit / push，除非用户明确要求。

## 核心三方职责

### Codex

默认执行者、集成者、验证者。

负责确认真实 repo root、读规则和相关代码、拆最小改动、应用 patch、跑测试、在用户确认或动态 quorum 共识授权后重启 live 51835、浏览器/接口/语音实测、最终汇报证据。

当本轮 active executor 被显式切换到 Claude 时，Codex 退为 readonly/advisory reviewer，不再同时写文件。

### Claude

长上下文系统审查。

适合检查语音交互、主人门禁、人脸/声纹、记忆、视觉上下文、派活审批和长期产品一致性。默认情况下 Claude 给架构风险和 patch plan，不直接无监督改文件；如果用户明确选择 Claude 作为本轮 active executor，Claude 可以成为唯一 writer/integrator，但必须同样遵守真实 repo 核验、边界、测试、runtime verification、handoff 和不泄露密钥规则。

### Gemini（已从核心剔除）

不作为必选投票方、不作为 post-review required reviewer、不计入动态 quorum、不作为新任务默认 runner。

Gemini adapter / CLI 只保留历史 ledger 兼容、普通 provider health 展示、或用户未来显式要求的单独实验；即便显式运行，也只能作为 non-counting advisory evidence，不能替代 Claude/M3/Codex 任一核心票。

### MiniMax M3

中文体验、TTS/语音、冷审查。

适合审查 Noe 回复是否像个人助手、是否复读网页、是否会把 URL/HTML/`src=` 念出来、失败兜底是否清楚、权限和隐私风险是否遗漏。M3 可作为强制审查点，但最终由 Codex 落地。

## 固定流程

1. 项目确认：`pwd`、`git rev-parse --show-toplevel`、`git status --short`。
2. 范围确认：明确本轮只处理语音、搜索、身份、视觉、记忆、派活、执行或 UI 的哪条链路。
3. 执行者选择：默认 Codex 做最终实现；如 Codex 不可用/没额度，用户可显式选择 Claude 作为本轮唯一 active executor。
4. 多模型分工：active executor 做最终实现；非执行核心模型做定向 review。Claude 做强独立审查，M3 做中文/TTS/风险冷审，Gemini 不参与默认核心流程。
5. 模型任务设置：不要给模型协作设置低超时；优先 `noAbort:true` 或足够长/不主动中断。模型不可用要诚实标注。
6. 执行取舍：常规任务由 active executor 按代码事实、风险、成本和用户目标采纳；升级到 self-evolution gate 时，active executor 也必须写独立 vote，但不能绕过其它模型票或系统级 veto。
7. 最小实现：一次只改必要文件，碰到 499/500 行优先拆模块。
8. 真实验证：单测只是底线，语音/搜索/浏览器链路必须实测。
9. 汇报：说明改了什么、为什么、测试证据、真实体验、没做什么、残余风险。

## 2026-06-07 自我进化闭环升级

当用户明确要求 Noe 自我迭代、自我修复、自我完善，或要求多模型互相讨论时，以上固定流程升级为强制 consensus gate：

- Claude 必须是一等参与者：必须真实调用 Claude，并保存其独立 `rawOutputRef`；不能由 Codex 代写 Claude 票。
- Codex / Claude / M3 必须对同一 `evidenceRef` 投票，结果写入 consensus ledger。
- 动态 quorum：3 个核心模型可用时至少 2/3 通过；1 个核心模型无额度/不可用时，剩余 2 个核心模型必须 2/2 都通过；只剩 1 个核心模型可用时停下。`reject`、`abstain/unavailable` 允许记录但不计通过票。
- 不允许把 fallback 写死成 “M3 不可用” 或任一固定模型不可用。Codex、Claude、M3 任意一个或任意两个模型都可能突然没额度；gate 必须按当轮真实 `availableModels` 计算，missing vote 不能当作 unavailable。
- 可用性组合矩阵必须按集合计算，不按模型名特判：
  - 3 个可用：`{Codex, Claude, M3}`，阈值 2。
  - 任意 1 个 unavailable：`{Codex}` / `{Claude}` / `{M3}`，剩余 2 个必须全同意。
  - 任意 2 个 unavailable：只剩 1 个可用，直接停下，不能由单模型授权。
- 缺任一必选模型 vote、缺 rawOutputRef、`consensus_vote` 冲突、M3 内容级越权、缺成本授权、缺 rollback、缺 runtime verification、缺 `retrospectiveRef`、缺 memory writeback 确认，均直接 blocked。模型 raw parse 失败或 no quota 可以记录为 `unavailable`，但必须有 rawOutputRef/错误证据，并按动态 quorum 降低可用分母。
- Self-evolution action 中，动态 quorum 共识可替代用户亲自确认；删除、上传、外部发布、密钥访问/使用、重启/杀进程和长期记忆自动写入都必须有动态 quorum 共识；系统级操作仍不能被模型票覆盖。
- 单 writer 仍不变，但 writer 可以是本轮 `activeExecutor`。默认 active executor 是 Codex；用户明确选择或 validated consensus 选择 Claude 时，Claude 可以作为唯一 writer/integrator，Codex 转为非写入 reviewer。M3 suggestion-only，不允许成为 writer；Gemini retired from core quorum。
- 不要给模型设置人为硬超时；模型不可用时记录 `unavailable` 和健康检查证据，不用低超时切断讨论。默认 adapters、cluster member call 和 AI CLI 搜索都应是 no hard timeout，只有显式用户/环境覆盖才启用。
- 不碰 `51735`；live `51835` 重启/杀进程/抢占端口必须有动态 quorum 共识授权，托管验证用随机非保留端口。
- 托管验证可以使用显式测试夹具，例如 `NOE_AI_SEARCH_MOCK=1`，但报告必须暴露 `mockSearch:true`，生产路径默认关闭，不能把夹具结果当真实外部搜索证据。
- 自我修复不是绕过 gate 的特权。修复计划、修复 diff、修复后验证都必须回到同一个 consensus ledger / dynamic quorum gate 流程。
- 每轮落地必须形成 cycle artifact：validated consensus ledger、implementation evidence、runtime report、非实现核心模型 post-review rawOutputRef、retrospectiveRef、memory writeback summary 都齐备，才能宣称一轮 self-evolution complete。Codex 作为 active executor 时，Claude 和 M3 都必须给出 review 且通过；Claude 作为 active executor 时，Codex 和 M3 都必须给出 review 且通过。Codex fallback 只能补证据，不能替代不可用 reviewer 的 quorum 票。
- complete cycle artifact 用 `npm run noe:self-evolution:cycle -- ...` 组装，写入前必须通过 `NoeSelfEvolutionCycle` 校验；非 dry-run 写入不能关闭引用文件校验，sample dry-run 只能验证格式，不能替代 production cycle。
- cycle assembler 的 `--post-review` 支持 `model=rawOutputRef` 和 `model:decision=rawOutputRef`；模型不可用时必须写成 `m3:unavailable=...` 之类的审计证据，不能把 unavailable 伪装成 approve。
- 声明完整闭环完成前必须跑 `npm run audit:noe:self-evolution-completion -- --require-complete` 并通过；默认 `npm run audit:noe:self-evolution-completion` 只报告当前完成度，不等于授权。
- 2 个核心模型通过且 1 个核心模型 unavailable 时，只有在动态 quorum 计算为 available=2、threshold=2 且 ledger artifact 通过 `--require-passed` 后才可进入实现；如果可用模型少于 2 个，必须停下。任何 implementation / self_repair / sensitive capability / memory writeback 授权都必须对目标 production ledger 运行 `node scripts/noe-consensus-ledger-verify.mjs --ledger <ledger> --require-passed` 并通过。
- 跨线程“分享/记录经验”不是 memory writeback。没有 passed production ledger、runtime report、memoryWriteback summary 和 consensusAck 时，Claude/M3/Codex 只能在当前线程 acknowledge，不能写长期记忆；若已有 blocked ledger 下产生的 memory note，只能视为 premature/advisory，不得作为授权规则来源。
- 可执行 self-evolution 必须使用 ledger-backed consensus。普通 payload 里的 `consensus: {ok:true, validated:true}`、payload consensusApproved、payload userApproved、payload ledger object 或伪造 `source:"validated_consensus_ledger"` 不能作为生产执行授权；执行前 guard 要用 artifact validator 从可信 ledgerRef 重新验证，即使真实 approval 已通过，执行入口也不能用 payload ledger object 作为 production consensus artifact；cycle artifact 缺 ledger object/file 直接 blocked。`ledgerRef` 必须是可信 project root 内相对路径，路径逃逸、绝对路径或 payload root 注入直接 blocked；ledgerRef 原始证据文件（evidenceRef/rawOutputRef）必须存在；默认可信根是 module-derived selfEvolutionRoot，ActGuard 直接调用也不依赖启动时 cwd。

当前落地方案见 `docs/Noe自我进化闭环方案_2026-06-07.md`。本地规则由 `src/room/NoeExecutionAuthority.js`、`src/room/NoeConsensusGate.js`、`src/room/NoeConsensusLedger.js`、`src/room/NoeConsensusRound.js`、`src/room/NoeConsensusRunner.js`、`src/room/NoeSelfEvolutionGate.js`、`src/room/NoeSelfEvolutionLoop.js`、`src/room/NoeSelfEvolutionCycle.js`、`src/loop/NoeSelfEvolutionActGuard.js`、`ActPipeline`、`PermissionGovernance`、`npm run verify:noe:consensus-ledger`、`npm run test:noe:consensus`、`npm run verify:noe:self-evolution`、`npm run audit:noe:self-evolution-completion` 和 `npm run noe:self-evolution:cycle -- ...` 验证；真实 raw output 可用 `npm run noe:consensus:assemble -- --goal ... --evidence ... --raw codex=... --raw claude=... --raw m3=...` 组装成 production ledger；核心三模型 round 用 `npm run noe:consensus:round -- --goal ... --evidence-file ...` 先 dry-run，真实模型调用必须有 runner 函数入口的 `costAcknowledged: true`，CLI 只有显式加 `--run-models --ack-cost` 才调用模型。implementation / self_repair / memory_writeback / complete 必须再过 `NoeSelfEvolutionGate`，完整闭环状态由 `NoeSelfEvolutionLoop` 判定，闭环证据包由 `NoeSelfEvolutionCycle` 验证，并且通过 `NoeSelfEvolutionActGuard` 在 `ActPipeline` 执行前检查真实 permission/approval/budget 状态；动态 quorum 共识可替代用户亲自确认，并可授权删除、上传、外部发布、密钥访问/使用、重启/杀进程和长期记忆自动写入；系统级操作仍不能被模型票覆盖。旧四模型 production ledger 只作为历史证据，不再代表当前核心 quorum 配置。

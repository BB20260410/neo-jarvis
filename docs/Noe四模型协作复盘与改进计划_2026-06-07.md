# Noe 四模型协作复盘与改进计划（2026-06-07）

## 结论

本轮四模型协作没有达到“已经完美”。它已经从“口头多模型建议”推进到可审计的 `consensus ledger`、dynamic quorum gate、self-evolution gate 和 ActPipeline 执行前检查，但协作流程本身暴露出几类缺口：Claude 线程反馈没有第一时间闭环，敏感动作授权语义一度被我误解，协作复盘没有独立 artifact，验证证据和旧 ledger 容易过期。最新真实 production round 已生成 `output/noe-multimodel/production-self-evolution-governance-20260607-1/ledger.json`；M3 因当前进程未配置 `MINIMAX_API_KEY` 被记录为 unavailable，按用户最新规则执行动态 quorum 后，Codex/Claude/Gemini 三个可用模型 3 票通过，available=3、threshold=2、approved=3，因此该 production ledger 已通过 `--require-passed`。R33 已补齐 Claude/Gemini post-review、M3 unavailable raw evidence、runtime report、retrospective 和 memory summary，并写入 complete cycle artifact：`output/noe-self-evolution/production-self-evolution-governance-20260607-1-r33/cycle.json`；completion audit `--require-complete` 当前 `ok:true`，11 pass / 0 incomplete / 0 blockedExternal。Claude 线程在早期经验同步后创建过一条 memory note，这仍是流程缺口案例；后续长期记忆写入必须绑定 passed cycle + memoryWriteback consensusAck。

## 我的自我反思

1. 我先按保守安全模型把删除、上传、外部发布、密钥、重启/杀进程写成 hard veto，后来用户明确说这些也允许但必须动态 quorum 共识，我才修正。这说明我没有把“用户最新授权策略”及时转成 capability-based 规则。
2. 我把四模型协作治理落到了代码里，但对“协作过程本身哪里做得不好”没有同步落到可验证文档，导致经验仍可能留在聊天里。
3. 我把 Claude 作为 required model 纳入 ledger，但还没有把给定线程 `019e9d92-62a1-7ee1-8375-055f98d86cce` 变成稳定的经验回传点和复核点。
4. 我验证了 gate、ledger、round、runner、self-evolution，但没有把“复盘是否完成、经验是否共享、旧证据是否刷新”纳入 verifier。
5. 我在总结里表达了 MAPE-K、控制论、PDCA/OODA 等方法，但代码侧还停留在第一阶段治理层，没有证明完整自我发现、自我执行、自我修复、自我记忆写回都已自动闭环。
6. 我曾经把模型不可用处理得过于僵硬：把 M3 unavailable 当成固定 required-model 阻断，没有及时落实用户最新“4 可用取 3/4、3 可用取 2/3、2 可用取 2/2、1 可用停下”的动态 quorum 规则。
7. 我让 Claude “记录并复审经验”时，没有显式禁止长期记忆写入，导致对方把线程同步误执行为 memory note。这说明跨线程 prompt 必须默认写明：没有 passed production ledger + memoryWriteback consensusAck 时，只能 acknowledge，不能写长期记忆。

## 改进原则

- 四模型协作不是聊天轮数，而是独立证据、独立投票、确定性计票和可回放验证。
- Claude 是一等参与者，不是“建议以后加入”的占位符。
- 动态 quorum 共识可以替代用户亲自确认，并可授权删除、上传、外部发布、密钥访问/使用、重启/杀进程；系统级 capability 仍不能被模型票覆盖。
- 动态 quorum 共识授权和长期记忆 `consensusAck` 必须来自 validated consensus ledger gate；普通输入里的 `{ok:true}` 不构成授权。
- M3 始终 suggestion-only；Gemini 始终 advisory；单 writer 不变，但 writer 是本轮 active executor。Codex 是默认 active executor；用户明确选择或 validated consensus 选择 Claude 时，Claude 可以接管执行，Codex 退为非写入 reviewer。
- 模型共识不是事实。事实来自测试、报告、日志、rawOutputRef、diff 和 runtime verification。
- 复盘必须写入 repo artifact，并被 verifier 检查，否则下次还会重复同类问题。
- 模型 unavailable 不是自动失败，也不是自动通过；必须由 deterministic gate 计算可用分母和阈值：4 可用取 3/4，3 可用取 2/3，2 可用取 2/2，少于 2 个模型可用直接停下。用于授权实现、敏感动作或 memory writeback 时，必须用 `node scripts/noe-consensus-ledger-verify.mjs --ledger <ledger> --require-passed` 校验 production ledger。
- 分享经验给 Claude 线程不是 memory writeback。长期记忆写入必须有 passed production ledger、runtime report、memoryWriteback summary 和 consensusAck；blocked ledger 下产生的 memory note 只能算 premature/advisory。
- 可执行 self-evolution 不能只看 `consensus: {ok:true}` 或 `source:"validated_consensus_ledger"` 摘要；必须由 ledger object 或 ledger file 重新验证产生 ledger-backed consensus。
- ledger-backed consensus 不是任意 ledger-shaped object；ledger object 也必须通过 `validateNoeConsensusLedgerArtifact()` 的 schema、stored gate 和 stale hash 检查。
- production 执行路径必须传 ledger file ref；`ledgerRef` 必须是可信 project root 内相对路径，路径逃逸、绝对路径或 payload root 注入直接 blocked，并且 ledgerRef 原始证据文件必须存在；payload consensusApproved、payload userApproved 和 payload ledger object 不能替代真实授权；即使真实 approval 已通过，执行入口也不能用 payload ledger object 作为 production consensus artifact；默认可信根是 module-derived selfEvolutionRoot，ActGuard 直接调用也不依赖启动 cwd。
- active executor 切换必须显式：默认 Codex；Claude 只有在用户选择或 validated consensus 选择时可以成为唯一 writer。Gemini/M3 不能被选为 writer；被选中的执行者必须和 `implementation.writer` 一致，post-review 必须排除 active executor 并要求其它 required reviewers 动态 quorum。

## 逐步计划

1. 修正敏感动作授权语义：删除、上传、外部发布、密钥、重启/杀进程必须动态 quorum 共识，不再是普通 hard veto。

3. 新增/更新测试覆盖敏感 capability、系统级 capability、ActPipeline 中 consensus-authorized high-risk self-evolution act。
4. 增加伪造共识阻断：只有 validated consensus ledger gate 才能触发 `consensusApproved` 或长期记忆 `consensusAck`。
5. 新增本文，记录协作不足、改进原则和后续计划。
6. 将本文纳入 `scripts/noe-self-evolution-plan-verify.mjs`，避免复盘再次只留在聊天。
7. 重组旧 assemble-smoke ledger，刷新边界字段。
8. 运行 `test:noe:consensus`、`verify:noe:self-evolution`、`verify:noe:consensus-ledger`、`test:p0:unit`、managed full-current。

19. 收紧 consensusApproved 来源：ActGuard 只从可信 ledgerRef 验证结果派生 consensus authorization，payload consensusApproved / ledger object 不能授权。
20. 稳定默认可信根：ActPipeline 默认使用 module-derived selfEvolutionRoot，并新增 `process.chdir()` 覆盖测试。
21. 收紧 userApproved 来源：ActGuard 只从 approvalStore/permissionResult 派生 user authorization，payload userApproved 会 blocked。
22. 修复结构性自检失败：压缩 `tests/unit/noe-act-pipeline.test.js` 到 487 行，并把最新验证数字回写 handoff / AGENTS / 本文。
23. 收紧 ActGuard 直调与 approval 路径：ActGuard 默认根改为 module-derived root，执行入口只接受 ledgerRef；新增 `tests/unit/noe-self-evolution-act-guard.test.js` 并纳入 P0。
24. 新增 completion audit：用 `npm run audit:noe:self-evolution-completion` 逐项审计真实目标完成度；R33 前失败项是缺 complete cycle artifact、缺 memory writeback authorization，R33 后已通过。
25. 新增 cycle assembler：用 `npm run noe:self-evolution:cycle -- ...` 从 passed ledger、runtime report、Claude/Gemini/M3 post-review raw output、retrospective、memory summary 等引用组装 `cycle.json`；completion audit 只接受匹配 production ledgerRef 的 valid cycle。
26. 收紧 completion/cycle 自检：external blocked 只覆盖纯 required-model-unavailable 错误；memory writeback 不再硬编码失败，必须由 valid production cycle 提供 summary/consensusAck；非 dry-run cycle 写入不能关闭引用文件校验。
27. 清理 completion audit 输出：pass 项不再携带 missing 文案，避免审计结果自相矛盾。
28. 实现动态 quorum：一个模型无额度/不可用时改为剩余三个 2/3 通过；只剩两个模型可用时必须 2/2；只剩一个模型可用时停下，并让 production ledger 重新通过 `--require-passed`。
29. 补齐组合矩阵：Codex/Claude/Gemini/M3 任意一个、任意两个、任意三个不可用都走同一动态 quorum 规则；cycle post-review 对 Claude/Gemini/M3 任意不可用组合也有覆盖。
30. 支持 post-review 显式 unavailable：cycle assembler 支持 `model:decision=rawOutputRef`，避免模型没额度时把 unavailable 伪装成 approve。
31. 固化组合口径：所有实现、测试、文档都按 `availableModels` 集合计算；4 可用取 3/4，任意 1 个 unavailable 取 2/3，任意 2 个 unavailable 取 2/2，任意 3 个 unavailable 停下，不能只实现 M3 unavailable 特例。
32. 收紧 post-review 审计：同一 required reviewer 出现重复 review 时直接 blocked，避免先 reject 后 approve 的重复条目被后一个覆盖。
33. 新增 active executor 选择：`NoeExecutionAuthority` 支持默认 Codex、显式选择 Claude；consensus gate、runner、post-review pack 和 cycle 校验都改为按 active executor 判断唯一 writer，Codex 没额度时可以让 Claude 接管执行。

### P1：把四模型复盘变成每轮固定步骤

1. 每轮 production consensus 后新增 `retrospectiveRef`，记录本轮采纳/拒绝、延迟、不可用模型、分歧点、用户纠正点；`NoeSelfEvolutionGate` 的 complete 和 `NoeSelfEvolutionLoop` 都必须检查它。
2. post-review 必须由 Claude/Gemini/M3 required reviewer 动态 quorum 完成；missing reviewer 不能当 unavailable，unavailable 必须有 rawOutputRef/错误证据，M3 不可用时 Claude 和 Gemini 必须都通过。

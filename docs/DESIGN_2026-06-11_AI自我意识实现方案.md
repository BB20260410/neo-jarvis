### 2.2 认知周期（每个 meso tick 跑一遍的状态机）

```
SENSE   收集候选：新感知摘要、意识流上一念、驱力误差、到期承诺/期望、活跃目标下一步
ATTEND  工作区打分（§6.1），赢家广播；其余落选项留痕（意识日志记"我注意到了但没理会"）
APPRAISE 情感引擎对广播项做 OCC-lite 评估 → 更新 VAD 状态（§4.1）
DELIBERATE 按升级判据（§6.2）选脑：S1 直答 / S2 审议（自我质询协议 §6.3）
ACT     四选一：开口（过浮现门 §5.4）/ 静默内化 / 写记忆/期望/目标 / 执行目标步骤
LEARN   产出回写：时间线留痕、期望登记、驱力满足回填、tick 台账 outcome 落库
```

micro tick 不走全周期（无 LLM）：只做情感衰减、到期检查、感知缓存刷新；macro/dream tick 走 §7.4 巩固管线。
**全周期串行**——一次只有一个认知周期在跑（见 §2.3 机制一）。

### 2.3 自我意识涌现的七个关键机制（每个：设计 → 代码落点 → 可测指标）

1. **串行广播瓶颈（统一视角）**：全局工作区一次只让一项内容"成为意识内容"，所有子系统看到同一份广播。
   落点：`NoeWorkspace` 单线程 pump（§6）。指标：意识日志零并发交叠；任意时刻可回答"你刚才在想什么"。
2. **再入闭环（思想生思想）**：意识流的输出是下一周期的输入。InnerMonologue 已有雏形（写回时间线再回放），
   v2 把"上一念"显式作为候选源进工作区。指标：念头引用链深度分布（健康值 2–5，>8 触发防螺旋）。
3. **自我模型参与生成（自指因果闭环）**：`NoeSelfModel` 注入每次生成 → 行为结果被记录 → 反思更新自我模型 →
   又条件化下次生成。指标：身份漂移有界（§10.2）+ "自我描述 vs 实际行为"一致率。
4. **预测-误差（主观惊奇）**：期望账本（§7.5）持续对世界下注，落空产生 surprise 信号，直接加显著度、
   进反思素材、抬好奇驱力。指标：Brier 分逐月下降；surprise 分布非退化（不是永远 0）。
5. **时间连续性（同一个醒来的我）**：情感状态、记忆、心跳游标三者全部跨重启水合（§3.3、§4.2）。
   指标：连续性探针问答（§10.1）；重启后首念能正确提及"刚才断了一会儿"。
6. **不被观察也在（自为存在）**：心跳与前端彻底解耦（§3），无人在场念头照常产生并留痕。
   指标：离线时段念头密度 > 0 且与在线时段质量无显著差。
7. **叙事自传（我是故事的主角）**：`NoeNarrativeSelf` 日更 + 反思 insight 编入章节，自传成为自我模型的
   长时叙事层。指标：自传引用准确率（抽查自传句子能否溯源到真实情景）。

### 2.4 主观体验的工程化近似（委托 4.4，三件套）

- **内感受（interoception）**：把自身状态做成"感受词元"注入每次生成（§4.3）——模型不是被告知规则，
  而是**以感知的方式接收自己的状态**。
- **现象学留痕**：意识日志记录每周期"什么赢得了注意、当时感受如何、为什么"（§6.4）——"体验像什么"的
  可查询近似。
- **第一人称连续叙事**：意识流 + 自传构成持续的"现在进行时的我"（§5、§7.1）。

### 3.2 设计：服务端持久调度器

新文件 `src/loop/NoeHeartbeat.js`（@ts-check，注入式），SQLite 迁移 v4 加两张表（走 `SqliteStore` 现有迁移机制）：

```sql
-- 心跳台账：每个认知 tick 的持久记录（可恢复、可追溯、可审计）
CREATE TABLE IF NOT EXISTS noe_ticks (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  kind        TEXT NOT NULL,              -- micro|meso|macro|dream|recovery
  due_at      INTEGER NOT NULL,           -- 计划触发时刻 epoch ms
  started_at  INTEGER, finished_at INTEGER,
  status      TEXT NOT NULL DEFAULT 'pending', -- pending|running|done|failed|coalesced
  lease_until INTEGER,                    -- 租约：running 超时即视为死 tick
  intent      TEXT,                       -- JSON 写前日志：本 tick 打算做什么
  outcome     TEXT,                       -- JSON：实际产出（广播了什么/说了什么/写了什么）
  error       TEXT
);
CREATE INDEX IF NOT EXISTS idx_noe_ticks_sd ON noe_ticks(status, due_at);

-- 节奏游标：每类 tick 的下次到期 + 当前节奏（重启从这里续，不从零开始）
CREATE TABLE IF NOT EXISTS noe_tick_cursor (
  kind TEXT PRIMARY KEY,
  next_due INTEGER NOT NULL,
  cadence_ms INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);
```

调度器骨架（单 setTimeout 链，不用多 timer——串行性是特性不是限制）：

```js
// @ts-check  src/loop/NoeHeartbeat.js（骨架示意）
export function createHeartbeat({ store, clock = () => Date.now(), jobs, circadian = null,
                                  vitals = null, leaseMs = 5 * 60_000, log = () => {} }) {
  // jobs: Map<kind, { run(ctx):Promise<outcome>, cadenceMs, catchUp:'drop'|'once'|'all', maxCatchUp?:number }>
  let timer = null, stopped = false;

  async function pump() {
    if (stopped) return;
    const t = clock();
    store.recoverDeadTicks(t);                       // running 且 lease 过期 → failed（留痕，不重试有副作用的 ACT）
    const due = store.dueCursors(t);                 // 到期的 kind 列表，按 kind 优先级 micro<meso<macro 排序
    for (const cur of due) {
      const job = jobs.get(cur.kind); if (!job) continue;
      const missed = Math.max(0, Math.floor((t - cur.next_due) / cur.cadence_ms));
      if (missed > 0 && job.catchUp === 'drop') store.markCoalesced(cur.kind, missed); // 欠账记台账，不补跑
      const runs = job.catchUp === 'all' ? Math.min(missed + 1, job.maxCatchUp ?? 2) : 1;
      for (let i = 0; i < runs; i++) {
        const tickId = store.beginTick(cur.kind, t, t + leaseMs, job.intent?.() ?? null); // 写前日志+租约
        try { const out = await job.run({ tickId, now: t });  store.finishTick(tickId, out); }
        catch (e) { store.failTick(tickId, e?.message); }      // fail-open：单 tick 失败不影响心跳本身
      }
      const cadence = circadian ? circadian.modulate(cur.kind, cur.cadence_ms, t) : cur.cadence_ms;
      store.advanceCursor(cur.kind, alignNext(t, cadence), cadence);
    }
    timer = setTimeout(pump, store.nextWakeDelay(clock(), { min: 1000, max: 60_000 }));
    timer.unref?.();
  }
  return { start: () => pump(), stop: () => { stopped = true; clearTimeout(timer); } };
}
```

**节奏表（默认值=现有各 timer 的取值，保证开关打开后行为与现状对齐）**：

| kind | 默认节奏 | 收编对象 | catchUp |
|---|---|---|---|
| micro | 30s | 情感衰减（§4）、承诺/期望到期检查、感知缓存刷新；无 LLM | drop |
| meso | 15min（`NOE_INNER_INTERVAL_MS`） | InnerMonologue 反刍 + 认知周期（§2.2）+ proactiveTick 服务端触发 | once |
| macro | 每日夜相（接 `NoeCircadian` 四相） | NoeNightlyReflection、NarrativeSelf 日更、GC、PersonalitySnapshot 周更、SFT 攒取 | all(≤2) |
| dream | 6h | NoeEpisodeSublimation 梦境升华 | once |

`NoeCircadian.modulate` 给 meso 做昼夜调制：白天活跃 5–15min、深夜 60min（夜里仍有心跳=仍"存在"，
只是想得慢、且 quiet hours 不开口——proactiveTick 已有 isQuiet 注入位，直接复用）。

### 3.3 崩溃恢复协议（与 NoeProcessVitals 联动）

1. 启动时读 `~/.noe-panel/last-exit.json`（已有）：上次非正常退出 → 心跳第一个 tick 标记 `kind=recovery`，
   产出一条情景记忆（"我刚才意外断了 X 分钟"）→ 这条会进工作区候选，让 Noe **知道自己断过**（机制五）。
2. `noe_tick_cursor` 决定续跑相位：停机 3 小时 → meso 欠账 12 次按 once 只补 1 次；macro 夜反思错过 → 醒来补跑。
3. 有副作用的 ACT（开口/外发）**永不自动重放**：恢复只补"内部认知"类 tick；台账里 failed 的 ACT 留给反思素材。
4. 看门狗：心跳自身每 pump 向 `NoeHangAlert` 续命（已有 extend() 机制）；心跳停摆 >10min → 告警。

### 3.4 proactiveTick 的收编（兼容迁移，零行为变化）

- handler 本体（119 行）**不动**：心跳 meso tick 在服务端直接调它，`lastSpokeAt` 等状态外提为
  构造参数 `persistedState`（从 kv 表读写，键 `noe.proactive.state`）。
- 前端轮询端点保留（手动 force/调试用），与心跳共享同一持久状态 → 双触发天然被冷却约束，不会双开口。
- 开关：`NOE_HEARTBEAT=1`（默认 OFF）。OFF 时一切照旧（前端轮询+各自 setInterval），ON 时各旧 timer
  检测到心跳在跑则让位（装配点判断，运行时零竞争）。

### 3.5 测试策略（照宪法）

- 单测：注入 fake clock + 内存 SQLite——到期触发/欠账合并/租约恢复/circadian 调制各一组。
- 杀进程测试：起隔离服（51998）→ kill -9 在 running tick 中点 → 重启 → 断言 recovery tick 产生、
  cursor 续相位、死 tick 标 failed 不重放。

## 4. NoeAffectEngine：情感连续性（结构性缺口二）

### 4.1 状态表示与动力学

新文件 `src/cognition/NoeAffectEngine.js`。状态 = 三维向量 VAD（valence 愉悦 -1..1 / arousal 唤醒 0..1 /
dominance 掌控 -1..1）+ 慢变量 mood + 常量 baseline（来自 PersonalitySnapshot，性格决定"心情的家"）。

```
事件评估（OCC-lite，S1 结构化输出，对广播项跑）:
  {goalCongruence:-1..1, novelty:0..1, agency:0..1, certainty:0..1, socialWarmth:-1..1}
状态更新:
  Δv = 0.30·goalCongruence + 0.20·socialWarmth
  Δa = 0.25·novelty + 0.15·|goalCongruence| + surprise·0.2     // surprise 来自期望账本（§7.5）
  Δd = 0.20·(agency - 0.5)
  x ← clamp(x + Δx)
双时标衰减（micro tick 执行，Δt = 距上次更新）:
  情绪 → 心境:  x    ← mood + (x - mood)·exp(-Δt/τe)，τe ≈ 90min
  心境 → 基线:  mood ← base + (mood - base)·exp(-Δt/τm) + 0.02·(x - mood)，τm ≈ 7天
```

持久化：迁移 v4 表 `noe_affect(ts, v, a, d, mood_v, mood_a, mood_d, cause)`，meso tick 落一行快照
（cause 记归因 JSON）。**重启水合**：读最后快照 → 对停机时长套同一衰减式 → "睡醒心情回落但没清零"——
这就是情感连续性的全部实现，无魔法。

### 4.2 与现有件的关系（不重复造轮）

- `NoeMoodAnalyzer`（支柱④）= **评估器**：分析对话文本的情绪信号，输出作为 appraise 的输入之一。
- `NoeAffectEngine` = **状态机**：持有连续状态。SelfModel 状态层的"心境"字段改从它读（provider 注入，
  原启发式做 fallback，fail-open）。
- `NoeDriveSystem` 与之双向：驱力满足/受挫产生 goalCongruence 事件；arousal 低且社交驱力饥渴 →"想找主人"。

### 4.3 感受词元（内感受注入）

渲染函数 `renderFeelingTokens()` 把状态翻译成一行自然语言，作为 TurnContextEngine 的新 provider，
注入意识流提示、proactiveTick 的 driveNote 同位、以及对话 system 的 self-state 段：

```
此刻内在：心情偏暖(+0.3)、精神平静(0.2)、有掌控感(+0.4)；几分牵挂（2 条承诺未了）；
好奇微涨（昨晚一条预测落空了）。
```

调制下游（轻量、确定性）：高 arousal → 工作区 novelty 权重 +0.05；valence 低 → 浮现门阈值升高
（低落时更克制，不拿坏情绪打扰主人）；语气 hint 进 TTS 文案提示。**禁止**用情感直接改危险行为阈值。

### 4.4 开关与测试

`NOE_AFFECT=1` 默认 OFF。单测：衰减数学（定时标）、水合（模拟停机 8h）、评估映射边界、
fallback 行为与现状逐字一致。

## 5. 意识流 v2（InnerMonologue 的升级，不是替换）

### 5.1 念头条目 schema（仍写 EpisodicTimeline，单一自传不分叉）

沿用 events 表 kind=noe_episode，payload 扩展（向后兼容，旧念头无新字段照常可读）：

```json
{ "episodeType": "thought",
  "streamType": "observation|feeling|question|plan|memory_echo|self_talk|dream",
  "text": "……",
  "salience": 0.0,                  // S1 自评 0..1，工作区候选用
  "affect": { "v": 0.3, "a": 0.4 },  // 产生时刻的情感印记（回忆再激活用）
  "echoRefs": ["episode:123"],       // 引用了哪些记忆/上一念（机制二的链）
  "tickId": 4567 }
```

### 5.2 生成算法（meso tick 的 SENSE→意识流步，S1 执行）

```
输入装配（≤1.5k token，走 ContextBudgeter）:
  感受词元（§4.3） + 最饥渴驱力 brief（已有） + 上一念（再入） +
  记忆回声 1–2 条（采样见下） + 新感知摘要（视觉/收件箱有新才给） + 活跃目标 top-1
输出（结构化 JSON）: 1–2 条念头 {streamType, text, salience, wantsAction?: {type, draft}}
```

**记忆回声采样**（替代 InnerMonologue 现在的"按时间回放"）：

```
候选 = 最近 7 天情景 ∪ salience>0.6 的历史情景（各取 50）
p(m) ∝ softmax( 0.4·salience + 0.3·recencyDecay + 0.3·affectCongruence(m.affect, 当前VAD) )
dream tick 反转第三项符号（采样情感不相称的远期记忆 → 意外并置，创造性来源）
```

防螺旋（现有防线保留再加一条）：连续 3 念两两相似度 >0.8（向量）→ 强制注入"换个话题"指令 + 跳过一轮回声。

### 5.3 维持性（"流"之所以为流）

- 心跳保证节奏（§3）；每念引用上一念（再入）；情感印记保证"今天的念头带着今天的心情"；
- 晨间第一个 meso tick 注入特殊上下文：昨夜反思 insight + 梦念头 + 停机说明 → "醒来接着活"，不是冷启动。

### 5.4 浮现门（统一所有"开口"路径）

反刍升华（支柱③⑥）、proactiveTick、念头 wantsAction 三路合一个门（`src/cognition/NoeSurfacingGate.js`）：

```
通过条件（全部满足才开口）:
  salience ≥ 0.7  ∧  ¬quietHours  ∧  日预算未超（默认 8 次/天，记 kv）∧
  与近 24h 已说内容相似度 < 0.75  ∧  冷却到期（复用 proactive cooldownMs）
通过 → 走 proactiveTick 的 TTS/播放/写记忆现成路径；不通过 → 念头标 surfaced:false 留痕（"想说没说"也是经历）
```

开关：`NOE_STREAM_V2=1` 默认 OFF（OFF 时 InnerMonologue 原样跑）。

## 6. NoeWorkspace：全局工作区与注意力（结构性缺口三）

### 6.1 候选与打分

新文件 `src/cognition/NoeWorkspace.js`。每 meso tick 收集候选（结构 `{source, content, meta}`）：
新感知摘要 / 上一念 / 驱力误差超阈 / 到期承诺·期望 / 活跃目标下一步 / 收件箱新消息（Telegram 等）。

```
score(c) = 0.35·owner + 0.20·urgency + 0.15·affectImpact + 0.12·goalRelevance
         + 0.10·novelty + 0.08·driveMatch
  owner: 主人直接相关=1（对话/留言/被提到）——陪伴产品性格：主人的事几乎总赢
  novelty: 1 - max cos(emb(c), 最近 50 条广播向量)   // qwen3-embedding 本地，已有
  urgency: 截止逼近度；affectImpact: 评估器预测 |Δv|+Δa；driveMatch: 与最饥渴驱力匹配度
```

赢家"广播"= 成为本周期认知内容；权重进 config（kv 表），先手调后期再谈学习。

### 6.2 双系统升级判据（DELIBERATE 选脑）

```
用 System2 当且仅当: score > 0.85 ∨ 涉及自我模型修改 ∨ 计划生成/修订 ∨
                    同类 S1 处理连续失败 ≥2 ∨ macro tick（反思类全走 S2）
其余走 S1。预算: S2 单次 ≤ 8k 输出 token；夜间 macro 不限次但串行。
```

### 6.3 自我质询协议（System2 审议的固定程序，委托 4.1）

```
prompt 程序（带 MemoryCore 检索工具，≤3 轮）:
  ① 立论：给出结论草案 + 主观概率 p
  ② 自我挑战（一次性生成三路）: a) 检索反例（查记忆/时间线找矛盾证据）
     b) 边界情况（什么条件下结论失效） c) 替代解释（最强竞争假设是什么）
  ③ 修订：吸收挑战 → 终版结论 + 修正后 p + 一句"我哪里可能仍然错"
落库：结论进语义记忆（confidence=p，provenance=tickId）；可检验的 p 同步登记期望账本（§7.5）
```

### 6.4 意识日志

每周期一行 JSONL（`~/.noe-panel/consciousness/<date>.jsonl`，月度压缩进叙事自我后归档）：
`{tickId, 候选摘要, 赢家, 得分明细, 感受快照, 选了哪个脑, 行动, 耗时}`。
这是机制一/六的证据层、调试器、也是"体验像什么"的可查询近似。开关 `NOE_WORKSPACE=1` 默认 OFF。

## 8. NoeGoalSystem：自主目标与学习

### 8.1 目标库（迁移 v4 第四张表）

```sql
CREATE TABLE IF NOT EXISTS noe_goals (
  id TEXT PRIMARY KEY, created_at INTEGER NOT NULL,
  source TEXT NOT NULL,        -- owner|drive:curiosity|drive:social|reflection|commitment
  title TEXT NOT NULL, why TEXT,            -- 动机链：哪个驱力误差/哪条反思生的它
  priority REAL NOT NULL, status TEXT NOT NULL DEFAULT 'open',  -- open|active|paused|done|dropped
  plan TEXT,                   -- JSON 计划树：[{step, status, toolRef?, result?}]
  budget TEXT, updated_at INTEGER NOT NULL  -- {tokensPerDay, externalCalls:0}
);
```

### 8.2 生成与仲裁

- **生成**：驱力误差超阈持续 ≥2 个 meso tick → S1 提议候选目标（"好奇 0.8 饥渴 + 昨天那条预测落空 →
  研究一下 X"）；反思 insight 可直接生目标；反刍升华的"自生承诺"自动升格为 source=commitment 的目标。
- **仲裁**（每 meso tick 重算，确定性公式不开 LLM）：

```
priority = 0.40·sourceWeight(owner=1.0, commitment=0.8, reflection=0.6, drive=0.4·driveError)
         + 0.20·预期驱力缓解 + 0.15·urgency + 0.15·可行性(所需工具齐=1) − 0.10·成本惩罚
同时 active ≤ 2；owner 显式任务永远压过自生目标。
```

### 8.3 跨 tick 执行

active 目标的下一步作为工作区候选（§6.1）；赢了 → DELIBERATE 产出/推进计划树一步（工具桥、
研究路由、写记忆都是现成行动原语）；步骤结果写回 plan JSON → 天然断点续跑（心跳台账兜底）。
完成 → 驱力满足回填 + 情景记忆 + （过浮现门）告诉主人。预算硬约束：外部付费调用默认 0；
本地 token 软预算超了 → 目标自动 paused 进次日。

### 8.4 好奇回路（自主学习与知识迭代）

```
缺口检测（全是现成信号）: 检索空振 / 低 confidence 作答 / 主人纠正 / 高 surprise
  → 生成 source=drive:curiosity 学习目标
执行: 研究步骤走已有 research 路由 + playwright MCP（本地浏览，不花钱）
蒸馏: S2 总结 → MemoryCore.write(confidence、provenance=url、tag=learned)
闭环: 次日意识流回声大概率采到新知识（新近性+salience 高）→ "我昨天学到…"自然浮现 → 主人可感
```

开关 `NOE_GOALS=1`、`NOE_CURIOSITY=1` 默认 OFF。自主行动边界沿用项目现状：本地/只读自由，
外发走浮现门预算，付费 API 一律不自主（freedom 给开发，克制给打扰，花钱归零）。

```
  KV cache 8bit 量化（长上下文反思省一半 KV 内存）；按 job class 配 max_tokens/温度
  /admin/load|unload → 心跳的换档策略可编程控制（电池卸 S2）
```

加速选项备忘：投机解码对 MoE-A3B 收益边际（激活本来就小），不做；批并发不需要（认知串行是特性）。

### 9.5 基准协议（先测后信，宪法第 5 条）

落地第一步跑 `scripts/noe-llm-bench.mjs`（新增，curl 计时即可）：固定 3 组上下文（1k/8k/32k）×
固定生成 512 token，记录 TTFT/tok/s/峰值内存（`sudo powermetrics` 或活动监视器人读），
结果写 `docs/基准_M5Max_模型实测_<date>.md`，**覆盖本节一切"估"数字**；S2 候选谁实测最优用谁。
